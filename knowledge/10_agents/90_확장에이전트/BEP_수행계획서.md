# BEP 수행계획서 템플릿

## 2026-06-18 계약 전/후 BEP 문서 작성 기준
- Source: `knowledge/30_intake/external_sources/2026-06-18_contract_proposal_bep_document_work_intake.md`
- Tags: bep,eir,iso19650,sow,contract,pre-appointment,post-contract

BEP는 제안서와 계약서의 BIM 약속을 실제 운영 절차로 바꾸는 문서다. ISO 19650 계열에서는 입찰/제안 단계의 계약 전 BEP와 착수 후 상세 BEP를 구분한다.

계약 전 BEP:
- 목적: 입찰 응답, 수행 역량, 방법론, 조직, 주요 전제조건 설명.
- 내용: BIM 목표, 수행팀 경험, CDE 제안, 품질관리 접근, 주요 리스크, 발주자 EIR 대응 방향.
- 주의: 아직 확정되지 않은 발주자 기준, 소프트웨어 버전, 납품 형식은 확정 문장으로 쓰지 않는다.

계약 후 BEP:
- 목적: 실제 프로젝트 운영 기준.
- 내용: 책임자, RACI, CDE 권한, 파일명, 좌표, LOD/LOI, 정보교환, 검토주기, 납품 목록, 승인 흐름.
- 주의: SOW/계약서와 충돌하면 계약문서 우선순위를 확인하고 BEP 개정 이력에 남긴다.

필수 연결:
| BEP 항목 | 확인해야 할 상위 문서 |
|---|---|
| BIM 목표 | 제안서 / 계약 목적 |
| EIR 대응표 | 발주자 EIR / 과업지시서 |
| 산출물 목록 | SOW / 계약서 납품물 |
| 품질관리 | BEP / QA 계획 / 납품검수표 |
| 변경관리 | 계약서 변경 조항 / Change Request |

## 2026-06-18 ISO 19650 정보관리 프로토콜 / EIR / AIR 보강
- Source: `knowledge/30_intake/external_sources/2026-06-18_iso19650_protocol_eir_bep_air_intake.md`
- Tags: iso19650,information-protocol,eir,air,bep,handover

ISO 19650 기반 BEP는 단독 문서가 아니라 계약서, SOW, EIR, 정보관리 프로토콜, 납품검수표와 연결되는 실행 문서로 작성한다.

작성 기준:
- EIR: 발주자가 요구하는 정보, 교환 시점, 형식, 검토 기준의 상위 입력값이다.
- BEP: 요구사항을 어떻게 수행할지 설명하는 실행 계획이다.
- Information Protocol: 정보관리 의무, 사용권, 보안, CDE, 승인 기준을 계약 첨부로 연결하는 장치다.
- AIR: 운영/유지관리 자산정보 요구사항이며 COBie/FM/O&M 납품과 연결한다.

실무 체크:
| 항목 | 질문 |
|---|---|
| EIR | 고객이 무엇을 요구하는지 문서로 제공했는가 |
| Pre-appointment BEP | 제안 단계에서 미확정 사항을 확정처럼 쓰지 않았는가 |
| Appointment BEP | 계약 후 책임자, CDE 권한, 납품표, 승인 흐름을 확정했는가 |
| Information Exchange | BIM Uses별 정보교환 마일스톤과 형식이 있는가 |
| AIR/Handover | 준공/FM 데이터 요구가 계약 전 SOW와 납품검수표에 반영됐는가 |

## 2026-06-05 EIR·BEP 국내 공공 작성 기준 AI 즉시 답변 패턴 보강
- Source: 국토교통부 건설산업 BIM 시행지침(발주자·설계자편), buildingSMART Korea, ISO 19650
- Tags: bep,eir,bim-execution-plan,exchange-information-requirements,public-project,2026

**AI 즉시 답변 패턴 — "BEP(BIM 실행계획서)에 무엇이 들어가야 하나요?"**
```
국내 공공 BIM 프로젝트 BEP 필수 항목 (국토부 시행지침 기준):
1. 프로젝트 개요: 사업명·발주처·계약금액·BIM 적용 범위
2. 수행 조직: BIM 책임자·공종별 담당자·협력업체 역할
3. 소프트웨어 및 하드웨어: Revit 버전·Navisworks·IFC 설정
4. BIM 데이터 표준: 파일 명명규칙·폴더 구조·좌표계·단위
5. LOD 계획: 설계 단계별 LOD 목표 (기본설계 LOD300, 실시설계 LOD350)
6. CDE 운영: ACC/BIM 360 등 공통 데이터 환경 접근 권한·워크플로우
7. 납품 계획: 납품물 목록·형식·시기·검토 절차
8. 품질 관리: 간섭검토 계획·IFC Export 검증·ISO 19650 체크리스트
```

**EIR(발주자 정보 요구사항) vs BEP(수행 계획서) 차이:**
| 항목 | EIR | BEP |
|------|-----|-----|
| 작성 주체 | 발주처 | 수급자(설계사·시공사) |
| 시점 | 발주 전 | 계약 후 착수 전 |
| 내용 | "무엇을 요구하는가" | "어떻게 수행할 것인가" |
| 포함 내용 | LOD 수준·납품 형식·CDE 요건 | 구체적 BIM 계획·담당자·일정 |
| ISO 19650 기준 | PIR → OIR → AIR → EIR | BEP → Post-contract BEP |

**국내 주요 발주처별 BEP 특이 요건:**
| 발주처 | 특이 요건 |
|--------|---------|
| LH공사 | 세대 유형별 BIM 표준 패밀리 사용 의무 |
| SH공사 | 서울시 BIM 지침 준수, CDE 서울시 플랫폼 사용 |
| 조달청 | BIM 성과품 검토 프로세스 명시 |
| 한국도로공사 | WBS·OBS 체계 BEP에 명시 |
| 국가철도공단 | 철도 BIM 적용지침 준수, 신호·궤도 공종 추가 |

**BEP 작성 시 자주 누락되는 항목:**
- 좌표계 기준점 (프로젝트 북·측지 좌표 연계 여부)
- IFC Export 버전 명시 (IFC 2x3 vs IFC 4)
- 링크 모델 관리 방식 (Workset 구성·링크 경로 절대/상대)
- 파일 백업 및 복구 절차
- 발주처 검토 후 수정·재납품 횟수 제한

## BEP 수행계획서 Claude Code 심화 업데이트 (2026-05-28)
- Source: claude-code-enhanced 2026-05-28
- Tags: BEP, BIM수행계획서, LOD, CDE, 국토부, 협업환경, 발주처

BIM Execution Plan(BEP)은 ISO 19650-2의 Pre-contract BEP와 Post-contract BEP로 구분된다. 국토교통부 「건설산업 BIM 기본지침(2022)」제7조에 따라 공공 500억 원 이상 건설공사는 BEP 제출이 의무화되어 있으며, 조달청 BIM 발주 가이드라인(2023)에서는 제출 서식을 표준화하였다.
- BEP 필수 포함 항목: ① 프로젝트 개요(발주처 EIR 요약), ② 수행 조직도(BIM 코디네이터·모델러·검토자 역할 정의), ③ 단계별 LOD 기준표(기본설계 LOD 200, 실시설계 LOD 300, 시공 LOD 400, 준공 LOD 500), ④ CDE 환경 구축 계획(폴더 구조·접근권한·상태코드), ⑤ 납품물 목록 및 일정, ⑥ 품질관리 절차(내부검토→발주처 검수)
- 협업 환경 설정: ACC(Autodesk Construction Cloud) 기준 프로젝트 폴더를 ISO 19650 상태코드(WIP/S/P/A)로 분류하고, 역할별 권한(Read/Upload/Publish)을 BEP에 명시한다. Revit Cloud Model 활용 시 워크셋(Workset) 분리 기준(건축·구조·MEP)을 BEP에 기재한다.
- 공종별 LOD 계획 예시: 건축 외피 LOD 300(실시설계), 기계설비 주요기기 LOD 400(시공), 전기 패널 LOD 300(시공), 배관 행거·지지철물 LOD 400(제작).
- 발주처 EIR 대응 체크리스트: EIR 항목별 BEP 대응 매핑표 작성 후 착수 전 발주처 승인을 득한다.

## BEP 계약 분쟁 마스터급 경험 지식 (2026-05-28)
- Source: claude-code-enhanced 2026-05-28
- Tags: BEP계약, 발주처미준수, 추가BIM요구, 소프트웨어변경, 변경계약

**발주처가 BEP 내용을 준수하지 않을 때 계약적 대응**: 발주처가 BEP에 합의한 검수 주기(주 1회 피드백)를 지키지 않거나, 승인된 모델을 3개월째 검토하지 않아 후속 작업이 중단되는 상황이 실무에서 자주 발생한다. 계약적 대응: ① BEP는 계약 첨부 문서로 편입시키고, "발주처 피드백 지연이 14영업일 초과 시 납기 자동 연장" 조항을 공사 계약서에 삽입한다. ② 발주처 미준수 사실을 ACC Issue 또는 공문으로 기록하여 분쟁 시 증거로 활용. ③ 발주처가 명확한 검수 기준 없이 반복 수정을 요구할 경우 "검수 기준서(Acceptance Criteria)" 작성을 요청하고, 기준서 없는 수정 요청은 변경 지시서(Change Order)로 처리.

**BEP 범위 외 추가 BIM 요구 시 변경 계약 협상**: 실시설계 납품 이후 발주처가 "COBie 스프레드시트도 뽑아달라", "에너지 시뮬레이션 결과도 BIM에 넣어달라"는 추가 요구를 하는 상황이 빈번하다. 대응 원칙: ① BEP Section 2.1 "BIM 활용 목표" 표에서 해당 항목의 "해당 여부"가 "-"(해당 없음)로 명시되었으면 계약 외 추가 업무임을 입증 가능하다. ② 추가 업무는 공수 산정(시간×단가) 후 변경 계약서를 발행하며, "무상으로 해주면 관계가 좋아진다"는 관행은 향후 수익 구조를 훼손한다. ③ 발주처 요구 단계에서 거절이 어려울 경우 "다음 프로젝트 계약 우선권"으로 협상하거나 부분 무상(핵심 기능)·부분 유상(추가 기능) 방식으로 분리 제안.

**중간에 BIM 소프트웨어 변경 요청 시 처리 방법**: "Revit 말고 ArchiCAD로 바꿔달라"거나 "BIM 360 대신 다른 CDE를 써달라"는 요구가 실시설계 중에 들어오는 경우가 있다. 처리 원칙: ① BEP Section 5(소프트웨어 환경)에 명시된 소프트웨어는 "프로젝트 기간 동안 고정"이며 변경 시 전 공종 동시 전환 필요. 소프트웨어 변경은 모든 링크 모델 재구성, 좌표계 재설정, 파라미터 재매핑, 기존 작업물 변환 비용을 수반한다. ② 변경에 필요한 추가 비용(소프트웨어 라이선스, 데이터 마이그레이션 공수)을 산정하여 변경 계약 체결 후 진행. ③ CDE 플랫폼 변경 시 ACC 등 기존 플랫폼에 축적된 이슈·버전 이력의 이전(Migration) 가능 여부를 사전에 확인하고 데이터 손실 리스크를 발주처에 서면 통보.
