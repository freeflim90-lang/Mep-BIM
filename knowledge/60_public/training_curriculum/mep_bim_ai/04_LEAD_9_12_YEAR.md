## 1. 과정 목표

9~12년차는 프로젝트 수행 기준을 만들고, 공종별 산출물의 품질을 승인하며, 반복 업무를 자동화 요구사항으로 전환할 수 있어야 한다.  
9년차는 BEP 작성과 표준 수립을 직접 경험하고, 10년차부터는 공종 리드로서 후배를 지도하면서 프로젝트 전체의 품질과 납품을 책임진다.

## 6. BEP 세부 작성 가이드

BEP는 프로젝트 BIM 수행의 계약적 근거이자 실무 지침이다.

### BEP 필수 섹션 구성

| 섹션 | 핵심 내용 | 작성 시 주의사항 |
|---|---|---|
| 프로젝트 정보 | 발주처, 규모, 발주 방식, BIM 목적 | 계약 범위와 불일치 금지 |
| 조직 및 RACI | 발주처·시공사·설계사·BIM 팀 역할 | 책임 공백 없이 매트릭스 작성 |
| BIM 목표 | LOD, 활용 목적, 성과 기준 | 정량적 목표(LOD 350, 기계실 등) 명시 |
| 소프트웨어·버전 | Revit, Navisworks, BIM 360 버전 | 협력사 소프트웨어 호환성 확인 |
| 파일 명명 규칙 | 프로젝트-공종-구역-Revision | 프로젝트 공통 기준으로 단일화 |
| CDE 구조 | WIP/Shared/Published 폴더·권한 | 발주처 요구 CDE 플랫폼 확인 |
| 모델 분할 기준 | 구역, 층, 공종별 분할 방법 | 파일 크기·협업 효율 균형 |
| 간섭 검토 계획 | 주기, 규칙, 보고 형식 | 해결 기한과 책임자 포함 |
| 납품 계획 | 납품 형식, 날짜, 검수 기준 | 계약 납품 일정과 동기화 |
| 변경 관리 | 변경 접수, 승인, 모델 반영 절차 | 무상 변경 범위 명시 |
| 보안 | 자료 반출, 외부 AI 입력, 접근 권한 | 프로젝트 보안 등급 적용 |

### BEP 작성 절차

1. 발주처 EIR(고용주 정보 요건) 검토 → 요건 미달 항목 목록화
2. 협력사·공종별 능력 조사 → 소프트웨어, 인력, 경험 확인
3. BEP 초안 작성 → 조직 내 검토 후 발주처 제출
4. 발주처 의견 반영 → 개정 이력 관리(Rev.01, 02)
5. 착공 전 최종 승인 → 승인된 BEP가 수행 기준이 됨

## 7. ISO 19650 및 BIM 표준 이해

| 기준 | 핵심 내용 | 리드 적용 포인트 |
|---|---|---|
| ISO 19650-1 | BIM 정보 관리 개념, 정보 요건 정의 | EIR/BEP 체계 이해 |
| ISO 19650-2 | 자산 납품 단계 정보 관리 절차 | CDE 운영, 납품 흐름 |
| ISO 19650-3 | 운영 단계 정보 관리 | FM 연계 COBie 납품 기준 |
| ISO 19650-5 | 보안 고려사항 | 민감 자료 취급, 접근 제한 |
| IFC 표준 | 개방형 데이터 교환 형식 | IFC 내보내기 설정 검증 |
| COBie | 시설관리용 데이터 시트 형식 | 장비·공간 파라미터 사전 계획 |

### 국내 기준 연계

| 기준 | 적용 포인트 |
|---|---|
| 국토부 BIM 적용 지침 | 공공 발주 BIM 납품 요건 기준 |
| 조달청 BIM 성과기준 | LOD 정의, 적용 단계별 납품 형식 |
| 건설기술진흥법 | BIM 적용 의무화 대상 사업 구분 |

## 8. 팀 운영 및 멘토링

### 업무 배분 원칙

| 원칙 | 내용 |
|---|---|
| 역량 기반 배분 | 담당자 연차·경험에 맞는 업무 배정 |
| 기한 명확화 | 중간 검토일, 최종 제출일 모두 명시 |
| 병목 사전 파악 | 주간 업무 현황에서 지연 징후 조기 확인 |
| 오류 교육화 | 반복 오류는 개인 지적보다 기준 보완으로 해결 |

### 멘토링 방법

| 대상 | 멘토링 포인트 |
|---|---|
| 1~2년차 | 기본 품질 기준 이해, 도면-모델 일치 습관 형성 |
| 3~5년차 | 판단 근거 설명하기, 스스로 보고서 작성 능력 |
| 6~8년차 | 회의 주도, 이슈 종결, 리스크 사전 보고 습관 |

### 주간 업무 점검 운영

- 매주 월요일: 이번 주 마감 과제, 병목 확인
- 매주 금요일: 완료 여부 확인, 다음 주 우선순위 조정
- 월 1회: 개인별 성장 과제 리뷰, 교안 보완

## 9. 발주처·감리·협력사 대응

### 발주처 대응

| 상황 | 리드 대응 방법 |
|---|---|
| BIM 요건 불명확 | EIR 재확인 요청, 질의서(RFI) 공식 발행 |
| 납품 기준 변경 | 변경 범위 명확화 후 SOW 변경 협의 |
| 추가 요청 발생 | 무상/유상 여부 BEP/계약 기준으로 판단 |
| 납품 반려 | 반려 사유 목록화, 보완 기한 협의, 재제출 계획 보고 |

### 감리 대응

| 상황 | 리드 대응 방법 |
|---|---|
| 감리 검토 의견 | 의견 분류(반영/불반영/협의) 후 회신 작성 |
| 현장 확인 요청 | 해당 구간 모델 캡처·좌표 정보 준비 |
| 법규 지적 | 설계팀과 협의 후 모델 수정, 이력 관리 |

### 협력사 검수

| 단계 | 내용 |
|---|---|
| 온보딩 | BEP 핵심 기준, 파일 명명, CDE 사용법 전달 |
| 1차 제출 | 샘플 구역·층 우선 검수, 기준 재교육 |
| 본 검수 | 품질 체크리스트 기준 반려 의견 작성 |
| 반려 처리 | 반려 사유·보완 기준·재제출 기한 명시 |
| 최종 승인 | 보완 완료 확인 후 Shared→Published 배포 |

## 12. Add-in 개발 심화

리드는 Add-in을 직접 개발하지 않더라도, 개발팀과 협업하기 위해 API 구조를 이해하고 요구사항을 명확하게 작성해야 한다.

### Revit API 핵심 개념

| 개념 | 설명 | 리드 활용 포인트 |
|---|---|---|
| Transaction | 모델 변경 작업의 단위 | 변경 범위 정의에 활용 |
| FilteredElementCollector | 요소 수집 쿼리 | 검수 대상 요소 범위 지정 |
| BuiltInParameter | Revit 내장 파라미터 코드 | 파라미터 명세 작성 |
| FamilyInstance | 패밀리 배치 요소 | 장비·피팅 파라미터 검토 |
| View / Sheet | 뷰와 도면 시트 구조 | 납품 뷰 자동화 요건 |

### 기능 명세서 작성 기준

```
기능명: [동사+목적어] (예: 파라미터 누락 요소 강조 표시)

문제:
  - 현재 어떤 업무를 수작업으로 하고 있는가?
  - 오류 빈도, 소요 시간은?

입력:
  - 어떤 모델, 파라미터, 기준값이 필요한가?

처리:
  - 어떤 조건으로 요소를 필터링하는가?
  - 어떤 값을 읽거나 쓰는가?

출력:
  - 화면 강조, XLSX 보고서, 로그 중 무엇인가?

예외:
  - 링크 파일 포함 여부, 파라미터 없는 요소 처리

검증 방법:
  - 샘플 모델에서 예상 결과와 실제 결과 비교
```

### QA 시나리오 설계 원칙

| 시나리오 유형 | 설명 |
|---|---|
| 정상 케이스 | 기준 충족 요소가 올바르게 통과되는지 |
| 경계 케이스 | 기준 경계값(예: LOD 300 vs 350) 처리 |
| 오류 케이스 | 파라미터 누락, 잘못된 값 입력 시 동작 |
| 회귀 케이스 | 이전 버전에서 발생한 버그가 재발하지 않는지 |
