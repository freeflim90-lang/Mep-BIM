## 1. 티켓 목적

Revit Add-in 오류 로그와 CS 재현 로그에서 고객명, 프로젝트명, 모델 경로, 이메일, 전화번호, 라이선스 키, API 토큰, 고객 모델 원본 경로가 남지 않도록 로그 마스킹 기능을 설계한다.

목표:
- 개발자는 안전하게 디버깅 가능한 최소 정보를 남긴다.
- CS는 고객에게 요청할 로그 범위를 명확히 안내한다.
- 보안관은 `KST06 보안제한` 데이터가 로그·Obsidian·AI 입력으로 확산되지 않도록 검토한다.

## 3. 범위

포함:
- Add-in 로컬 로그
- 오류 리포트 텍스트
- CS 첨부 로그 사전 점검
- 원격 분석 payload 초안
- Obsidian에 남길 익명화 오류 패턴

제외:
- 고객 모델 원본 업로드 기능
- 라이선스 서버 인증 로직 본 구현
- 외부 AI API 전송 기능
- 보안 사고 대응 프로세스 전체

## 6. 개발 요구사항

| ID | 요구사항 | 완료 기준 |
|---|---|---|
| LOG-001 | 로그 작성 전 민감정보 마스킹 함수를 통과한다 | 모든 로그 라이터에 적용 |
| LOG-002 | 라이선스 키와 토큰은 저장하지 않는다 | 테스트 로그에서 패턴 0건 |
| LOG-003 | 모델 파일 경로는 전체 경로가 아니라 확장자와 익명 ID만 남긴다 | `[MODEL_REF]` 형태로 저장 |
| LOG-004 | 오류 리포트에 허용 필드 목록이 명시된다 | CS 전달 템플릿과 일치 |
| LOG-005 | 마스킹 실패 시 원문 저장 대신 로그 작성을 중단한다 | fail-closed 동작 |
| LOG-006 | Obsidian 지식화용 요약은 익명화된 증상·원인·해결 절차만 포함한다 | KST06 원문 미포함 |

결론:
로그 마스킹 기능은 고객 모델과 개인정보를 보호하기 위한 필수 보안 티켓으로 처리합니다.
