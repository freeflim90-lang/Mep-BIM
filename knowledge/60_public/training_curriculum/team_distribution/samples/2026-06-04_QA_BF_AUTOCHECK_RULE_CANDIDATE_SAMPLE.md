## 1. 샘플 목적

이 샘플은 BF/편의시설 기준을 QA_테스터가 자동검수 룰 후보로 바꿀 때, 자동화 가능한 항목과 수동 검토 항목을 분리하는 훈련 문서다.

핵심 원칙:
- 자동검수 룰은 누락, 입력 여부, 링크 존재, 장애물 후보를 찾는 도구다.
- 자동검수 PASS는 BF 적합, 인증 통과, 납품 합격을 의미하지 않는다.
- 대상시설, 의무/권장, 예비인증/본인증, 지자체 조건, 특기시방서 해석은 수동 검토로 남긴다.
- `KST04 자동수집` 수치나 항목명은 공식 전문과 프로젝트 조건 확인 전 차단 룰로 쓰지 않는다.

## 2. QA 판단 요약

| 항목 | 현재 판단 |
|---|---|
| 결론 | BF/편의시설 자동검수는 `존재 확인`, `파라미터 입력`, `증빙 링크`, `장애물 후보 탐지`까지만 룰 후보로 등록한다. |
| 근거 | BF 공식 업데이트의 장애인등편의법, 시행령 별표, 시행규칙 별표, BF 인증 규칙, 한국장애인개발원 BF 인증제도 페이지. |
| 적용 범위 | 건축 BIM 모델 QA, 납품검수 전 자체 점검, 개발/R&D 자동화 요구사항 후보. |
| 주의 | 자동검수 결과를 인증기관 판정이나 발주처 최종 검수 결과로 표현하지 않는다. |
| 다음 액션 | QA 룰 후보를 TC로 만들고, 수동 검토 항목은 PM/RFI 또는 인증기관 확인 항목으로 넘긴다. |

## 3. 자동검수 룰 후보

| Rule ID | 룰 후보 | 확인 방식 | KST | 기대 결과 |
|---|---|---|---|---|
| BF-QA-001 | BF 검토 대상 공간/구역 태그 존재 | 지정 파라미터 또는 뷰 필터 확인 | `KST04` | 누락 후보 목록 생성 |
| BF-QA-002 | 접근로, 주차, 출입구, 복도, 승강기, 화장실, 안내설비 객체 존재 | 카테고리/패밀리/공간 매핑 확인 | `KST04` | 항목별 존재/미존재 리포트 |
| BF-QA-003 | BF 관련 파라미터 입력 여부 | 빈값, TBD, 00000, 미정 값 탐지 | `KST04` | 입력 누락 또는 더미 값 목록 |
| BF-QA-004 | 도면 번호, 모델 뷰, BCF, RFI, 체크리스트 링크 존재 | 링크 필드 또는 문서 참조 확인 | `KST04` | 증빙 누락 목록 |
| BF-QA-005 | 문, 복도, 화장실, 접근로 주변 장애물 후보 | 간섭/거리/공간 점유 후보 탐지 | `KST04` | 수동 확인 필요 이슈 생성 |
| BF-QA-006 | 예비/본인증 문서 연결 여부 | 인증 단계 필드와 문서 링크 대조 | `KST03` | 인증 단계 미확정 경고 |

## 4. 자동검수에서 제외할 항목

| 제외 항목 | 제외 이유 | 후속 처리 |
|---|---|---|
| BF 대상시설 여부 최종 판정 | 시설 용도, 규모, 공공/민간, 발주처 요구 확인 필요 | PM/RFI |
| 의무/권장 편의시설 구분 | 시행령/시행규칙 별표와 프로젝트 조건 대조 필요 | 설계자/전문가 검토 |
| 예비인증/본인증 단계별 요구 차이 | 실제 인증 문서와 심사 단계 확인 필요 | 인증기관 또는 BF 담당자 확인 |
| 지자체 사전검사 조건 | 관할 조례와 실무 해석 차이 가능 | 지자체/발주처 질의 |
| 인증 통과 여부 | 모델 형상만으로 확정 불가 | 심사 문서와 공식 회신 연결 |

## 5. 테스트 케이스 샘플

| TC ID | 시나리오 | 입력 조건 | 기대 결과 | 결과 상태 |
|---|---|---|---|---|
| TC-BF-QA-001 | BF 파라미터 누락 모델 | 대상 공간에 BF 필드가 비어 있음 | 누락 목록과 위치 뷰 출력 | QA_FAIL |
| TC-BF-QA-002 | 증빙 링크 없는 체크 항목 | BF 검토표는 있으나 RFI/도면 링크 없음 | 증빙 누락 경고 | QA_PASS_WITH_NOTES |
| TC-BF-QA-003 | 자동 치수 PASS 모델 | 문/복도/화장실 치수 후보가 기준 후보 범위 안에 있음 | `자동검수 후보 PASS`, 최종 적합 판정 보류 | QA_PASS_WITH_NOTES |
| TC-BF-QA-004 | 대상성 미확정 모델 | 시설 용도와 인증 단계가 미입력 | `KST03 적용주의` 경고, PM 확인 요청 | QA_BLOCKED |
| TC-BF-QA-005 | 인증 통과 보장 문구 포함 | 보고서에 인증 통과 보장 표현 존재 | 차단 이슈 생성 | QA_FAIL |

## 6. QA 결과 상태 코드

| 상태 | 의미 | 다음 단계 |
|---|---|---|
| QA_PASS | 자동검수 후보 항목이 모두 입력되고 증빙 링크가 있다 | 납품검수팀 수동 검토로 전달 |
| QA_PASS_WITH_NOTES | 자동검수는 통과했지만 KST03 주의가 남아 있다 | PM/RFI 또는 전문가 확인 병행 |
| QA_BLOCKED | 대상성, 인증 단계, 지자체 조건 등 필수 판단 정보가 없다 | 요구사항분석/PM으로 반환 |
| QA_FAIL | 누락, 더미 값, 인증 통과 보장 문구, 증빙 미비가 있다 | 모델러/개발/R&D 수정 요청 |

## 7. 보고서 문구 샘플

```text
본 BF 자동검수 결과는 모델 내 객체 존재, 파라미터 입력, 증빙 링크, 장애물 후보를 확인한 QA 결과입니다.

본 결과는 BF 적합 판정, 인증기관 심사 결과, 발주처 최종 납품 합격을 의미하지 않습니다. 대상시설 여부, 의무/권장 구분, 예비인증/본인증 단계, 지자체 사전검사 조건, 특기시방서 해석은 별도 수동 검토와 공식 회신으로 확정해야 합니다.
```

## 8. 개발/R&D 전달 요구사항

- 룰 결과는 `PASS`, `PASS_WITH_NOTES`, `BLOCKED`, `FAIL`로 구분한다.
- `KST03` 항목은 자동 합격 처리하지 않고 확인 요청 이슈를 생성한다.
- `KST04` 기반 치수 후보는 설정 파일에 출처와 확인일을 함께 저장한다.
- 인증 통과 보장 문구가 보고서에 들어가면 QA_FAIL로 처리한다.
- 보고서에는 모델 뷰, 요소 ID, 도면 번호, RFI/BCF 링크를 함께 남긴다.

## 9. 리뷰 기준

Pass:
- 자동검수 후보와 수동 검토 항목이 분리돼 있다.
- `KST03`과 `KST04`의 한계가 보고서 문구에 보인다.
- QA 결과 상태가 다음 액션으로 연결된다.
- 인증 통과 또는 납품 합격을 자동검수 결과로 단정하지 않는다.

Revise:
- 자동검수 PASS를 BF 적합으로 표현했다.
- 자동 수집 치수를 차단 기준으로 사용했다.
- 대상성, 인증 단계, 지자체 조건을 룰 하나로 확정하려 했다.
- 증빙 링크와 담당자 후속 액션이 없다.
