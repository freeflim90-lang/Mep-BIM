**Q: Dynamo 그래프가 특정 모델에서만 실패하면 무엇을 확인하나요?**
A: 노드 오류보다 입력 데이터 구조, 카테고리 존재 여부, 파라미터명, Revit 버전, 패키지 버전을 먼저 확인한다. 모델별 예외는 입력 검증 노드로 흡수한다.


실무 보강 (2026-06-20 답변 품질 보강):
1. 기준 확인: `Dynamo` 담당자는 이 질문을 단순 설명이 아니라 KST02 이상 운영 기준, 프로젝트 범위, 고객 영향도, 납품/계약 책임을 함께 확인하는 실무 판단 문제로 본다.
2. 조건 분기: 확정 기준, 현장 예외, 고객 요청, 내부 승인 여부를 먼저 나눈다. 수치나 법규가 필요한 경우에는 시행일, 적용 대상, 원문 출처, LOD 300/350 같은 모델 상세 수준을 확인하기 전까지 단정하지 않는다.
3. 다음 액션: 24시간 안에 현재 자료와 누락 자료를 정리하고, 7일 안에 담당자·기한·검증 방법이 있는 조치 항목으로 바꾼다. 반복 문의는 QA로 남기고 2회 이상 반복되면 KB 본문 승격 후보로 올린다.
4. 리스크 경계: 비용, 일정, 안전, 개인정보, 법무, 고객 약속으로 번질 수 있는 내용은 즉답보다 확인 로그와 승인 경로를 우선한다. 불확실한 답은 '가능/불가'보다 확인 조건과 대안 2개를 함께 제시한다.
5. 답변 형식: 결론 1문장, 근거 2개, 확인할 자료 3개, 다음 행동 1개 순서로 응답한다. Source: LUA BIM LABS agent QA quality baseline. Tags: qa,quality,field-case,kst02,risk,2026.

## 2026-06-21 Dynamo 자동화 운영 추가 Q&A
- Source: LUA BIM LABS coverage hardening
- Tags: dynamo,revit,automation,package,parameter,2026

**Q: Dynamo 자동화가 실행은 되지만 결과 수량이 맞지 않으면 무엇을 확인하나요?**
A: 결론: 그래프 성공 여부보다 입력 필터, 카테고리, 파라미터명, 단위, 제외 조건을 먼저 확인한다. 1. Revit 버전, Dynamo 버전, 패키지 버전, 대상 뷰/워크셋, LOD 300/350 범위를 기록한다. 2. 조건상 수량표와 모델 카테고리가 다르면 필터 오류로 보고, 반면 Shared Parameter GUID가 바뀐 경우에는 템플릿 관리 이슈로 분류한다. 수량 오류는 비용, 납품, 책임 리스크로 이어지므로 24시간 내 자동화 담당자가 샘플 10개 요소 ID를 검토하고 7일 내 수정 그래프, 담당자, 기한, 승인 로그를 공유한다.

**Q: Dynamo 패키지 업데이트 후 그래프가 깨지면 바로 최신 패키지로 고정하나요?**
A: 결론: 바로 고정하지 않고 프로젝트 표준 패키지 버전과 회귀 테스트 결과를 먼저 본다. 1. 기존 패키지 버전, 신규 버전, Revit 빌드, 노드 변경 내역, 입력/출력 스키마를 비교한다. 2. 조건상 보안 패치나 필수 오류 수정이면 24시간 내 검토 후 승인 후보로 두고, 반면 결과값이 바뀌면 이전 버전 롤백 또는 별도 브랜치로 분리한다. Autodesk/Revit/Dynamo 환경과 BEP 자동화 기준을 근거로 남긴다. 실패를 방치하면 모델 손상, 수량 오류, 일정 리스크가 있으므로 담당자, 기한, 테스트 로그, 승인 상태를 기록한다. 7일 내 자동화 담당자가 재현 파일을 공유하고, 수정 그래프를 검토·보고한 뒤 배포 버전과 롤백 방법을 정리한다.

## 2026-06-21 Dynamo 그래프 배포 추가 Q&A
- Source: LUA BIM LABS coverage expansion
- Tags: dynamo,revit,automation,deployment,qa,2026

**Q: Dynamo 그래프를 고객 프로젝트에 배포하기 전 최소 테스트는 무엇인가요?**
A: 결론: 샘플 모델 1개 실행만으로 배포하지 말고 입력 검증, 결과 수량, 롤백, 로그 저장까지 확인한다. 1. Revit 버전, Dynamo 버전, 패키지 버전, 대상 카테고리, Shared Parameter GUID, LOD 300/350, Source/Tags를 기록한다. 2. 조건상 읽기 전용 검토 그래프인 경우 샘플 10개 요소로 검증하고, 반면 모델을 수정하는 그래프이면 백업 모델과 승인자 확인 후 실행한다. 미검증 배포는 모델 손상, 수량 오류, 일정 지연, 법무 책임 리스크를 만든다. 24시간 내 자동화 담당자가 테스트 로그를 공유하고 7일 내 승인권자, 기한, 배포/롤백 로그를 남긴다.

**Q: Dynamo 자동화 결과가 담당자마다 다르게 나오면 무엇을 표준으로 삼나요?**
A: 결론: 실행자 의견보다 동일 입력, 동일 버전, 동일 필터 조건에서 재현되는 결과를 표준으로 삼는다. 1. 실행 모델 Revision, Revit/Dynamo/패키지 버전, 입력 엑셀, 대상 뷰, 워크셋, BEP/EIR, Source/Tags를 확인한다. 2. 조건상 버전 차이인 경우 표준 환경으로 재실행하고, 반면 입력 데이터가 다르면 입력 검증 노드를 추가한다. 재현성 없는 결과는 납품 오류, 비용 산정 오류, 고객 승인 리스크를 만든다. 24시간 내 담당자가 재현 절차를 기록·공유하고 7일 내 검토자, 기한, 수정 그래프 로그를 정리한다.

## 2026-06-21 Dynamo 유지보수 추가 Q&A
- Source: LUA BIM LABS coverage expansion
- Tags: dynamo,revit,maintenance,exception,version,2026

**Q: Dynamo 그래프가 오래되어 노드 경고가 많지만 아직 실행되면 교체해야 하나요?**
A: 결론: 실행 여부만 보지 말고 경고 원인, 버전 호환성, 결과 재현성, 모델 수정 리스크를 기준으로 유지보수한다. 1. Revit/Dynamo/패키지 버전, Deprecated 노드, 대상 카테고리, Shared Parameter GUID, LOD 300/350, Source/Tags를 확인한다. 2. 조건상 읽기 전용 그래프이고 결과가 재현되면 일정에 맞춰 개선하고, 반면 모델을 수정하거나 수량 산출에 쓰이면 즉시 교체 또는 롤백 후보로 올린다. 우선 실행 가능 그래프와 납품 영향 그래프를 구분하고 다만 고객 모델 수정용은 예외 없이 승인 테스트를 거친다. 방치하면 모델 손상, 수량 오류, 일정 지연, 고객 승인 리스크가 생긴다. 24시간 내 자동화 담당자가 경고 목록과 샘플 결과를 기록·공유하고 7일 내 승인자, 기한, 수정 그래프 로그를 보고한다.
