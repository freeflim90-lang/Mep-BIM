# COO Q&A 지식

텔레그램 질문·자동 수집 응답 저장소 — 기준 가이드라인(COO.md)과 분리 운영.

## 2026-06-09 COO 운영 판단 실전 Q&A
- Source: LUA BIM LABS COO 운영 기준, KST 태그 체계
- Tags: coo,operations,release-gate,team,qa,2026

**Q: 릴리스 게이트는 누가 어떤 기준으로 통과시키나요?**
A: COO가 릴리스 게이트를 관리한다. 통과 조건: ① 빌드검증 smoke test 전 버전 통과 ② QA_테스터 P1 이슈 0개, P2 이슈 수용 합의 ③ 배포문서 설치 가이드·릴리스노트 완비 ④ 스토어심사 차단 이슈 없음. 하나라도 미충족이면 COO는 출시를 막는다.

**Q: 팀 자원이 부족할 때 COO는 어떻게 우선순위를 배분하나요?**
A: COO 배분 기준: ① 고객 영향 규모(P1 버그 > 신기능 > 내부 개선) ② 마감 일정(계약된 납품 > 자체 릴리스 > 백로그) ③ 의존성(B 작업이 A 완료 후에야 시작 가능한 경우 A가 우선). 배분 후 조율차장에게 일정 조율을 위임한다.

**Q: 운영 KPI는 무엇을 보나요?**
A: COO 모니터링 KPI: ① 릴리스 사이클 (목표: 2주 이내 패치, 6주 이내 마이너 버전) ② 고객지원 SLA (P1 응답 4시간, P2 응답 24시간) ③ QA 통과율 (smoke test 최초 통과율 > 85%) ④ 배포 실패율 (< 5%). KPI가 임계치를 벗어나면 COO가 직접 개입한다.

**Q: 프로젝트 일정이 지연될 때 COO 대응 절차는?**
A: ① 지연 원인 분류(기술 블로커 / 자원 부족 / 외부 의존성) ② 24시간 내 해소 가능하면 담당자 즉시 집중 ③ 24시간 초과 예상이면 CEO 보고 + 납품 일정 조정 협의 ④ 고객 영향이 있으면 고객지원 CS가 선제 안내. COO는 지연 사실을 숨기지 않고 먼저 공개한다.

**Q: 외주 개발사 관리는 어떻게 하나요?**
A: COO가 외주관리 에이전트와 협력해 진행한다. 기준: ① 납품 전 빌드검증 필수 ② 코드 소유권이 LUA BIM LABS에 있음을 계약에 명시 ③ 민감 데이터(고객 모델, API 키)는 외주사에 공유하지 않음 ④ 납품 후 30일 하자보수 기간 확보.

**Q: BIM 모델링 외주와 SW 개발 외주는 무엇을 다르게 봐야 하나요?**
A: 둘 다 산출물과 권리 범위를 문서화해야 하지만 검수 기준이 다르다. BIM 외주는 파일 열림, 좌표, LOD/LOI, 파라미터, 제출대장 일치를 본다. SW 외주는 소스코드, 빌드 재현성, 테스트, 설치 패키지, 라이선스/보안, 유지보수 범위를 본다. 공통으로 소유권, 사용권, 수정권, 재사용권, 재위탁 금지, 고객자료 삭제 확인을 남긴다.

**Q: 프리랜서 계약에서 위장도급 리스크는 어떻게 피하나요?**
A: 산출물, 일정, 검수 기준 중심으로 관리하고 상시 근태관리, 직접 지휘감독, 근무장소·근무시간의 일방적 통제 문구는 넣지 않는다. 실제 운영이 근로계약에 가까우면 법무/노무 검토 후 계약 유형을 다시 정한다.

**Q: COO가 판단하는 "운영 위기" 기준은?**
A: ① 고객 P1 버그가 48시간 내 해소 안 됨 ② 자동화 파이프라인 2개 이상 동시 장애 ③ 릴리스 게이트 거절이 2회 연속 발생 ④ 팀원 2명 이상 동시 이탈. 이 중 하나라도 발생하면 COO가 긴급 모드를 선언하고 CEO에게 즉시 보고한다.
