# 요구사항분석 Q&A 지식

텔레그램 질문·자동 수집 응답 저장소 — 기준 가이드라인(요구사항분석.md)과 분리 운영.

## 2026-06-18 RFP/SOW 요구사항 분석 Q&A
- Source: LUA BIM LABS RFP/SOW proposal compliance intake
- Tags: requirements,rfp,sow,acceptance-criteria,proposal,qa,2026

**Q: RFP나 과업지시서를 받으면 요구사항을 어떻게 쪼개나요?**
A: 문서를 조항 단위로 나누고 각 항목을 업무범위, 산출물, 성과 기준, 인수 기준, 보안/특수요건, 일정, 평가 기준으로 분류한다. 각 요구사항에는 출처 조항/페이지, 제안서 대응 위치, SOW/BEP 반영 위치, 검수 기준, 담당자를 붙인다.

**Q: BIM SOW의 수용 기준은 어떻게 써야 하나요?**
A: 측정 가능하게 쓴다. 예를 들어 “모델 품질 확보”가 아니라 “지정 Revit 버전에서 오류 없이 열림”, “공유좌표 기준 일치”, “필수 파라미터 누락 없음”, “검토 회차별 간섭 보고서 제출”, “제출대장과 파일 목록 일치”처럼 검사와 인수가 가능한 문장으로 쓴다.

**Q: 고객 요구가 제안서에는 좋아 보이지만 범위가 불명확하면 어떻게 하나요?**
A: `확인 필요`로 표시하고 RFI 또는 사전 질의에 올린다. 범위, 산출물, 검수 기준, 비용 영향이 확정되지 않은 요구사항을 제안서에 확정 약속으로 쓰지 않는다. 확정 전 표현은 “협의 후 확정”, “옵션”, “별도 견적”으로 분리한다.

## 2026-06-09 요구사항분석 실전 Q&A
- Source: LUA BIM LABS 개발 기획 운영 기준, User Story 작성 경험
- Tags: requirements,user-story,mvp,scope,qa,2026

**Q: 요구사항을 어떻게 정리해야 하나요?**
A: "As a [사용자 유형], I want [기능], So that [목적]" 형식으로 작성한다. 각 User Story에 수용 기준(Acceptance Criteria)을 붙인다: "Given [초기 조건], When [액션], Then [기대 결과]". 요구사항분석 에이전트는 이 형식으로 정리 후 프로그램개발과 견적심사원에 넘긴다.

**Q: MVP 범위와 제외 범위는 어떻게 결정하나요?**
A: 기준: ① MVP = 핵심 고통(pain) 1가지를 해결하는 최소 기능 ② 제외 범위 = "있으면 좋지만 첫 버전에 없어도 되는" 기능 ③ Pro 기능 = 추후 유료 플랜으로 이동할 기능. 판단 기준: 첫 출시 후 90일 내 고객 피드백 없이도 가치가 검증되는가. 검증 안 되는 기능은 Pro로 분류한다.

**Q: 개발 공수 산정은 어떻게 하나요?**
A: 요구사항분석이 User Story와 수용 기준을 확정하면 견적심사원이 공수를 산정한다. 요구사항분석은 ① 모호한 요구사항을 명확히 하고 ② 의존성을 파악하고 ③ 제외 범위를 명시해야 견적심사원이 정확하게 계산할 수 있다. 요구사항 변경이 있으면 견적심사원에게 재산정을 요청한다.

**Q: 요구사항 변경(Change Request)이 왔을 때 처리 방법은?**
A: ① 변경 유형 분류(기능 추가 / 기능 변경 / 버그 수정) ② 기능 추가·변경이면 범위 초과 여부 판단 ③ 범위 초과면 별도 견적 산정 후 CEO/COO 승인 ④ 긴급 버그 수정이면 COO가 릴리스 게이트 우선순위 조정 ⑤ 변경 이력을 요구사항 문서에 기록. 확인 없이 변경을 수용하지 않는다.

**Q: 외부 API 연동이 필요한 기능은 어떻게 처리하나요?**
A: 요구사항분석 단계에서 외부 API 의존성을 명시해야 한다. 체크 항목: ① API 키 비용 ② 속도 제한(rate limit) ③ 오프라인 환경에서 동작 여부 ④ 개인정보 전송 여부. 개인정보 전송이 있으면 라이선스_보안관이 필수 검토한다. 오프라인 환경 필요 시 로컬 대안 구조를 함께 제안한다.
