**Q: 라이선스 공유나 우회 사용이 의심되면 고객에게 어떻게 말하나요?**
A: 결론: 고객을 비난하지 않고 사실 확인과 정상 전환 절차를 안내한다. 1. 담당자는 계정, 기기, IP, 로그인 기록, 결제 상태, Source, Tags, KST02 기준을 24시간 안에 확인하고 보안 로그를 기록한다. 2. 조건상 공유나 우회 사용이 반복되면 기능 제한, 정상 라이선스 전환, 7일 소명 기한을 요청한다. 반면 기기 교체나 VPN 오류처럼 합리적 예외가 있으면 고객지원운영과 함께 확인한다. 다만 개인정보, 법무, 환불 책임 리스크가 있으면 예외 없이 승인 담당자와 로그를 공유한 뒤 답변한다.

**Q: 라이선스 위반 의심 로그가 있지만 고객 업무가 멈췄다면 바로 차단하나요?**
A: 결론: 즉시 차단보다 피해와 보안 리스크를 동시에 평가한다. 1. 담당자는 동시 접속 수, 24시간 로그인 패턴, 결제 이력, 고객 계약, Revit/BIM360 업무 영향, Source와 Tags를 확인해 증거 로그를 남긴다. 2. 조건상 계정 탈취, 토큰 유출, 결제 우회가 명확하면 기능 제한을 요청하고 법무 검토를 공유한다. 반면 정상 고객의 기기 변경 오류라면 임시 접근을 허용할 수 있다. 다만 예외 승인에는 담당자, 기한, 7일 재검토 로그가 반드시 필요하다.

**Q: 고객이 라이선스 정책을 몰랐다고 하면 어떤 자료를 보내나요?**
A: 결론: 제재 문구보다 정책, 사용 범위, 정상 전환 옵션을 먼저 안내한다. 1. 담당자는 사용자 수, 기기 수, 동시 접속 제한, 결제 주기, 보안 책임, 30일 전환 조건을 표로 정리해 Source와 Tags를 남긴다. 2. 조건상 교육 부족이면 고객지원운영에 안내 문서를 요청하고, 반면 반복 우회나 공유 정황이 있으면 법무와 운영 리더 승인 후 단계적 제한을 검토한다. 다만 환불, 개인정보, 계약 해지 리스크가 있으면 예외 없이 담당자와 승인 로그를 기록해 공유한다.


실무 보강 (2026-06-20 답변 품질 보강):
1. 기준 확인: `라이선스_보안관` 담당자는 이 질문을 단순 설명이 아니라 KST02 이상 운영 기준, 프로젝트 범위, 고객 영향도, 납품/계약 책임을 함께 확인하는 실무 판단 문제로 본다.
2. 조건 분기: 확정 기준, 현장 예외, 고객 요청, 내부 승인 여부를 먼저 나눈다. 수치나 법규가 필요한 경우에는 시행일, 적용 대상, 원문 출처, LOD 300/350 같은 모델 상세 수준을 확인하기 전까지 단정하지 않는다.
3. 다음 액션: 24시간 안에 현재 자료와 누락 자료를 정리하고, 7일 안에 담당자·기한·검증 방법이 있는 조치 항목으로 바꾼다. 반복 문의는 QA로 남기고 2회 이상 반복되면 KB 본문 승격 후보로 올린다.
4. 리스크 경계: 비용, 일정, 안전, 개인정보, 법무, 고객 약속으로 번질 수 있는 내용은 즉답보다 확인 로그와 승인 경로를 우선한다. 불확실한 답은 '가능/불가'보다 확인 조건과 대안 2개를 함께 제시한다.
5. 답변 형식: 결론 1문장, 근거 2개, 확인할 자료 3개, 다음 행동 1개 순서로 응답한다. Source: LUA BIM LABS agent QA quality baseline. Tags: qa,quality,field-case,kst02,risk,2026.

## 2026-06-21 라이선스 보안 운영 추가 Q&A

**Q: 고객사가 퇴사자 계정으로 계속 접속하면 어떻게 안내하나요?**
A: 결론: 바로 비난하지 않고 계정 소유권, 사용 주체, 데이터 접근 범위, 정상 전환 방법을 확인한다. 1. 계정 이메일, 마지막 로그인 24시간 기록, 기기 수, IP, 결제 상태, 고객 관리자, Source/Tags를 기록한다. 2. 조건상 관리자 실수나 인수인계 지연인 경우 7일 내 계정 이전을 안내하고, 반면 퇴사자 계정 공유가 반복되면 기능 제한과 법무 검토 후보로 올린다. 방치하면 개인정보, 보안 누수, 계약 위반, 책임 리스크가 생긴다. 라이선스 보안관은 승인권자, 기한, 전환/차단 로그를 공유·보고한다.

**Q: 라이선스 서버 오류 때문에 정상 고객이 인증 실패하면 보안 차단으로 봐야 하나요?**
A: 결론: 보안 차단으로 단정하지 않고 서버 상태, 결제 상태, 네트워크, 토큰 만료를 분리해 확인한다. 1. 오류 코드, 인증 시각, 서버 상태, 결제 갱신일, 기기 ID, VPN/IP, Source/Tags를 확인한다. 2. 조건상 서버 장애나 토큰 만료인 경우 임시 인증 또는 우회 안내를 검토하고, 반면 비정상 다중 접속이면 보안 점검으로 분류한다. 오판하면 고객 업무 중단, 환불, 법무, 신뢰 리스크가 커진다. 24시간 내 담당자가 장애 로그를 요청·기록하고 7일 내 승인자, 기한, 복구/재발방지 로그를 남긴다.
