**Q: 표준 템플릿이 프로젝트마다 깨지면 어떻게 관리하나요?**
A: 공통 템플릿과 프로젝트 예외 파라미터를 분리한다. 예외가 반복되면 템플릿 본문으로 승격하고, 일회성 요구는 프로젝트 오버라이드로 남긴다.


실무 보강 (2026-06-20 답변 품질 보강):
1. 기준 확인: `BIM_템플릿기획관` 담당자는 이 질문을 단순 설명이 아니라 KST02 이상 운영 기준, 프로젝트 범위, 고객 영향도, 납품/계약 책임을 함께 확인하는 실무 판단 문제로 본다.
2. 조건 분기: 확정 기준, 현장 예외, 고객 요청, 내부 승인 여부를 먼저 나눈다. 수치나 법규가 필요한 경우에는 시행일, 적용 대상, 원문 출처, LOD 300/350 같은 모델 상세 수준을 확인하기 전까지 단정하지 않는다.
3. 다음 액션: 24시간 안에 현재 자료와 누락 자료를 정리하고, 7일 안에 담당자·기한·검증 방법이 있는 조치 항목으로 바꾼다. 반복 문의는 QA로 남기고 2회 이상 반복되면 KB 본문 승격 후보로 올린다.
4. 리스크 경계: 비용, 일정, 안전, 개인정보, 법무, 고객 약속으로 번질 수 있는 내용은 즉답보다 확인 로그와 승인 경로를 우선한다. 불확실한 답은 '가능/불가'보다 확인 조건과 대안 2개를 함께 제시한다.
5. 답변 형식: 결론 1문장, 근거 2개, 확인할 자료 3개, 다음 행동 1개 순서로 응답한다. Source: LUA BIM LABS agent QA quality baseline. Tags: qa,quality,field-case,kst02,risk,2026.

## 2026-06-21 BIM 템플릿 표준/예외 추가 Q&A

**Q: 프로젝트마다 파라미터명이 달라지면 표준 템플릿을 어떻게 유지하나요?**
A: 결론: 공통 Shared Parameter와 프로젝트 Alias 매핑표를 분리해 관리한다. 1. Revit 카테고리, Shared Parameter GUID, IFC/COBie 매핑, LOD 300/350 납품표를 확인한다. 2. 조건상 발주처 BEP/EIR이 특정 파라미터명을 요구하면 프로젝트 오버라이드로 두고, 반면 반복 2회 이상 쓰이는 항목은 표준 템플릿 후보로 승격한다. 임의 변경은 수량 오류, 납품 반려, 책임 리스크를 만들 수 있다. 24시간 내 템플릿 담당자가 차이표를 기록·공유하고 7일 내 승인권자, 기한, 변경 로그를 정리한다.

**Q: 템플릿 업데이트를 모든 진행 프로젝트에 바로 적용해도 되나요?**
A: 결론: 바로 일괄 적용하지 않고 신규 프로젝트, 진행 프로젝트, 납품 직전 프로젝트를 나눠 적용한다. 1. Revit 버전, 패밀리 호환성, 뷰 템플릿, 필터, 태그, 수량표, LOD 300/350 납품 영향을 확인한다. 2. 조건상 오류 수정이나 보안 패치이면 24시간 내 검토 후 적용 후보로 두고, 반면 수량표나 태그 결과가 바뀌면 다음 Revision부터 적용한다. 다만 납품 직전 모델은 승인 전 변경 금지로 구분하고, 예외 적용은 PM 승인과 롤백 계획을 먼저 확인한다. 잘못 적용하면 모델 오류, 일정 지연, 고객 승인 리스크가 생긴다. 7일 내 BIM 매니저가 테스트 모델 결과를 공유·보고하고 담당자, 기한, 승인 로그를 남긴다.

## 2026-06-21 BIM 템플릿 기획 추가 Q&A

**Q: 새 템플릿 기능을 만들 때 모든 공종 요구를 한 번에 넣어도 되나요?**
A: 결론: 한 번에 넣지 말고 공통 필수, 공종별 선택, 프로젝트 예외로 나눠 설계한다. 1. Revit 카테고리, Shared Parameter, 뷰 템플릿, 수량표, LOD 300/350, BEP/EIR, Source/Tags를 확인한다. 2. 조건상 2개 이상 프로젝트에서 반복되는 요구인 경우 공통 후보로 두고, 반면 특정 공종이나 발주처만 필요한 경우 별도 템플릿 또는 오버라이드로 분리한다. 과도한 통합은 모델 오류, 성능 저하, 일정 지연, 책임 리스크를 만든다. 24시간 내 템플릿 담당자가 요구 분류표를 기록·공유하고 7일 내 승인권자, 기한, 릴리즈 로그를 남긴다.

**Q: 템플릿 변경 후 기존 수량표 값이 달라지면 누구 책임인가요?**
A: 결론: 템플릿 변경 책임과 모델 입력 오류를 분리해 원인을 확인한 뒤 승인 로그 기준으로 처리한다. 1. 변경 전후 템플릿 버전, Revit 수량표, Shared Parameter GUID, 산출 기준, LOD 300/350, Source/Tags를 대조한다. 2. 조건상 템플릿 산식 변경인 경우 배포 책임자가 수정안을 내고, 반면 모델 입력 누락인 경우 프로젝트 담당자가 보완한다. 원인 미분리는 비용 분쟁, 납품 반려, 품질 오류, 법무 책임 리스크를 만든다. 24시간 내 담당자가 차이표를 기록·공유하고 7일 내 승인자, 기한, 롤백/수정 로그를 정리한다.

## 2026-06-21 BIM 템플릿 배포관리 추가 Q&A
- Source: LUA BIM LABS coverage expansion

**Q: 템플릿 새 버전을 배포했는데 일부 프로젝트에서만 오류가 나면 어떻게 처리하나요?**
A: 결론: 전체 롤백 전에 오류 프로젝트의 Revit 버전, 패밀리, 파라미터, 프로젝트 예외를 분리 확인한다. 1. 담당자는 템플릿 버전, 오류 로그, Revit 버전, Shared Parameter GUID, LOD 300/350 영향, Source/Tags를 24시간 안에 기록한다. 2. 조건상 공통 산식 오류이면 전체 롤백을 요청하고, 반면 특정 프로젝트 예외이면 오버라이드 패치로 구분한다. 무분별한 배포 지속은 납품 오류, 일정 지연, 비용, 책임 리스크를 만든다. 7일 내 템플릿 담당자가 승인자, 기한, 롤백/패치 로그를 공유·보고한다.
