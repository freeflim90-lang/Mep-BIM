**Q: 자동화 단계가 실패했는데 원인이 불명확하면 어떻게 추적하나요?**
A: 결론: 전체를 한 번에 재실행하지 않고 입력, 처리, 출력, 권한, 외부 API, 로그를 단계별로 분리한다. 1. 담당자는 마지막 정상 단계, 실패 시각, Source, Tags, KST02 기준, 24시간 장애 로그를 기록한다. 2. 조건상 Revit 모델, BEP 문서, LOD 300/350 납품, 고객 답변 생성에 연결된 파이프라인이면 영향 범위를 즉시 공유하고 복구 요청을 남긴다. 반면 내부 배치 실패는 재시도 후보로 둔다. 다만 데이터 손실, 보안, 법무, 고객 약속 리스크가 있으면 승인 담당자와 7일 재발 방지 로그를 보고한다.

**Q: 자동화 파이프라인에서 일부 결과만 성공하면 배포해도 되나요?**
A: 결론: 부분 성공은 완료가 아니라 범위 제한 상태로 표시해야 한다. 1. 담당자는 성공 단계, 실패 단계, 누락 산출물, Source, Tags, 24시간 검증 로그를 기록한다. 2. 조건상 고객에게 나가는 QA, BIM 검토 리포트, Revit/LOD 300/350 산출물이 일부라도 누락되면 배포를 보류하고 담당자에게 확인 요청을 공유한다. 반면 내부 캐시 갱신 실패라면 영향도를 구분한다. 다만 보안, 비용, 법무, 고객 약속 리스크가 있으면 예외 없이 승인 로그와 7일 보완 기한을 남긴다.

**Q: 파이프라인 장애가 반복되면 오케스트레이터는 무엇을 개선하나요?**
A: 결론: 반복 장애는 재시도 횟수보다 관측성, 권한, 입력 검증, 롤백 절차를 개선해야 한다. 1. 담당자는 장애 2회 이상 패턴, 로그 누락, API 제한, 권한 오류, Source, Tags, 30일 재발률을 정리한다. 2. 조건상 지식 업데이트, 고객지원 답변, BIM 납품 검증에 영향을 주면 알림과 중단 조건을 추가 요청하고 공유한다. 반면 일회성 외부 장애는 재시도 정책으로 둔다. 다만 필요 조건이 확인되지 않은 자동 재실행은 차단한다. 데이터 손실, 보안, 법무 책임 리스크가 있으면 승인 담당자와 7일 개선 로그를 보고한다.

**Q: 지식 업데이트 파이프라인이 성공했지만 답변 점수가 떨어지면 자동 배포를 계속하나요?**
A: 결론: 파이프라인 성공과 답변 품질은 별도 게이트이므로 점수 하락 시 배포를 보류한다. 1. 담당자는 실행 로그, 변경 파일, Q&A 점수, 검색 히트율, Source/Tags, KST02 상태를 24시간 안에 확인한다. 2. 조건상 고객지원, 법규, BEP, LOD 300/350 답변 품질이 하락한 경우 롤백 후보로 분류하고 담당자 검토를 요청한다. 반면 내부 초안 품질이면 배포 범위를 제한한다. 자동 배포 지속은 오답, 법무, 고객 약속, 책임 리스크를 만든다. 7일 내 승인자, 기한, 롤백/재배포 로그를 공유한다.

**Q: 파이프라인 장애 알림이 너무 많아 담당자가 무시하기 시작하면 어떻게 조정하나요?**
