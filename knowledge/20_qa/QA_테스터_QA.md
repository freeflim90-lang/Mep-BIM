**Q: 버그 보고서는 어떻게 작성해야 하나요?**
A: 버그 보고서 필수 항목: ① 버그 제목 ② 재현 환경(OS, Revit 2023~2026, .NET 버전) ③ 재현 절차(step-by-step) ④ 기대 동작 ⑤ 실제 동작 ⑥ 첨부 자료(스크린샷, 로그 파일, 예외 메시지 전문) ⑦ 우선순위(P1/P2/P3) ⑧ 영향 범위와 회피 방법이다. Revit 에러 로그는 `%LocalAppData%\Autodesk\Revit\Autodesk Revit 20xx\Journals\` 폴더에서 확인한다. 조건상 P1은 24시간 내 담당자 지정과 임시 회피 안내가 필요하고, 7일 내 수정/보류/재현불가 판정을 기록한다. 개인정보, 고객 모델, 라이선스 키가 로그에 있으면 마스킹 후 공유해야 하며, 보안·법무·고객 신뢰 리스크 때문에 재현 불가 상태로 고객에게 단정 답변하지 않는다.

**Q: 회귀 테스트 케이스는 어떻게 선별하나요?**
A: 회귀 테스트 케이스 선별 기준: ① 과거 P1 버그 수정 항목 (반드시 포함) ② 핵심 기능 경로 (Happy Path) ③ 경계 조건 (빈 모델, 대형 모델, 특수 문자 포함 파일명) ④ 권한 조건 (라이선스 만료, 오프라인 환경). 신규 기능 추가 시 관련 기존 케이스를 먼저 재수행한다.

**Q: 고객이 보고한 버그를 재현할 수 없을 때는?**
A: 결론: 재현 실패를 종료로 보지 말고 증거 수집 상태로 분류한다. ① 고객에게 추가 정보 요청(Revit 2023~2026 버전, 모델 파일 크기, OS 정보, Journal 로그) ② 재현 환경 변수를 좁혀 격리 시도 ③ 고객 모델 파일은 민감 데이터가 포함될 수 있으므로 직접 공유 대신 최소 재현 파일 요청 ④ 재현 실패 상태로 장기 방치하지 않고 24시간 내 접수 확인, 72시간 내 진행 상황 안내, 7일 내 재현/보류/추가자료 요청 상태를 기록한다. 조건상 P1 장애나 데이터 손상 가능성이 있으면 QA_테스터 단독 판단이 아니라 프로그램개발, CS_기술지원관, 고객지원CS와 같이 처리한다. 개인정보·고객 모델 유출 리스크가 있으므로 로그는 마스킹 후 공유한다.
다만 예외 조건이 있거나 승인권자 판단이 필요한 경우에는 확정 답변으로 닫지 말고, 근거·담당자·기한을 기록해 공유한다.

**Q: QA 테스터가 빌드검증에 넘기는 항목과 직접 처리하는 항목의 차이는?**
A: QA_테스터 직접 처리: 기능 테스트, 재현 절차 작성, 케이스 문서화, P1/P2 분류. 빌드검증에 인계: 버전별 smoke test 수행, 빌드 산출물 검증, 릴리스 게이트 판단. QA_테스터는 기능 정확성에 집중하고, 빌드검증은 배포 가능 여부 판단에 집중한다.

**Q: 모델 QA 상품에서 자동 검사와 전문가 검토의 경계를 고객에게 어떻게 설명해야 하나요?**
A: 결론: 유료 QA 고객에게는 자동 검사는 "반복 가능한 규칙 검사", 전문가 검토는 "프로젝트 맥락 판단"이라고 나눈다. 자동 검사는 1. Hard Clash 0건 목표, 2. LOD 300/350 속성 누락, 3. 명명 규칙, 4. IFC/BCF 내보내기, 5. 시트/뷰/워크셋 위생처럼 규칙화 가능한 항목을 맡는다. 전문가 검토는 안전, 시공성, 유지관리, 계약 범위, 발주처 EIR/BEP 해석처럼 조건 분기가 필요한 영역이다. 보고서는 자동 점수 %, Critical/Major/Minor 등급, 재현 단계, 전문가 코멘트, 고객 승인 필요 항목을 분리한다. 다음 액션은 24시간 내 자동 검사 결과와 전문가 확인 필요 항목을 공유하고, 고객 승인 전에는 최종 책임 판단으로 안내하지 않는 것이다. 경계 문구는 "자동 검사는 후보를 줄이고, 최종 책임 판단은 전문가 검토와 고객 승인으로 확정합니다"로 둔다. 이 경계를 흐리면 품질 보증 오해와 법적 책임 위험이 생긴다.
