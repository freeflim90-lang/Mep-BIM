**Q: 문서는 정확한데 고객이 따라 하지 못하면 무엇을 바꿔야 하나요?**
A: 개념 설명을 줄이고 오류별 행동 순서를 앞에 둔다. "왜 그런지"보다 "지금 어디를 클릭하고 무엇을 확인할지"를 먼저 쓴다. 긴 문서는 참조용으로 두고, CS 답변에는 3단계 체크리스트를 제공한다.


실무 보강 (2026-06-20 답변 품질 보강):
1. 기준 확인: `테크니컬_라이터` 담당자는 이 질문을 단순 설명이 아니라 KST02 이상 운영 기준, 프로젝트 범위, 고객 영향도, 납품/계약 책임을 함께 확인하는 실무 판단 문제로 본다.
2. 조건 분기: 확정 기준, 현장 예외, 고객 요청, 내부 승인 여부를 먼저 나눈다. 수치나 법규가 필요한 경우에는 시행일, 적용 대상, 원문 출처, LOD 300/350 같은 모델 상세 수준을 확인하기 전까지 단정하지 않는다.
3. 다음 액션: 24시간 안에 현재 자료와 누락 자료를 정리하고, 7일 안에 담당자·기한·검증 방법이 있는 조치 항목으로 바꾼다. 반복 문의는 QA로 남기고 2회 이상 반복되면 KB 본문 승격 후보로 올린다.
4. 리스크 경계: 비용, 일정, 안전, 개인정보, 법무, 고객 약속으로 번질 수 있는 내용은 즉답보다 확인 로그와 승인 경로를 우선한다. 불확실한 답은 '가능/불가'보다 확인 조건과 대안 2개를 함께 제시한다.
5. 답변 형식: 결론 1문장, 근거 2개, 확인할 자료 3개, 다음 행동 1개 순서로 응답한다. Source: LUA BIM LABS agent QA quality baseline. Tags: qa,quality,field-case,kst02,risk,2026.

## 2026-06-21 기술문서 고객 적용 추가 Q&A

**Q: BIM 기능 설명서가 개발자 기준으로 쓰여 있으면 고객용으로 어떻게 바꾸나요?**
A: 결론: 내부 구조 설명보다 고객 작업, 입력값, 결과 확인, 오류 대응 순서로 다시 쓴다. 1. 담당자는 대상 사용자, Revit/BIM360 화면명, 입력 파일, 기대 결과, Source/Tags, KST02 기준을 24시간 안에 확인한다. 2. 조건상 LOD 300/350 납품이나 BEP 검수에 연결되면 검수 전 체크리스트와 실패 시 조치를 추가한다. 반면 내부 API 설명은 개발 문서로 분리한다. 개발자 용어를 그대로 두면 고객 오해, 지원 비용, 법무, 책임 리스크가 생긴다. 7일 내 승인자, 담당자, 기한, 문서 검토 로그를 공유한다.

**Q: 고객 안내 문서에 제한사항을 쓰면 제품이 약해 보이지 않나요?**
A: 결론: 제한사항은 약점이 아니라 오사용을 막는 안전장치로 고객 행동 기준과 함께 써야 한다. 1. 담당자는 지원 범위, 미지원 조건, 오류 메시지, 대체 절차, Source/Tags, 적용 버전을 확인한다. 2. 조건상 법규, 보안, 비용, 납품 품질에 영향을 주는 경우 24시간 내 제한사항과 승인 필요 조건을 명시하고 검토를 요청한다. 반면 단순 편의 기능 제한은 FAQ로 안내한다. 제한 누락은 고객 약속, 법무, 브랜드, 책임 리스크를 만든다. 7일 내 승인자, 기한, 릴리스 노트 반영 로그를 공유·보고한다.

**Q: 릴리스 노트에서 내부 이슈를 어디까지 공개하나요?**
A: 내부 이슈 번호나 개발자 용어는 숨기고, 사용자 영향과 조치만 쓴다. 예를 들어 "NullReferenceException 수정"보다 "일부 모델에서 대시보드가 열리지 않던 문제 수정"처럼 고객 언어로 바꾼다.
