# QA - 적용했을때의 장단점이 궁금해

## 질문
> **카테고리:** Claude대화 | **날짜:** 2026-05-30
> **출처:** Claude Code 세션

적용했을때의 장단점이 궁금해

## 답변

현재 프로젝트 기준으로 장단점을 정리하면:

**장점**

- **환경 일관성** — Python 버전, 패키지 의존성이 컨테이너 안에 고정되어 "내 맥에서만 됨" 문제 없음
- **서비스 격리** — 백엔드 서버, 크론 스크립트, Cloudflare 터널이 서로 독립적으로 실행
- **재시작 자동화** — `restart: always` 한 줄로 서버 다운 시 자동 재기동 (현재 LaunchAgents보다 관리 편함)
- **배포 이식성** — 나중에 VPS나 클라우드로 옮길 때 그대로 가져갈 수 있음
- **로그 통합** — `docker logs` 로 서비스별 로그 한 곳에서 확인

**단점**

- **초기 설정 비용** — Dockerfile + `docker-compose.yml` 작성에 시간 필요 (프로젝트 규모상 반나절 정도)
- **현재 LaunchAgents 이전** — 기존 자동화 스크립트들을 컨테이너 방식으로 재구성해야 함
- **Mac 리소스** — 가상화 레이어가 있어 네이티브보다 약간 느림 (OrbStack은 이 부분을 많이 개선했음)
- **파일 경로 변경** — NAS 마운트나 로컬 경로를 볼륨으로 다시 매핑해야 함

**결론:** 개인 서버로 24시간 돌릴 거라면 장기적으로 이득이 큽니다. 단기 부담은 초기 설정뿐이에요.

진행해볼까요?
