## 2026-06-09 QA 테스트 실전 Q&A
- Source: LUA BIM LABS QA 운영 기준, Revit Add-in 테스트 경험
- Tags: qa,testing,bug-report,regression,smoke-test,2026

**Q: 버그 보고서는 어떻게 작성해야 하나요?**
A: 버그 보고서 필수 항목: ① 버그 제목 (한 줄 요약) ② 재현 환경 (OS, Revit 버전, .NET 버전) ③ 재현 절차 (step-by-step) ④ 기대 동작 ⑤ 실제 동작 ⑥ 첨부 자료 (스크린샷, 로그 파일, 예외 메시지 전문) ⑦ 우선순위 (P1/P2/P3). Revit 에러 로그는 `%LocalAppData%\Autodesk\Revit\Autodesk Revit 20xx\Journals\` 폴더에서 확인한다.

**Q: 회귀 테스트 케이스는 어떻게 선별하나요?**
A: 회귀 테스트 케이스 선별 기준: ① 과거 P1 버그 수정 항목 (반드시 포함) ② 핵심 기능 경로 (Happy Path) ③ 경계 조건 (빈 모델, 대형 모델, 특수 문자 포함 파일명) ④ 권한 조건 (라이선스 만료, 오프라인 환경). 신규 기능 추가 시 관련 기존 케이스를 먼저 재수행한다.

**Q: P1과 P2 버그를 어떻게 분류하나요?**
A: P1 (릴리스 차단): 핵심 기능 실행 불가 / Revit 충돌(crash) / 고객 데이터 손실 / 라이선스 인증 실패 / 설치 불가. P2 (중요, 협의 필요): 비핵심 기능 오동작 / UI 표시 오류 / 성능 저하 (정상 대비 3배 이상) / 특정 Revit 버전에서만 발생. P3 (보완): 사소한 UI 개선 / 문서 오타.

**Q: 고객이 보고한 버그를 재현할 수 없을 때는?**
A: ① 고객에게 추가 정보 요청 (Revit 버전, 모델 파일 크기, OS 정보, Journal 로그) ② 재현 환경 변수를 좁혀 가며 격리 시도 ③ 고객 모델 파일은 민감 데이터가 포함될 수 있으므로 직접 공유 대신 최소 재현 파일(가능한 경우) 요청 ④ 재현 실패 상태로 장기 방치하지 않고 72시간 후 고객에게 진행 상황 안내. 고객지원 CS와 협력해 고객 커뮤니케이션을 처리한다.

**Q: QA 테스터가 빌드검증에 넘기는 항목과 직접 처리하는 항목의 차이는?**
A: QA_테스터 직접 처리: 기능 테스트, 재현 절차 작성, 케이스 문서화, P1/P2 분류. 빌드검증에 인계: 버전별 smoke test 수행, 빌드 산출물 검증, 릴리스 게이트 판단. QA_테스터는 기능 정확성에 집중하고, 빌드검증은 배포 가능 여부 판단에 집중한다.
