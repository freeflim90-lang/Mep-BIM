릴리스 게이트 기준:
□ 빌드 검증: 지원 Revit 버전 전체 smoke test 통과
□ QA: P1/P2 버그 0건, P3 이하 알려진 이슈 문서화
□ 문서: 설치 가이드, 릴리스 노트, 스토어 설명 최신화
□ 법무: EULA, 개인정보 정책 검토 완료
□ 가격/결제: 구독 플랜 및 Entitlement 연동 테스트 완료
운영 KPI: 설치 성공률 95%↑, 지원 문의 응답 SLA 준수율 90%↑, 1개월 이내 환불률 5%↓.


## 개발-출시-지원 사이클 관리 (2026-05-19 17:26:40)
- Source: LUA BIM LABS domain knowledge baseline 2026-05-19
- Tags: operations,delivery,lifecycle

Sprint 주기: 2주. 스토어 제출 주기: 분기 1회 이상.
개발 완료 → 내부 QA(1주) → 빌드 검증(2~3일) → 스토어 제출(심사 1~2주) → 출시.
긴급 패치: P1 버그 발생 시 핫픽스 브랜치 → 긴급 빌드 검증 → 스토어 업데이트 제출(심사 단축 요청 가능).
고객 피드백 → 제품 백로그 → 분기 로드맵 업데이트 사이클 운영.


## COO 운영 페르소나 업무 기준 (2026-05-26)
- Source: LUA BIM LABS 조직 역할 정의 2026-05-26
- Tags: persona,role,workflow,coo

**프로젝트 운영 상태 판단 기준:**
- 일정 지연 허용 범위: Sprint 내 3일 이내 지연은 내부 조정, 1주 초과 지연은 COO 보고 및 원인 분석 필수.
- 리소스 과부하 기준: 담당자 주간 실효 작업 시간 45시간 초과 시 재배분 검토, 50시간 초과 시 즉시 조율 개입.
- 복수 프로젝트 병행 시 한 담당자가 3개 이상 주요 프로젝트에 동시 배정되면 COO 승인 필요.
- 프로젝트 범위 변경(Scope Change) 발생 시 24시간 이내 COO에게 영향도 보고.

**납품물 품질 게이트 운영 기준:**
- 게이트 1 (개발 완료): 코드 리뷰 완료 + 단위 테스트 통과율 90%↑
- 게이트 2 (내부 QA 통과): P1/P2 버그 0건, 지원 Revit 버전 전체 Smoke Test 통과
- 게이트 3 (스토어 제출 전): 문서(설치 가이드/릴리스 노트/스토어 설명) 최신화 + 법무 검토 완료
- 게이트 4 (출시 후 72시간): 설치 성공률 모니터링, CS 긴급 문의 건수 확인
- 게이트 불통과 시 다음 단계 진행 금지, COO 승인 후 예외 처리 가능.

**팀 간 조율 프로세스 (개발↔QA↔스토어 제출):**
- 개발 완료 → QA 인계 시: 테스트 환경, 재현 시나리오, 알려진 이슈 목록 필수 인계 문서 작성.
- QA 완료 → 스토어 제출 인계 시: QA 통과 확인서 + 빌드 해시값 + 제출 체크리스트 공유.
- 스토어 심사 중 수정 요청 발생 시: 개발팀 48시간 이내 대응, COO 진행 상황 공유.
- 긴급 패치: P1 버그 발생 즉시 COO 알림 → 핫픽스 브랜치 → 긴급 QA(1일) → 스토어 업데이트.

**COO 일일/주간 운영 체크 항목:**
- 일일: 진행 중 프로젝트 블로커 유무, CS 긴급 문의 건수, 빌드/QA 파이프라인 상태
- 주간: Sprint 완료율, 일정 지연 항목, 리소스 과부하 현황, 게이트 통과 현황, 다음 주 주요 마일스톤

- Revit Add-in 릴리스 주기는 Autodesk의 연 1회 메이저 버전 업데이트 사이클에 맞춰 계획해야 한다. 매년 3~4월 Autodesk 신버전 출시 후 60일 이내 호환성 패치 배포가 App Store 평점 유지의 핵심이다. 2주 단위 스프린트로 마이너 패치를 운영하는 것이 품질 리스크 최소화에 효과적이다.

- QA 게이트 3단계 구조: ①개발자 자체 테스트(단위/통합) → ②QA 테스터 기능 검증(Revit 2022~2025 버전 호환성 포함) → ③스테이징 환경 베타 배포(내부 사용자 5명 이상). 각 게이트 통과 기준 미달 시 자동 릴리스 블록 처리.
- SLA 위반 시 COO가 24시간 내 루트 코즈 분석(RCA) 보고서를 작성하고, 재발 방지 대책을 다음 스프린트 백로그 최상위에 등록하는 프로세스를 표준화해야 한다.
- 운영 비용 최적화: 클라우드 인프라는 월 매출의 5~8% 이하로 유지하고, 고객 지원 자동화(FAQ 봇, 자가 해결 포털)로 인당 지원 티켓 수를 월 30건 미만으로 관리한다.

## COO 마스터급 경험 지식 (2026-05-28)
- Source: claude-code-enhanced 2026-05-28
- Tags: COO,번아웃감지,MVP협상,리모트팀,노동법,채용리스크

팀원 번아웃 조기 감지 지표: ①GitHub 커밋 빈도가 2주 연속 30% 이상 감소 ②슬랙 응답 시간이 평소 대비 2배 이상 지연(1시간→2시간 초과) ③Sprint 완료율이 80%→50% 이하로 하락 ④1:1 미팅에서 "그냥요", "괜찮아요" 단답 응답 증가. 이 지표 중 3개 이상 동시 발생하면 COO가 직접 비공식 대화(15분 체크인)를 주선하고, 다음 Sprint에서 해당 팀원의 작업량을 20% 강제 삭감한다.

크리티컬 프로젝트 일정 압박 시 MVP 범위 조정 협상: 납기 D-14 시점에서 전체 기능의 80%만으로도 고객 핵심 가치를 제공 가능한 "MVP 릴리스"를 고객에게 먼저 제안한다. 나머지 20% 기능은 후속 패치(D+30) 계약으로 분리하고, 패치 납기 지연 시 페널티 조항을 삽입하여 신뢰를 유지한다. 이때 범위 조정 합의는 반드시 이메일로 문서화해야 계약 분쟁 시 방어 근거가 된다.

리모트 팀 품질 관리: 비동기 코드 리뷰는 제출 후 24시간 이내 1차 코멘트, 48시간 이내 최종 승인 또는 반려를 SLA로 설정한다. 리뷰어가 "LGTM(Looks Good To Me)"만 남기는 패턴이 2회 이상 반복되면 COO가 리뷰 품질 미흡으로 간주하고 페어 리뷰(2명 동시) 기준으로 전환한다.

법인 규모별 채용 리스크: 상시 근로자 5인 이상부터 근로기준법 전면 적용(해고 제한·연장근로 할증·연차유급휴가)이 시작된다. 10인 이상부터는 취업규칙 작성·신고 의무가 발생하며, 미신고 시 500만 원 이하 과태료 부과된다. 4인에서 5인으로 채용 시 기존 인력의 연장근로 동의서 재작성과 취업규칙 초안 작성을 채용 발령 전에 완료해야 한다.
