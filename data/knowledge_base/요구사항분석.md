- BIM Add-in 요구사항 수집 방법론: 발주처 EIR(Employer's Information Requirements) 문서 분석 → 수급사 BEP(BIM Execution Plan) 작성 요건 도출 → Revit Add-in 기능 명세 변환의 3단계 프로세스를 적용한다. ISO 19650-2:2018 기준 정보 납품 요건(IDR)을 기능 요구사항으로 직접 매핑한다.
- 이해관계자 인터뷰 템플릿(2025): 건축사, 구조엔지니어, MEP 설계자, 현장 BIM 코디네이터, 발주처 BIM 담당자 5개 그룹별로 인터뷰 질문지를 표준화한다. "현재 워크플로우에서 가장 반복적인 작업은?", "Revit에서 수작업으로 처리하는 데이터 추출 작업이 있나요?" 형태의 문제 중심 질문이 효과적이다.
- 기능 요구사항 우선순위화(MoSCoW): Must Have(핵심 기능, Revit API 직접 연동), Should Have(편의 기능, UI 개선), Could Have(확장 기능, 외부 API 연동), Won't Have(현 버전 제외) 4단계로 분류한다. Revit 2025 API의 `DirectContext3D`, `ExternalEvent` 등 신규 API를 활용할 수 있는 Should/Could 항목을 별도 관리한다.
- AI 보조 요구사항 분석: Claude API를 활용하여 고객 미팅 녹취록 또는 이메일 텍스트에서 기능 요구사항을 자동 추출하고 GitHub Issues 형식으로 변환하는 파이프라인을 Python 3.12로 구현한다. 추출된 요구사항은 담당자 검토 후 Obsidian KB에 승격한다.
- 변경 요구사항 추적 관리: GitHub Projects 또는 Jira를 사용하여 요구사항 ID(REQ-001 형식) → 설계 문서 → 코드 커밋 → 테스트 케이스를 양방향으로 추적한다. 요구사항 변경 시 영향 범위(Impact Analysis)를 자동으로 산출하는 스크립트를 유지한다.

REQ 작성 규칙:
```yaml
req:
  id: REQ-001
  source_group: 성장전략그룹
  owner_group: R&D_개발지원그룹
  user_story: "BIM QA 담당자로서 필수 파라미터 누락을 자동 검출하고 싶다."
  inputs: ["RVT model", "required parameter rule json"]
  outputs: ["Excel report", "Revit element id list"]
  api_dependency: "Revit API read-only"
  risk: "linked model parameter mismatch"
  test_case: "TC-FUNC-REQ-001"
```

요구사항이 `api_dependency`, `test_case`, `excluded_scope`를 갖지 않으면 개발 착수 전 보류한다.

- Source: claude-code-enhanced 2026-05-29
- Tags: 요구사항분석, BIM AddIn, 비정형건물요구사항, 실패패턴, 범위크리프

### 요구사항 분석 실패 패턴 5가지 (BIM Add-in 개발 맥락)

**1. "있으면 좋겠는데요" 요구사항 → 범위 크리프**
- 원인: 고객의 비공식 요청을 REQ 항목으로 등록하지 않고 개발 착수
- 증상: 납품 직전 "원래 이거 해달랐잖아요" → 추가 개발 요구
- 해결: 모든 요구사항 GitHub Issues 등록 의무화, 구두 요청은 서면 확인 후 등록

**2. Revit 버전 미명시 요구사항**
- 원인: "Revit에서 동작해야 함" 기재 → 개발팀이 2025 버전 기준 개발 → 고객 Revit 2022 사용
- 증상: ElementId.IntegerValue deprecated (2024+ 에러), .NET 버전 충돌
- 해결: 요구사항 템플릿에 `지원_Revit_버전: [2022, 2023, 2024, 2025]` 필수 항목 포함

**3. 비정형 건물 특수 요구사항 누락**
- 원인: 일반 건물 BIM Add-in 요구사항을 비정형 건물에 그대로 적용
- 증상: 곡면 요소 선택 API 오류, 패널 수량 산출 불가
- 해결: 비정형 건물 Add-in 요구사항에 명시:
  - `DirectShape` 요소 처리 가능 여부
  - `Adaptive Component` 지원 여부
  - 자유 곡면 면적 산출 방식 (근사값 허용 오차%)

**4. 성능 요구사항 미명시 → 배포 후 속도 불만**
- 원인: "빠르게 동작해야 함" 수준의 요구사항
- 해결: 정량 기준 명시: "10,000 요소 처리 시 30초 이내", "UI 응답 1초 이내"

**5. 다중 모델 환경 미고려**
- 원인: 단일 RVT 파일 가정으로 개발 → 실제 프로젝트 분리 모델(Linked RVT) 환경
- 해결: 요구사항에 "링크된 모델 요소 처리 여부" 명시 (`GetLinkedFileInstance`, `RevitLinkInstance` 지원)

### BIM Add-in 요구사항 표준 템플릿

```markdown
## REQ-{번호}: {기능명}

### 기본 정보
- 우선순위: 필수/선택
- 대상 Revit 버전: [2022] [2023] [2024] [2025]
- 대상 건물 유형: [정형] [비정형(경사)] [비정형(곡면)]

### 기능 설명
{사용자가 원하는 기능을 1~3문장으로}

### 입력/출력
- 입력: {Revit 요소 유형, 파라미터 목록}
- 출력: {결과물 형식 — 파라미터 갱신 / 엑셀 / 다이얼로그}

### 성능 요구사항
- 처리 대상 최대 요소 수: {N}개
- 허용 응답 시간: {X}초 이내

### 제약 조건
- [ ] Linked RVT 요소 처리 필요
- [ ] DirectShape 요소 처리 필요
- [ ] Adaptive Component 처리 필요

### 테스트 케이스
1. 정상: {조건} → {기대 결과}
2. 예외: {조건} → {기대 결과}
```

### 비정형 건물 요구사항 수집 추가 질문 목록

CS 담당자가 비정형 건물 프로젝트 고객에게 추가로 확인해야 할 항목:
1. 설계 소프트웨어: Rhino/Grasshopper 파일이 존재하는가?
2. 패널화 설계: 곡면을 평면 패널로 분할하는 작업 포함 여부
3. 납품 IFC 버전: IFC 2x3 vs IFC 4 (곡면 엔티티 지원 차이)
4. 수량 산출 허용 오차: 곡면 면적 근사 계산 오차 ±X% 허용 여부
5. Dynamo 스크립트 납품 포함 여부
