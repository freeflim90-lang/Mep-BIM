## 로컬 Qwen Coder 7B 1차 개발 운영 기준 (2026-05-20)
- Source: LUA BIM LABS local development policy
- Tags: local-coder,qwen,development,validation

개발 업무지시가 들어오면 조율차장은 먼저 목적, 성공 기준, 제외 범위, 검증 방법을 plan으로 확정한 뒤 Autodesk API 의존 여부를 판정한다. FastAPI, 프론트엔드, 문서, 테스트 코드, 순수 C# 도메인 로직처럼 Revit/Navisworks 실행 환경이 필요 없는 작업은 `Qwen_Coder_8B`가 로컬에서 1차 구현 초안을 작성한다.

Qwen의 산출물은 최종 코드가 아니라 프로그램개발 검토용 초안이다. 프로그램개발은 diff, 영향 범위, 예외 처리, 보안/라이선스 영향, 테스트 가능성을 검토하고 QA_테스터와 빌드검증으로 넘긴다.

Add-in 개발은 Qwen을 주 구현자가 아니라 초안 가이드라인 담당으로 사용한다. Qwen은 명령 구조, 순수 로직 분리, 테스트 가능한 인터페이스, 문서/체크리스트 초안까지만 작성한다. Revit API 또는 Navisworks API가 필요한 작업은 Qwen이 초안/정적 검토까지만 수행하고 최고지배자가 실제 Revit/Navisworks 설치 환경에서 테스트한 뒤 확정한다. 특히 Transaction, Document/Element 변경, ClashResult, ModelItem, SearchSet, `.addin`/플러그인 매니페스트, Store 제출 산출물은 최고지배자 검증 게이트를 필수로 둔다.

검증 단계에서도 Qwen을 먼저 사용한다. Qwen은 로컬 단위 테스트, 정적 검토, 샘플 데이터 검증, 파일 생성 검증을 수행하고, 이 검증으로 해결되지 않거나 실제 Autodesk API/외부 API 호출이 반드시 필요한 상황인지 내부적으로 판정한다. API 활용은 이 판정 이후에만 검토한다.

권장 환경변수:
```env
LOCAL_CODER_ENABLED=true
LOCAL_CODER_PROVIDER=ollama
LOCAL_CODER_MODEL=qwen2.5-coder:7b
LOCAL_CODER_BASE_URL=http://127.0.0.1:11434
```

@workflow local_qwen_development step: Revit/Navisworks API 키워드가 감지되면 최고지배자 실기 테스트 검증 대기로 전환한다

## 프로그램 개발 Claude Code 심화 업데이트 (2026-05-28)
- Source: claude-code-enhanced 2026-05-28
- Tags: development,Revit-API,CSharp,dotnet8,Add-in,2025

- Revit API 2025 핵심 변경 사항: Revit 2025는 .NET 8로 전환되어 기존 .NET Framework 4.8 기반 코드와 바이너리 비호환이다. `IExternalApplication`, `IExternalCommand` 인터페이스는 동일하나, `Autodesk.Revit.DB.Solid.CreateFromFaceArray()` 등 일부 메서드가 Deprecated되었다. `RevitAPI.dll` 버전을 `2025.0.0.0`으로 교체하고 멀티타게팅 빌드 설정을 적용한다.
- C# 최신 기법 적용(C# 12/.NET 8): Primary Constructor, Collection Expressions, `Span<T>` 기반 메모리 효율화를 Revit Add-in 개발에 적용한다. `async/await` 패턴은 Revit의 단일 스레드 모델과 충돌하므로 `ExternalEvent` + `IExternalEventHandler` 패턴으로 비동기 작업을 처리한다.
- Revit API 핵심 패턴(2025): `FilteredElementCollector`에 `WhereElementIsNotElementType()` + `OfClass()` 체이닝으로 메모리 효율적 요소 수집; `Transaction`은 `using` 패턴으로 자동 롤백 보장; `ElementId`를 `long`으로 캐스팅하는 방식이 Revit 2024부터 표준화됨.
- 코드 품질 관리 도구: Visual Studio 2022 17.x + ReSharper 2025.x 조합으로 정적 분석을 수행한다. SonarQube Community Edition을 CI 파이프라인에 통합하여 코드 복잡도, 중복도, 보안 취약점을 자동 스캔한다. Revit Add-in 특화 규칙(단일 스레드 위반 탐지)을 커스텀 분석기로 추가한다.
- Autodesk Platform Services(APS) 연동: Revit Add-in에서 APS Design Automation API를 활용하면 클라우드에서 Revit 모델을 무인 처리할 수 있다. Python 3.12 또는 Node.js 20 기반 APS SDK로 작업 큐를 관리하고, 처리 결과를 Webhook으로 수신한다.

## 프로그램개발 마스터급 경험 지식 (2026-05-28)
- Source: claude-code-enhanced 2026-05-28
- Tags: development,memory-leak,IDisposable,dotnet8,nullable,GitHub-Actions,field-experience

Revit Add-in C# 코드 메모리 누수 진단 — IDisposable 미구현 패턴: Revit API 객체 중 `FilteredElementCollector`, `Transaction`, `SubTransaction` 등은 `IDisposable`을 구현하고 있어 `using` 블록으로 감싸지 않으면 GC가 수집하지 못한 채 메모리를 점유한다. 진단 방법은 Visual Studio Diagnostic Tools의 메모리 스냅샷 비교로 힙에 남아 있는 Revit API 객체를 탐색하거나, dotMemory로 세션 간 메모리 증분을 측정한다. 패턴 예시: `new FilteredElementCollector(doc)` 반환 후 `Dispose()`를 명시적으로 호출하지 않은 경우가 가장 흔하다. 규칙으로 "모든 Revit API 객체는 `using` 또는 `try-finally`로 관리"를 팀 코드 리뷰 체크리스트에 추가한다.

.NET 8 마이그레이션 시 자주 발생하는 컴파일 오류: ① Nullable Reference Type 경고(CS8600~CS8629) — .NET 8 기본값이 nullable 활성화로, 기존 코드에서 null 허용 경고가 대량 발생한다. 우선 `#nullable disable`로 억제 후 파일 단위로 점진 전환한다. ② `BinaryFormatter` 제거(SYSLIB0011) — .NET 5 이후 보안 이유로 제거됨. `System.Text.Json` 또는 `MessagePack`으로 대체한다. ③ Windows 전용 API(CA1416) — `Registry`, `Process.Start()` 일부가 플랫폼 어노테이션 경고를 발생시키며 `[SupportedOSPlatform("windows")]`로 명시한다.

고객 현장에서만 재현되는 버그 원인 분석 절차: ① Revit Journal 파일(`%LocalAppData%\Autodesk\Revit\Autodesk Revit 202x\Journals\`) 수집 — 마지막 세션의 명령 시퀀스와 예외 스택 트레이스 확인. ② 고객 모델 파일의 경고 확인 — `Model Review`로 손상된 패밀리나 링크 파일 여부 탐색. ③ 운영 환경 차이 확인 — Revit 버전 빌드 번호, .NET 런타임 버전, 다른 Add-in 목록이 개발 환경과 동일한지 비교. ④ 최소 재현 모델(Minimal Reproducible Example)을 고객에게 요청하여 문제 범위를 좁힌다.

GitHub Actions 빌드 실패 원인 패턴 5가지: ① NuGet 패키지 복원 실패 — `nuget.config`에 사내 피드 인증이 누락된 경우. ② RevitAPI.dll 참조 누락 — 로컬 경로 참조를 GitHub Actions 환경에서 찾지 못하는 경우, NuGet 패키지(Revit.API.202x)로 교체한다. ③ .NET 멀티타게팅 충돌 — `net8.0`과 `net48` 타겟을 동시에 빌드할 때 공유 어셈블리 버전 불일치. ④ 코드 서명 타임아웃 — 타임스탬프 서버 응답 지연으로 `signtool.exe` 실패. 재시도 로직 추가 필요. ⑤ 테스트 어셈블리 플랫폼 불일치 — RTF 테스트가 x64를 요구하는데 `Any CPU` 설정으로 인해 x86으로 빌드된 경우.

## R&D 개발지원그룹 개발 운영 기준 (2026-05-28)
- Source: LUA BIM LABS group knowledge update 2026-05-28
- Tags: R&D개발지원그룹,development,architecture,qwen,gate

프로그램개발은 [[R&D_개발지원그룹]]의 구현 책임 지식이다. 개발 착수 전 [[요구사항분석]]의 REQ가 있어야 하며, Revit/Navisworks API 실기 검증이 필요한 항목은 로컬 초안 완료와 별도로 검증 게이트를 남긴다.

개발 단계 구분:
| 단계 | 담당 | 완료 기준 |
|---|---|---|
| Spec Review | 요구사항분석/프로그램개발 | REQ, 제외 범위, 테스트 조건 확인 |
| Architecture | 프로그램개발 | 도메인 로직과 Autodesk API 어댑터 분리 |
| Draft | Qwen_Coder_8B/프로그램개발 | 정적 검토 가능한 초안 또는 테스트 코드 |
| API Gate | Revit_Addin/Navisworks_Addin | 실제 Autodesk 환경 검증 필요 여부 표시 |
| QA Handoff | 빌드검증/QA_테스터 | smoke test와 회귀 테스트 실행 가능 |

원칙:
- 모델 변경 기능은 기본값을 dry-run/report-first로 둔다.
- 순수 도메인 로직은 Revit 없이 테스트 가능해야 한다.
- 외부 API, 고객 모델 경로, 개인정보는 기본 로그에서 제외한다.
- R&D 중 발견한 반복 패턴은 [[지식업데이트]]와 [[테크니컬_라이터]]로 환류한다.
