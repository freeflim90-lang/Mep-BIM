## 1. 목적

본 문서는 팀원이 Telegram으로 BIM, 설계, 시공, 개발, 교육 관련 질문을 요청했을 때 기존 Obsidian 지식을 먼저 검색하고, 답변이 부족한 경우 보강 요청을 받아 지식베이스와 Obsidian을 갱신하는 운영 기준을 정의한다.

핵심 원칙은 `기존 지식 우선, 부족할 때 보강, 모든 요청과 답변을 조직 지식으로 축적`이다.

## 2. 팀원 사용 버튼

팀원에게는 명령어를 학습시키지 않고 Telegram 하단 버튼 3개만 제공한다. 명령어는 관리자와 고급 사용자용으로만 남긴다.

| 버튼 | 목적 | 사용 예시 |
|---|---|---|
| `📚 지식질문` | Obsidian 기준 1차 질문 | 버튼 클릭 후 `공조덕트 점검구 기준 정리해줘` |
| `🔎 더 찾아줘` | 직전 답변 기준 자동 지식 수집/업데이트/재답변 | 버튼 클릭 또는 `더 찾아줘` 입력 |
| `🧩 개발` | Qwen 기반 개발/자동화 요청 접수 | 버튼 클릭 후 `Navisworks 간섭 결과를 Excel로 정리하고 싶어` |

관리자용 명령어는 `/teamhelp`, `/ask`, `/more`, `/ok`, `/fix`를 유지한다.

## 3.0 근거기반 응답 형식

팀원 지식질문과 `더 찾아줘` 재답변은 다음 5칸 구조를 기본으로 한다. 짧은 답변이 필요한 경우에도 근거 상태와 주의 문장은 생략하지 않는다.

| 칸 | 작성 기준 |
|---|---|
| 결론 | 질문에 대한 1차 판단을 한두 문장으로 말한다. |
| 근거 | 사용한 Obsidian 지식, 공식 기준, 내부 문서, 프로젝트 기록을 적는다. |
| 적용 범위 | 공종, 단계, 프로젝트 조건, 버전, 법규 적용 대상을 분리한다. |
| 주의 | `KST03 적용주의`, `KST04 자동수집`, 보안/고객정보 제한을 표시한다. |
| 다음 액션 | 추가 확인, 공식 출처 재검증, 모델/문서 반영, 담당자 확인을 지정한다. |

답변 예시 구조:

```text
결론: [질문에 대한 현재 판단]
근거: [Obsidian 문서/공식 기준/내부 기준]
적용 범위: [공종/단계/조건]
주의: [확인 필요 또는 KST 상태]
다음 액션: [누가 무엇을 확인/반영할지]
```

## 3.1 Q&A 지식 축적 기준

팀원의 모든 지식 질문은 단순 로그가 아니라 Obsidian 지식 노트로 축적한다.

| 항목 | 기준 |
|---|---|
| 저장 위치 | `obsidian_vaults/lua_bim_lab_global_map/NAS_Knowledge/Team_Telegram_QA` |
| 인덱스 | `MOC - Team Telegram QA.md` |
| 기본 상태 | `answered-pending-feedback` |
| 충분 확인 | `verified-by-requester`로 변경 |
| 보강 요청 | `knowledge-gap-needs-review`로 변경 |
| 승격 판단 | 반복 질문, 교육자료 후보, 표준문서 후보, FAQ 후보, 개발/자동화 후보 |

Obsidian Q&A 노트에는 질문자, 질문, 답변, 근거 문서, 지식화 판단 체크리스트를 포함한다. 개인정보, 고객명, 프로젝트명, 계약정보, 계정정보는 저장 전에 제거하거나 마스킹한다.

Q&A 노트에는 사용한 지식 상태 코드(`KST01`~`KST06`)도 함께 기록한다. `KST04 자동수집` 지식으로 답한 경우 답변에는 반드시 재검증 필요 문구를 포함한다.

## 3.2 `더 찾아줘` 자동 보강 기준

`더 찾아줘`는 단순 접수 명령이 아니라 다음 작업을 자동 실행하는 보강 루프다.

| 단계 | 자동 처리 |
|---|---|
| 1 | 직전 질문과 추가 요청 문장을 결합해 검색 질의를 구성 |
| 2 | 기존 내부 지식에서 부족한 부분을 지식 공백으로 기록 |
| 3 | 웹 검색 결과 또는 공식 문서 후보를 수집 |
| 4 | 수집 결과를 담당 지식베이스에 `needs-review` 상태로 저장 |
| 5 | Q&A 노트에 자동 수집 위치와 보강 내용을 누적 |
| 6 | Obsidian 전역 맵과 그래프를 갱신 |
| 7 | 갱신된 지식 기준으로 Telegram에 다시 답변 |

자동 수집 결과는 즉시 확정 지식이 아니라 `needs-review` 상태의 후보 지식이다. 이후 일일 큐레이션 검수에서 보안검토, 승격후보, 보강필요 여부를 다시 분류한다.

## 4. 보안 기준

- Telegram에는 고객명, 프로젝트명, 도면, 계약정보, 개인정보를 입력하지 않는다.
- 파일 업로드는 초기 운영 단계에서 사용하지 않는다.
- `📚 지식질문` 1차 답변은 로컬 Markdown/Obsidian 기준으로만 수행한다.
- `🔎 더 찾아줘`는 자동 수집을 실행하되, 결과는 확정 지식이 아니라 `needs-review` 후보 지식으로 기록한다.
- 팀원과 관리팀 요청에는 DeepSeek API를 사용하지 않는다.
- Qwen_Coder_8B와 로컬 지식 기반 개발은 허용한다.
- 관리팀 Excel 자동화는 별도 승인 없이 Qwen 기반으로 검토한다.
- 최종 개발/자동화 결과물은 Mac 컴퓨터에 파일로 저장하지 않고 Telegram으로 질문자에게 회신한다.
- 개인정보, 급여, 계약, 계정정보 등 중요 데이터는 답변에 그대로 사용하지 않는다.
- 외부 공식 문서 확인이 필요한 내용은 지식업데이트 담당자가 검토 후 반영한다.
- 개발 실행 요청은 자동 구현이 아니라 라우팅/초안/검토 단계로 제한한다.
- AI 실행 배치는 `26_AI_EXECUTION_PLACEMENT_POLICY.md`를 따른다.

## 5. 환경 설정

팀원 Telegram 그룹을 열려면 `.env`에 다음 값이 필요하다.

```bash
TELEGRAM_BOT_TOKEN=...
TELEGRAM_CHAT_ID=...
TELEGRAM_TEAM_CHAT_IDS=-100xxxxxxxxxx
```

`TELEGRAM_TEAM_CHAT_IDS`는 쉼표로 여러 그룹을 등록할 수 있다. 값이 없으면 `TELEGRAM_CHAT_ID`만 허용 채널로 사용한다.
