# LUA BIM LABS Local SSD Knowledge Accumulation Policy

문서번호: LUA-INT-027
작성일: 2026-05-22
소유조직: 지식큐레이터, 인프라_DevOps (Obsidian), 라이선스_보안관
적용범위: 로컬 Mac SSD, Obsidian Vault, 지식베이스, 원본 자료, 검색 인덱스, 팀원 Telegram Q&A 축적 데이터

## 1. 목적

본 문서는 현재 Mac SSD를 활용하여 LUA BIM LABS의 MEP BIM, 설계, 시공, Dynamo, Revit Add-in, 조직 운영 지식을 장기적으로 축적하기 위한 저장 용량, 분류, 검수, 보안 기준을 정의한다.

핵심 원칙은 `많이 저장하되, 답변 가능한 지식으로 정리하고, 민감정보는 저장·응답에서 분리한다`이다.

## 2. 현재 용량 기준

2026-05-22 기준 확인값:

| 항목 | 값 |
|---|---:|
| SSD 전체 용량 | 약 228 GiB |
| 현재 사용량 | 약 12 GiB |
| 현재 여유 공간 | 약 173 GiB |
| 현재 LUA BIM LABS 지식 관련 폴더 | 수 MB 수준 |
| 총 사용량 50% 기준 | 약 114 GiB |
| 50% 기준까지 추가 가능 추정치 | 약 100 GiB 내외 |

주의: macOS 시스템 데이터, 스냅샷, 캐시, 개발 환경이 증가할 수 있으므로 실제 운영 상한은 총 SSD의 50%보다 낮게 잡는다.

## 3. 권장 저장 상한

| 단계 | 총 SSD 사용률 | LUA BIM LABS 운영 판단 |
|---|---:|---|
| Green | 0~35% | 정상 축적. 원본, 요약, 인덱스 모두 저장 가능 |
| Yellow | 35~45% | 원본 중복 제거, 대용량 파일 압축, 인덱스 재검토 |
| Red | 45~50% | 신규 대용량 원본 저장 승인 필요. 요약/메타데이터 우선 |
| Stop | 50% 초과 | 자동 수집 중단. 외장/NAS/클라우드 이관 검토 |

운영 권장값:
- 자동 지식 축적의 1차 목표는 총 SSD 사용률 35%까지로 한다.
- 조직 공개 전 DB 강화 기간에는 최대 45%까지 허용한다.
- 50%는 비상 상한선이며, 지속 운영 목표가 아니다.

## 4. 저장 계층

| 계층 | 저장 대상 | 위치 | 보존 기준 |
|---|---|---|---|
| L1 확정 지식 | 답변에 직접 사용하는 정제 지식 | `data/knowledge_base` | 장기 보존 |
| L2 큐레이션 | 일일 검수, 승격 후보, 반복 질문 | `docs/knowledge_updates` | 장기 보존 |
| L3 Obsidian Vault | 문서 간 연결, 그래프, MOC | `obsidian_vaults` | 장기 보존 |
| L4 원본 자료 | PDF, DOCX, XLSX, 이미지, 외장하드 흡수 자료 | `data/raw_sources` 권장 | 원본성·권리 확인 후 보존 |
| L5 검색 인덱스 | 임베딩, 키워드 인덱스, 캐시 | `data/search_index` 권장 | 재생성 가능하면 압축 또는 삭제 가능 |
| L6 임시 수집 | 웹 수집, Telegram 임시 답변, 자동 보강 후보 | `data/intake_queue` 권장 | 검수 후 승격 또는 폐기 |

## 5. 지식 수집 우선순위

1. MEP BIM 실무 답변에 직접 필요한 지식
2. 설비 설계·시공·도면해석·계산·검토 기준
3. Revit, Dynamo, Navisworks, Add-in 개발 및 오류 오답노트
4. 팀원 Telegram 질문과 답변에서 반복되는 FAQ
5. 교육 커리큘럼, 온보딩, 사내 배포 자료
6. 수익형 상품화 후보와 Autodesk Store 패키징 자료
7. 조직 운영, 보안, 법무, 예산, 협업 프로세스 자료

## 6. 자동 축적 운영 기준

매일 07:00 지식 업데이트 후 다음 절차를 수행한다.

1. 신규 원본과 Q&A를 `intake` 상태로 저장한다.
2. 개인정보, 계정, 토큰, 고객명, 프로젝트명, 내부 경로를 마스킹한다.
3. 지식큐레이터가 분야를 분류한다.
4. 답변 가능성이 있는 항목은 `data/knowledge_base`에 요약 승격한다.
5. 원본은 `data/raw_sources`에 보관하되 중복 파일은 해시 기준으로 제거한다.
6. Obsidian Vault와 HTML 그래프를 재생성한다.
7. 일일 큐레이션 리포트에 저장량, 승격량, 보류량을 기록한다.

## 7. 품질 기준

저장량이 늘어도 다음 기준을 만족하지 못하면 지식 축적 완료로 보지 않는다.

| 기준 | 판단 방법 |
|---|---|
| 검색성 | 질문 의도에 맞는 상위 문서가 검색되는가 |
| 응답성 | Telegram 답변이 질문과 직접 일치하는가 |
| 연결성 | Obsidian에서 상위 주제, 하위 지식, 관련 담당자가 연결되는가 |
| 보안성 | 개인정보와 내부 민감정보가 답변에 노출되지 않는가 |
| 재사용성 | 교육자료, 표준문서, FAQ, 개발 태스크로 승격 가능한가 |

## 8. 보안 기준

- 팀원 질문 원문은 원칙적으로 저장하되, 개인 신상, 연락처, 계정, 토큰, 고객명, 프로젝트명은 마스킹한다.
- 민감 원본은 답변 지식으로 바로 사용하지 않고 `보안검토` 상태로 둔다.
- 외부 공개 전 `라이선스_보안관`과 `법무조항검토`의 검토를 거친다.
- DeepSeek API에는 개인정보, 내부 프로젝트명, 토큰, 계정, 고객자료 원문을 전달하지 않는다.

## 9. 용량 모니터링 항목

매일 또는 주간 리포트에 다음 값을 기록한다.

| 항목 | 명령 예시 |
|---|---|
| 전체 SSD 사용량 | `df -h /` |
| 지식베이스 용량 | `du -sh data/knowledge_base` |
| 문서 용량 | `du -sh docs` |
| Obsidian Vault 용량 | `du -sh obsidian_vaults` |
| 원본 자료 용량 | `du -sh data/raw_sources` |
| 검색 인덱스 용량 | `du -sh data/search_index` |

## 10. 권장 결론

현재 컴퓨터에서 LUA BIM LABS 지식을 SSD 총 사용률 50% 근처까지 축적하는 것은 가능하다. 다만 조직 운영 기준으로는 35%를 1차 목표, 45%를 강화 기간 상한, 50%를 자동 중단 기준으로 설정한다.

즉, SSD를 채우는 것이 목표가 아니라 `일반적인 설비·BIM·Dynamo·Revit 질문에 안정적으로 답변 가능한 지식 밀도`를 만드는 것이 목표다.
