## 1. 목적

본 문서는 LUA BIM LAB 조직을 외부로 공개하기 전, 내부 지식 DB와 Obsidian 지식망이 공개 가능한 품질과 보안 수준을 갖추었는지 점검하기 위한 기준이다.

## 2. 공개 등급 기준

| 등급 | 의미 | 예시 |
|---|---|---|
| Public | 외부 공개 가능 | 회사 소개, 제품 소개, 공개 교육 목차 |
| Client-Shareable | 고객 제출 가능 | 제안서, SOW, 납품 기준, 품질 리포트 샘플 |
| Internal | 내부 운영 전용 | 조직 운영, 의사결정, 교육 운영, 내부 체크리스트 |
| Confidential | 제한 접근 | 계정, 비용, 계약, 고객 정보, 보안 사고, 미공개 개발 전략 |

## 3. 공개 전 필수 점검

| 영역 | 점검 항목 | 상태 |
|---|---|---|
| 문서 분류 | 모든 문서에 공개 등급이 부여되어 있는가 | [ ] |
| 문서 상태 | draft, review, approved, archived 상태가 구분되어 있는가 | [ ] |
| 원본 연결 | Obsidian 색인에서 원본 문서로 추적 가능한가 | [ ] |
| 보안 | 토큰, 계정, 고객명, 개인정보, 내부 경로가 제거되었는가 | [ ] |
| 기능 표현 | 미검증 기능이 확정 표현으로 작성되지 않았는가 | [ ] |
| Revit API | 실제 Revit API 환경 검증 전 항목이 gate로 분리되어 있는가 | [ ] |
| 오류 지식 | 오류 오답노트에 원인, 수정, 검증, 재발 방지가 있는가 | [ ] |
| 중복 문서 | 같은 목적의 문서가 중복되어 혼선을 만들지 않는가 | [ ] |
| 검색성 | 핵심 문서가 전역 Obsidian 맵에서 검색 가능한가 | [ ] |
| 외부 패키지 | 공개용 문서 묶음이 내부 운영 문서와 분리되어 있는가 | [ ] |

## 4. 지식 DB 품질 점검

| 품질 기준 | 통과 기준 |
|---|---|
| 연결성 | 핵심 문서가 최소 1개 이상의 MOC 또는 허브 문서와 연결 |
| 최신성 | 상품, 교육, 조직 문서의 기준일이 명확함 |
| 재사용성 | 교육자료, 상품문서, 제안문구로 재사용 가능한 단위로 정리 |
| 검증성 | 개발 지식은 빌드/테스트/오류/결정 로그와 연결 |
| 추적성 | 중요한 결정은 의사결정 로그 또는 기준 문서와 연결 |

## 5. 공개 차단 조건

다음 조건 중 하나라도 해당하면 외부 공개를 보류한다.

- Confidential 문서가 Public 또는 Client-Shareable 패키지에 포함됨
- 고객명, 계정, 토큰, 내부 경로, 비용 구조가 노출됨
- Add-in 기능이 실제 구현/검증 범위보다 과장되어 표현됨
- Revit API 검증 전 항목이 확정 기능으로 표시됨
- 오류 오답노트 또는 테스트 실패 기록이 제품 문구에 반영되지 않음
- 문서 링크가 끊기거나 전역 Obsidian 맵에서 핵심 문서를 찾을 수 없음

## 6. 외부 공개 전 승인 흐름

1. 작성자가 문서 공개 등급을 지정한다.
2. 지식업데이트 담당이 Obsidian 전역 맵 반영 여부를 확인한다.
3. 보안 담당이 민감정보 노출 여부를 점검한다.
4. 제품/교육/영업 담당이 표현과 실제 제공 범위의 일치 여부를 확인한다.
5. 대표 또는 승인권자가 공개 패키지 배포를 승인한다.
