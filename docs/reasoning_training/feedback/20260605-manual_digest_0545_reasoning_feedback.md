## 후보 1 — Navisworks Manage 2025 API (COM API + .NET Managed API) 간섭검토 자동화

### Q. 지금 당장 확인할 근거는 무엇인가?
두 가지를 먼저 확인한다:
1. **API 가용성**: Navisworks 2025 COM API에서 `ClashDetective.RunTest()`, `ClashGroup` 객체가 2024 버전 대비 인터페이스 변경 없이 유지되는지 ADB(Autodesk Developer Blog) 릴리스 노트 확인
2. **현재 Add-in 의존 버전**: 기존 `Navisworks_Addin.md`에 기록된 API 호출 코드가 2025와 호환되는지 컴파일 오류 여부를 먼저 점검

### Q. 사람이 반복 판단하는 부분 → Add-in 후보
- "어떤 클래시 그룹을 보고서에 포함할 것인가" 판단 → 클라이언트별 `ClashFilter` 프리셋 파일 저장 및 자동 로드
- "보고서 양식을 이 클라이언트 형식으로 맞춰야 한다" 판단 → Jinja2 템플릿 엔진으로 Excel/PDF 자동 생성
- "Hard vs Soft vs Duplicate Clash 구분" → 유형별 자동 분류 후 요약 테이블 생성

### Q. 납품 전 품질 기준 합격/불합격 조건
- [ ] Hard Clash: 0건
- [ ] Soft Clash: 허용 목록(`approved_soft_clash.json`) 외 0건
- [ ] 보고서에 클라이언트 템플릿 적용 완료 확인
- [ ] 자동화 실행 로그(실행 시각, API 버전, 처리 건수) 포함

### Q. 고객 설명 언어
**시간 절감**이 1차 언어.  
간섭 보고서 수작업 작성은 숙련자 기준으로도 1~2시간이다. "Navisworks Add-in으로 보고서를 자동 생성해 납품 준비 시간을 80% 단축한다"는 프레임이 수주 제안 시 가장 빠르게 납득된다.

## 후보 2 — 추론훈련루프.md 10개 지식화 문제 (간섭 알림, 뷰 템플릿 diff 등)

### Q. 재현 조건, 영향 범위, 임시 조치 중 무엇을 먼저?
**재현 조건 먼저.**  
10개 항목 중 "어느 프로젝트 유형/LOD 단계에서 어떤 문제가 집중되는가"를 확인해야 지식화 우선순위가 결정된다. 현재 기록된 항목들은 증상 나열이므로, 과거 3개 프로젝트 회고에서 같은 항목이 반복 등장했는지 교차 확인한다.

### Q. 미검증 부분 — 공식 문서 / 프로젝트 사례 / 담당자 경험 중 무엇으로 막아야 하는가?
**프로젝트 사례**가 우선.  
간섭 알림 조건, 뷰 템플릿 diff, 외주 모델 품질 기준은 공식 문서(IFC 스펙)만으로 해결되지 않는 실무 예외가 많다. 담당자 경험은 변동성이 크다. 최소 2개 이상의 실제 프로젝트 사례에서 동일 패턴이 확인된 항목만 체크리스트로 승격한다.

### Q. 고객 설명 언어
**품질 안정**이 핵심.  
납품 반려, CS 재문의, 외주 모델 재작업은 모두 품질 기준 불명확에서 비롯된다. "납품 전 10단계 자동 검수 체크리스트로 반려율을 줄인다"는 언어.

## 후보 3 — 8개 우선순위 충돌 문제 (납품 검수 vs 신규 개발 등)

### Q. 재현 조건, 영향 범위, 임시 조치 중 무엇을 먼저?
**재현 조건 먼저.**  
"납품 검수와 신규 기능 개발이 동시에 들어오는 상황"은 발생 패턴이 있다. 납품 마감 2주 전에 신규 요청이 들어오는 빈도를 확인해야 임시 조치(냉동 규칙)가 효과를 낼 수 있다.

### Q. 사람이 반복 판단하는 부분 → 스크립트 후보
"지금 들어온 요청을 지금 해야 하는가, 다음 스프린트로 미뤄도 되는가" 판단 → 우선순위 매트릭스 스크립트:
- 입력: 요청 유형(납품 검수/신규 개발/CS/교육), 마감까지 남은 일수
- 출력: 즉시/다음 스프린트/백로그 자동 분류

### Q. 납품 전 품질 기준 합격/불합격 조건
- 납품 검수는 항상 모든 신규 개발 요청보다 선행한다 (냉동 규칙)
- 납품 마감 7일 이내에 신규 기능 요청이 들어오면 자동으로 "다음 스프린트" 분류
- 외주 피드백 처리: 납품 영향이 있는 것만 즉시, 나머지는 다음 루프

### Q. 고객 설명 언어
**리스크 감소**가 핵심.  
우선순위 충돌이 해소되지 않으면 납품 지연 또는 품질 저하 중 하나를 감수해야 한다. "우선순위 자동 분류로 납품 일정 리스크를 줄인다"는 프레임.

## 추론 결론

### 실무 첨언에서 업데이트된 판단
- Navisworks 2025 Add-in은 API 호환성 확인이 선행되지 않으면 자동화 범위를 과대 추정하게 된다.
- 10개 지식화 문제는 프로젝트 사례 2개 이상에서 반복 확인된 항목만 체크리스트로 승격한다.
- 우선순위 충돌은 규칙화(납품 검수 선행 원칙)가 되어야 반복 판단 비용이 사라진다.

### 앞으로의 사고 방식
- 자동화 후보를 볼 때 "API가 지원하는가"보다 "현재 코드베이스와 버전이 맞는가"를 먼저 확인한다.
- 지식화 문제 목록이 10개 이상이면 "프로젝트 사례 2개 이상 확인됐나?"로 절반 이하로 줄인다.
- 우선순위 충돌 유형은 규칙화 → 스크립트화 순서로, 먼저 팀이 동의하는 원칙을 만든다.
