## 후보 1 — 지식 품질 감사 (반복 Telegram QA, 혼합 언어 자동 수집 등 8개 문제)

### Q. 재현 조건, 영향 범위, 임시 조치 중 무엇을 먼저 확인해야 하는가?
**영향 범위를 먼저 확인한다.**  
반복 Telegram QA, 혼합 언어 수집, 뉴스 트렌드 기준, 보안 위험 자료는 각각 증상이 다른 품질 문제다. 재현 조건보다 "이 중 몇 개가 현재 축적된 KB 노트에 실제로 섞여 있는가"를 먼저 세어야 우선순위가 보인다. 임시 조치는 영향 범위 파악 후 가장 큰 오염원(보안 위험 자료, 혼합 언어 노트)에만 먼저 건드린다.

### Q. 사람이 반복 판단하는 부분 → 스크립트/Add-in 후보
- **중복 판단**: "이 뉴스/QA가 이전에 나온 적 있나?" → 의미 기반 중복 필터링 스크립트 (임베딩 유사도 0.92 이상 자동 병합)
- **언어 판단**: "이 노트가 한국어/영어 혼재인가?" → 언어 감지 후 자동 태그 `#mixed-lang` 부착 스크립트
- **보안 판단**: "이 내용에 내부 경로, 토큰, 고객명이 노출됐나?" → 기존 `has_sensitive_external_review_risk()` 패턴을 KB 수집 파이프라인에 연결

### Q. 납품 전 품질 기준 합격/불합격 조건
지식 KB 납품(외부 공개·상품화) 전 체크리스트:
- [ ] 보안 패턴(`api_key`, `/Users/`, 전화번호, 이메일) 0건
- [ ] `#mixed-lang` 태그 노트 0건 (또는 정제 완료 표시)
- [ ] 날짜 범위 벗어난 뉴스 인용 0건 (90일 초과 단독 근거 불가)
- [ ] 중복 의미 노트 병합 완료

### Q. 고객 설명 언어 — 시간 절감 / 품질 안정 / 리스크 감소 중?
**시간 절감**이 1차 언어다.  
고객(내부 팀원 포함)은 "찾으면 또 비슷한 게 나온다"는 혼란에서 시간을 낭비한다. "지식 큐레이션 자동화로 중복 노트를 60% 줄이고 질문→답변 시간을 단축한다"는 프레임이 가장 빠르게 공감된다.

## 후보 2 — BF/편의시설 포트폴리오형 파일럿 (CSO 전략)

### Q. 지금 당장 확인할 근거는 무엇인가?
세 가지를 순서대로 확인한다:
1. **법령 근거**: 장애인·노인·임산부 등 편의증진 보장에 관한 법률 및 KST03 항목 목록 → 자동검수 룰이 대응 가능한 항목 수 파악
2. **기존 Add-in 역량**: 현재 Model Quality Auditor 룰 엔진이 BF 파라미터를 이미 읽을 수 있는가 (기술 적합성)
3. **파일럿 고객 후보**: 현재 BF 인증을 앞둔 설계사/시공사 중 RFI 단계에 있는 고객이 있는가

### Q. 사람이 반복 판단하는 부분 → Add-in 후보
- BF KST03 항목별 "치수 충족 여부" 판단 → Add-in 자동 치수 비교 룰
- "이 항목이 설계 의도인가, 실수인가" 판단 → RFI 후보 자동 플래그 (확정이 아닌 안내)
- 보고서 작성: 항목명·위치·현행값·기준값 표 → Excel/PDF 자동 생성

### Q. 납품 전 품질 기준
BF 진단 파일럿 산출물 합격 조건:
- [ ] KST03 자동 검수 항목 100% 커버
- [ ] 불합격 항목에 법령 근거 조항 번호 포함
- [ ] 인증 대행/법규 적합 확정 표현 0건 (CSO 전략 제한 준수)
- [ ] 미검증 항목은 "RFI 후보" 명시

### Q. 고객 설명 언어
**리스크 감소**가 가장 강한 언어다.  
BF 인증 탈락은 준공 지연·재작업 비용으로 이어진다. "납품 전 BF 항목 사전 점검으로 인증 탈락 리스크를 줄인다"는 프레임. 단, CSO가 정의한 대로 "인증 보장"이 아닌 "리스크 발견 서비스"로 포지셔닝 유지.

## 후보 3 — 간섭 증가 알림, 뷰 템플릿 diff, 납품 검수 서비스 수요 등 10개 검증 설계

### Q. 재현 조건, 영향 범위, 임시 조치 중 무엇을 먼저?
**재현 조건 먼저.**  
10개 항목이 나열되어 있지만 실제로 "지금 이 프로젝트에서 자주 터지는 것"은 2~3개다. 먼저 "어느 LOD 단계, 어느 공종 조합에서 간섭이 급증하는가"를 확인해야 체크리스트 항목 수를 줄일 수 있다.

### Q. 납품 전 품질 기준 합격/불합격 조건
- [ ] Hard Clash: 0건 (또는 허용 간섭 목록에 포함된 것만)
- [ ] 뷰 템플릿 diff: 이전 버전 대비 의도적 변경분 외 0건
- [ ] IFC 내보내기: buildingSMART Validation Service 통과
- [ ] 외주 모델 접수: 기준 레벨/좌표계 일치 확인 완료

### Q. 고객 설명 언어
**품질 안정**이 핵심 언어다.  
납품 반려는 고객 신뢰도 손상이다. "납품 전 10단계 자동 검수로 반려율 0을 목표로 한다"는 프레임이 서비스 상품화 메시지와 직결된다.

## 추론 결론

### 실무 첨언에서 업데이트된 판단
- 지식 품질 감사는 영향 범위 먼저 → 보안 오염원 제거 → 중복 필터링 순서로 처리한다.
- BF 파일럿은 기술 적합성(기존 룰 엔진 활용 가능 여부)을 확인하지 않고 상품화 논의만 진행하면 개발 비용 추정이 불가능해진다.
- 납품 검수 10개 항목은 전부 체크리스트화하는 것이 아니라 LOD 단계별 재현 조건을 먼저 필터링해 실제 필요한 항목만 남긴다.

### 앞으로의 사고 방식
- 여러 문제가 나열될 때 "몇 개가 지금 실제로 발생 중인가"를 먼저 세고, 그 수에 비례한 행동만 취한다.
- 상품화 논의 전에 기존 역량(Add-in 룰 엔진, KB 파이프라인)으로 MVP가 가능한지 확인한다.
- 고객 언어는 문제 유형에 맞게 고른다: 자동화=시간 절감, 인증 리스크=리스크 감소, 납품 기준=품질 안정.
