# LUA BIM LABS 추론 훈련 문제 은행

- 생성일: 2026-06-04
- 목적: 문제 신호를 보고 원인, 예외, 실행 가능성, 자동화 후보를 분리해서 판단하는 훈련
- 사용법: 먼저 `문제`만 보고 1차 답안을 작성한 뒤, `해결책`과 비교한다. `첨언 요청 포인트`는 대표 또는 실무자에게 확인할 질문으로 사용한다.

## 1. 품질 신호: 간섭이 갑자기 늘어난 모델

### 문제
전기 트레이와 공조 덕트 간섭이 지난 주 28건에서 이번 주 143건으로 증가했다. 설계팀은 "모델링을 많이 했기 때문"이라고 설명한다. 이 설명만으로 충분한가?

### 해결책
충분하지 않다. 물량 증가에 따른 자연 증가인지, 좌표/레벨/패밀리 기준 오류로 인한 비정상 증가인지 분리해야 한다.

확인 순서:
1. 이번 주 신규 객체 수 대비 간섭 증가율을 비교한다.
2. 간섭이 특정 층, 특정 구역, 특정 패밀리에 집중되는지 확인한다.
3. 링크 모델의 좌표 변경, 레벨명 변경, 기준점 이동 여부를 확인한다.
4. 동일 객체쌍이 반복 검출되는 중복 간섭인지 확인한다.

### 첨언 요청 포인트
- 현장에서는 간섭 증가율이 어느 정도를 넘으면 비정상 신호로 보는가?
- 간섭을 바로 조정할지, 설계 의도 확인을 먼저 할지 판단하는 기준은 무엇인가?

### 승격 후보
- Model Quality Auditor 룰: `간섭 건수 증가율 / 신규 객체 수 증가율` 비교
- 교육자료: 간섭 수치 증가를 해석하는 법

## 2. 원인 추론: 납품 직전 뷰 템플릿이 깨지는 경우

### 문제
납품 하루 전, 여러 시트의 색상표현과 필터가 달라졌다. 작업자는 "누군가 템플릿을 잘못 건드린 것 같다"고 말한다. 가장 먼저 확인할 근거는 무엇인가?

### 해결책
개인 추측보다 변경 이력을 먼저 확인해야 한다. Revit 작업공유 환경이라면 중앙파일 동기화 시점, 사용자별 변경 로그, 템플릿 이름 변경, 필터 규칙 변경을 확인한다.

우선순위:
1. 문제가 발생한 시트들이 같은 뷰 템플릿을 공유하는지 확인한다.
2. 뷰 템플릿의 필터/가시성/색상표현 설정 변경 여부를 확인한다.
3. 특정 사용자의 동기화 이후에만 발생했는지 확인한다.
4. 템플릿이 아니라 링크 모델 표시 방식 변경인지 확인한다.

### 첨언 요청 포인트
- 실제 프로젝트에서 뷰 템플릿 변경 이력을 어떻게 보존하는가?
- 납품 전 템플릿 잠금 또는 승인 절차가 필요한가?

### 승격 후보
- 체크리스트: 납품 전 뷰 템플릿 스냅샷 확인
- 자동화 후보: 주요 템플릿 설정 diff 리포트

## 3. 사업화 추론: 반복 민원이 곧 제품 기회인가

### 문제
고객 세 곳에서 "BIM 납품 검수가 너무 어렵다"는 문의가 들어왔다. 바로 검수 자동화 SaaS를 만들면 되는가?

### 해결책
바로 제품화하기에는 이르다. 불편함의 표현은 같아도 실제 문제는 다를 수 있다.

분해:
1. 고객이 어려워하는 대상이 모델 품질인지, 문서 제출인지, 발주처 기준 해석인지 구분한다.
2. 반복되는 검수 항목이 정량 규칙인지, 실무 판단이 필요한 항목인지 분리한다.
3. 고객이 돈을 내고 해결하고 싶은 긴급 문제인지 확인한다.
4. 최소 서비스 형태가 컨설팅, 체크리스트, 스크립트, SaaS 중 무엇인지 판단한다.

### 첨언 요청 포인트
- 고객이 실제로 가장 두려워하는 검수 실패 항목은 무엇인가?
- 사람이 봐야 하는 항목과 자동화 가능한 항목의 경계는 어디인가?

### 승격 후보
- 인터뷰 질문지
- 납품 검수 MVP 범위 정의

## 4. 수리 추론: 자동화 ROI 계산

### 문제
한 명의 엔지니어가 매주 6시간을 패밀리 파라미터 정리에 쓴다. 자동화 도구 개발에는 48시간이 필요하고, 유지보수는 매월 3시간이 든다. 3개월 기준으로 도입할 가치가 있는가?

### 해결책
3개월을 12주로 보면 수작업 비용은 72시간이다. 자동화 비용은 개발 48시간 + 유지보수 9시간 = 57시간이다. 단순 시간 기준으로는 15시간 이득이다.

그러나 다음 조건을 함께 봐야 한다:
- 오류 감소 효과가 있는가?
- 여러 프로젝트에 재사용 가능한가?
- 파라미터 규칙이 자주 바뀌는가?
- 개발자가 병목 자원인가?

결론: 3개월만 봐도 약간의 이득이 있으나, 재사용 가능성이 높을 때 우선순위가 올라간다.

### 첨언 요청 포인트
- 유지보수 3시간이 현실적인가?
- 수작업 6시간 안에 검토/수정/커뮤니케이션 시간이 모두 포함되어 있는가?

### 승격 후보
- 자동화 ROI 계산 템플릿
- 반복 업무 우선순위 점수표

## 5. 오류 탐지: 과하게 좋은 지표

### 문제
새 QA 스크립트 실행 결과, 모델 오류가 기존 312건에서 4건으로 줄었다. 팀은 "품질이 좋아졌다"고 보고했다. 이 결론의 위험은 무엇인가?

### 해결책
지표가 급격히 좋아질 때는 실제 개선과 검출 실패를 모두 의심해야 한다.

확인할 내용:
1. 스크립트 규칙이 동일한가?
2. 검사 대상 카테고리가 줄어들지 않았는가?
3. 링크 모델 또는 워크셋 일부가 제외되지 않았는가?
4. 오류 정의가 완화되지 않았는가?
5. 샘플 객체를 수동 검토했을 때 결과가 일치하는가?

### 첨언 요청 포인트
- 품질 지표가 급변할 때 실무 검증 샘플은 몇 개가 적절한가?
- QA 스크립트 결과를 승인하려면 어떤 증빙이 필요한가?

### 승격 후보
- QA 결과 급변 감지 룰
- 검사 범위 누락 체크리스트

## 6. 우선순위 판단: 고객 A와 고객 B

### 문제
고객 A는 높은 금액의 커스텀 자동화를 원하지만 요구사항이 불명확하다. 고객 B는 낮은 금액이지만 요구사항이 명확하고 3개 회사에 재판매 가능성이 있다. 어느 쪽을 먼저 처리해야 하는가?

### 해결책
단순 매출액이 아니라 학습 가치와 재사용성을 함께 봐야 한다. 고객 A는 범위 폭증 위험이 크고, 고객 B는 제품화 가능성이 높다.

판단 기준:
- 요구사항 명확성
- 납기 압박
- 재사용 가능성
- 레퍼런스 가치
- 커스텀 개발 후 유지보수 부담

결론: 개발 리소스가 제한되어 있다면 고객 B를 먼저 작게 완성하고, 고객 A는 유료 진단/요구사항 정의 단계로 분리하는 편이 안전하다.

### 첨언 요청 포인트
- 고객 A의 예산이 요구사항 정의 비용까지 포함하는가?
- 고객 B의 재판매 가능성을 검증할 근거가 있는가?

### 승격 후보
- 프로젝트 수주 판단표
- 커스텀 자동화 사전 진단 상품

## 7. 논리 추론: 침묵은 승인인가

### 문제
추론 다이제스트를 보낸 뒤 대표가 "첨언 사항 없음"이라고 답했다. 이 내용을 자동화 백로그로 바로 승격해도 되는가?

### 해결책
아니다. 첨언 부재는 승인, 보류, 낮은 우선순위, 시간 부족 중 무엇인지 알 수 없다. 따라서 강한 결론으로 승격하지 말고 가설 상태로 유지해야 한다.

처리 방식:
1. "대표 검증 완료"가 아니라 "추가 첨언 없음"으로 기록한다.
2. 자동화 백로그가 아니라 관찰 후보 또는 교육자료 초안으로 둔다.
3. 다음 루프에서 실무 사례나 수치 근거를 추가한다.

### 첨언 요청 포인트
- "첨언 없음"을 어떤 상태값으로 기록할 것인가?
- 승인과 보류를 구분하는 짧은 응답 양식이 필요한가?

### 승격 후보
- 피드백 상태 분류 체계
- 추론 루프 운영 규칙

## 8. 현장 예외: 규칙은 맞지만 적용하면 안 되는 경우

### 문제
품질 규칙상 모든 덕트에는 시스템명이 있어야 한다. 그런데 특정 임시 검토 모델에는 시스템명이 비어 있는 덕트가 많다. 전부 오류로 처리해야 하는가?

### 해결책
모델 목적과 단계에 따라 다르다. 검토용 임시 모델, 컨셉 단계, 납품 모델은 요구 품질이 다르다.

판단:
- 납품 모델이면 오류로 본다.
- 내부 검토용이면 경고로 낮춘다.
- 컨셉 단계라면 시스템명 대신 구역/용도 정보가 있는지 확인한다.

### 첨언 요청 포인트
- 단계별 필수 파라미터 기준을 어떻게 나눌 것인가?
- 경고와 오류를 구분하는 현장 기준은 무엇인가?

### 승격 후보
- 단계별 QA 룰셋
- 경고/오류 심각도 분류표

## 9. 인과 추론: 교육 후 오류 감소

### 문제
패밀리 교육을 진행한 뒤 파라미터 오류가 40% 줄었다. 교육 효과라고 결론내려도 되는가?

### 해결책
가능성은 있지만 단정하면 안 된다. 같은 기간에 템플릿 개선, 검수 강화, 작업 범위 축소가 있었는지 확인해야 한다.

검증:
1. 교육 전후 작업자와 작업 범위가 같은가?
2. 오류 검출 기준이 같은가?
3. 교육을 받은 팀과 받지 않은 팀의 차이가 있는가?
4. 오류 유형 중 교육 내용과 직접 관련된 항목만 줄었는가?

### 첨언 요청 포인트
- 교육 효과를 측정할 대표 지표는 무엇인가?
- 교육보다 템플릿 개선이 더 큰 영향을 줬을 가능성은 없는가?

### 승격 후보
- 교육 효과 측정표
- 교육 내용별 오류 유형 매핑

## 10. 의사결정: 완벽한 자동화 vs 반자동 도구

### 문제
객체 분류 자동화 기능을 만들려 한다. 완전 자동 분류는 개발 4주, 정확도 85%가 예상된다. 반자동 추천 도구는 개발 1주, 사용자가 승인하면 정확도 97%다. 무엇을 선택할까?

### 해결책
실무 납품 품질이 중요하다면 반자동 추천 도구가 더 적합하다. 완전 자동화는 편하지만 오류 책임과 검토 비용이 커질 수 있다.

선택 기준:
- 오류가 발생했을 때 피해가 큰가?
- 사용자가 판단을 빠르게 확인할 수 있는가?
- 반복 작업의 피로를 줄이는 것이 목표인가, 사람 판단을 대체하는 것이 목표인가?

결론: 초기 제품은 반자동으로 시작하고, 승인 데이터를 모아 완전 자동화 범위를 좁힌다.

### 첨언 요청 포인트
- 사용자가 승인해야 하는 지점은 몇 단계가 적절한가?
- 85% 자동 분류 오류를 현업이 감당할 수 있는가?

### 승격 후보
- Human-in-the-loop 자동화 설계 원칙
- 승인 로그 기반 학습 데이터 수집

## 11. 모순 탐지: 요구사항의 충돌

### 문제
발주처는 "모든 객체를 상세하게 모델링하라"고 하면서 동시에 "파일 용량을 500MB 이하로 유지하라"고 요구한다. 어떻게 대응해야 하는가?

### 해결책
두 요구는 충돌할 수 있다. "상세하게"의 의미를 LOD, 필수 카테고리, 파라미터 수준으로 재정의해야 한다.

대응:
1. 상세 모델링 대상과 단순 표현 대상을 나눈다.
2. 파일 용량에 큰 영향을 주는 패밀리, 중복 객체, 과도한 형상을 식별한다.
3. 발주처에 용량 제한과 상세도 기준의 우선순위를 확인한다.
4. 납품 기준서에 예외 조건을 문서화한다.

### 첨언 요청 포인트
- 현장에서 "상세"를 발주처와 합의할 때 쓰는 기준 문구는 무엇인가?
- 용량 제한을 넘기지 않기 위한 패밀리 제작 원칙은 무엇인가?

### 승격 후보
- EIR/BEP 요구사항 충돌 검토표
- 패밀리 경량화 체크리스트

## 12. 데이터 해석: 평균의 함정

### 문제
모델 검수 평균 점수가 88점이라 양호하다고 보고되었다. 하지만 3층 기계실은 52점이다. 전체 평균만 보고 통과시켜도 되는가?

### 해결책
아니다. 평균은 특정 위험 구역을 숨길 수 있다. 납품 리스크는 평균보다 최저 점수, 핵심 구역 점수, 치명 오류 수에 더 민감하다.

확인:
- 점수가 낮은 구역이 핵심 설비 구역인가?
- 치명 오류가 집중되어 있는가?
- 낮은 점수가 일정 기준 이하이면 전체 통과를 막는 게 맞는가?

### 첨언 요청 포인트
- 전체 평균과 구역별 최소 점수 중 어느 기준을 우선할 것인가?
- 기계실, 전기실 같은 핵심 구역에 가중치를 줄 것인가?

### 승격 후보
- 구역별 품질 대시보드
- 핵심 구역 가중 점수 체계

## 13. 리스크 추론: 빠른 납품 요청

### 문제
고객이 "검수는 나중에 하고 오늘 먼저 납품해달라"고 요청한다. 팀은 관계 유지를 위해 그대로 진행하려 한다. 어떤 리스크가 있는가?

### 해결책
검수 없는 납품은 이후 하자, 책임 소재, 재작업 비용으로 돌아올 수 있다.

대응:
1. 미검수 상태임을 납품 메일에 명확히 기록한다.
2. 확인된 제한사항과 미확인 항목을 분리한다.
3. 최소 필수 검수만 수행한다.
4. 정식 납품이 아니라 임시 제출인지 합의한다.

### 첨언 요청 포인트
- 실무에서 최소 필수 검수 항목은 무엇인가?
- 고객 요청으로 검수를 생략할 때 어떤 문구를 남겨야 하는가?

### 승격 후보
- 임시 납품 확인 문구 템플릿
- 최소 검수 체크리스트

## 14. 패턴 인식: 반복되는 야근

### 문제
매주 금요일마다 BIM 팀 야근이 반복된다. 팀장은 "이번 주도 물량이 많았다"고 말한다. 어떤 질문을 해야 하는가?

### 해결책
물량만 원인으로 보면 개선이 어렵다. 업무 흐름, 승인 지연, 정보 도착 시점, 검수 병목을 함께 봐야 한다.

질문:
- 금요일에 몰리는 업무는 어떤 종류인가?
- 입력자료가 늦게 오는가, 내부 검수가 늦는가?
- 특정 담당자에게만 일이 집중되는가?
- 월요일부터 목요일까지 대기 시간이 있었는가?

### 첨언 요청 포인트
- 야근을 줄이기 위해 가장 먼저 시각화해야 할 업무 흐름은 무엇인가?
- 반복 야근이 자동화 문제인지 운영 문제인지 어떻게 구분하는가?

### 승격 후보
- 주간 업무 병목 분석표
- 작업 요청 접수 마감 규칙

## 15. 제품 판단: 기능 요청의 숨은 의도

### 문제
사용자가 "버튼 하나로 모든 도면을 자동 생성해달라"고 요청했다. 이 요청을 그대로 개발해야 하는가?

### 해결책
그대로 개발하기 전에 사용자의 실제 의도를 파악해야 한다. 사용자는 완전 자동 도면화를 원한다기보다 반복 세팅, 뷰 생성, 시트 배치, 태그 누락을 줄이고 싶을 수 있다.

분해:
- 어떤 도면이 대상인가?
- 사용자가 가장 오래 걸리는 단계는 무엇인가?
- 자동 생성 후 사람이 반드시 검토해야 하는 항목은 무엇인가?
- 실패했을 때 되돌리기와 로그가 필요한가?

### 첨언 요청 포인트
- 도면 자동화에서 사람이 반드시 개입해야 하는 단계는 어디인가?
- MVP로 가장 먼저 자동화할 단계는 무엇인가?

### 승격 후보
- 도면 자동화 기능 분해표
- 사용자 인터뷰 스크립트

## 16. 추론 검증: 좋은 설명이지만 근거가 약한 경우

### 문제
어떤 분석가가 "BIM 자동화 시장은 앞으로 빠르게 성장할 것이므로 지금 바로 대규모 개발팀을 꾸려야 한다"고 주장한다. 이 추론의 약점은 무엇인가?

### 해결책
시장 성장과 대규모 채용은 바로 연결되지 않는다. 중간 근거가 필요하다.

빠진 근거:
- 우리 고객군에서 실제 구매 의사가 있는가?
- 경쟁 제품과 차별점이 있는가?
- 현재 팀이 검증한 반복 문제는 무엇인가?
- 작은 제품으로 수요를 확인했는가?
- 대규모 개발팀을 유지할 매출 흐름이 있는가?

### 첨언 요청 포인트
- 대규모 개발 전에 반드시 통과해야 할 검증 단계는 무엇인가?
- 시장 성장 자료보다 더 신뢰할 내부 지표는 무엇인가?

### 승격 후보
- 사업화 추론 검증 체크리스트
- MVP 단계별 투자 기준

## 17. 분류 추론: 자동화 후보인지 교육 후보인지

### 문제
엔지니어들이 매번 같은 파라미터 입력 실수를 한다. 바로 자동화해야 하는가, 교육해야 하는가?

### 해결책
실수 원인에 따라 다르다.

자동화가 적합한 경우:
- 입력값 규칙이 명확하다.
- 예외가 적다.
- 반복 빈도가 높다.
- 오류 발생 시 수정 비용이 크다.

교육이 먼저인 경우:
- 규칙 이해 부족이 원인이다.
- 프로젝트마다 예외가 많다.
- 입력값이 설계 판단에 따라 달라진다.

결론: 먼저 교육자료와 체크리스트로 규칙을 안정화하고, 반복 패턴이 충분히 고정되면 자동화한다.

### 첨언 요청 포인트
- 이 실수가 규칙 미숙지 때문인가, UI/업무흐름 문제 때문인가?
- 예외 케이스가 얼마나 자주 발생하는가?

### 승격 후보
- 교육 우선/자동화 우선 판정표
- 파라미터 오류 유형별 대응 매트릭스

## 18. 우선순위 수식: 치명도와 빈도

### 문제
오류 A는 매일 발생하지만 수정이 쉽다. 오류 B는 한 달에 한 번 발생하지만 납품 반려를 유발한다. 어떤 오류를 먼저 해결해야 하는가?

### 해결책
빈도만 보면 A가 우선이지만, 치명도까지 고려하면 B가 더 중요할 수 있다.

간단한 점수:
- 우선순위 = 빈도 x 치명도 x 탐지 어려움
- A: 빈도 높음, 치명도 낮음, 탐지 쉬움
- B: 빈도 낮음, 치명도 매우 높음, 탐지 어려움

결론: 납품 반려를 일으키는 B는 별도 방어장치가 필요하다. A는 일괄 수정 스크립트나 체크리스트로 처리한다.

### 첨언 요청 포인트
- 납품 반려로 이어지는 오류 목록은 무엇인가?
- 빈도보다 치명도를 우선해야 하는 기준선은 어디인가?

### 승격 후보
- 오류 우선순위 산정표
- 납품 반려 방지 룰셋

## 19. 커뮤니케이션 추론: 고객에게 원인을 말하는 방식

### 문제
납품 지연의 직접 원인은 고객의 자료 지연이다. 하지만 메일에 "고객 자료가 늦어서 지연됩니다"라고 쓰면 관계가 나빠질 수 있다. 어떻게 표현해야 하는가?

### 해결책
책임 추궁보다 영향과 다음 액션 중심으로 써야 한다.

권장 표현:
"추가 자료 반영 범위가 확정되는 즉시 모델 검수와 시트 정리를 진행하겠습니다. 현재 일정 기준으로는 자료 확정 후 2영업일이 필요하므로, 오늘 중 확정 시 금요일 제출이 가능합니다."

핵심:
- 필요한 입력을 명확히 한다.
- 입력 지연이 일정에 미치는 영향을 수치로 말한다.
- 상대를 비난하지 않는다.
- 다음 선택지를 제시한다.

### 첨언 요청 포인트
- 고객에게 일정 영향을 설명할 때 어느 정도까지 직접적으로 말하는가?
- 내부 책임과 외부 요청을 문서화하는 기준은 무엇인가?

### 승격 후보
- 고객 일정 안내 문구집
- 지연 원인 기록 템플릿

## 20. 메타 추론: 문제를 잘못 정의한 경우

### 문제
팀이 "Revit이 느리다"는 문제를 해결하려고 고성능 PC 구매를 검토한다. 이것이 적절한 첫 해결책인가?

### 해결책
아닐 수 있다. "Revit이 느리다"는 증상이지 원인이 아니다.

원인 후보:
- 모델 파일이 과도하게 크다.
- 링크가 많거나 불필요한 링크가 있다.
- 패밀리 형상이 무겁다.
- 워크셋 관리가 부적절하다.
- 네트워크/중앙파일 접근 속도가 느리다.
- 특정 애드인이 병목을 만든다.

해결 순서:
1. 느린 작업이 열기, 저장, 뷰 전환, 동기화, 특정 명령 중 어디인지 분리한다.
2. 모델 진단과 PC 성능 진단을 분리한다.
3. 저비용 개선을 먼저 시도한다.
4. 하드웨어 구매는 병목이 실제로 PC 자원일 때 결정한다.

### 첨언 요청 포인트
- 현장에서 Revit 성능 문제를 진단할 때 가장 먼저 보는 지표는 무엇인가?
- 하드웨어 문제와 모델 관리 문제를 구분하는 기준은 무엇인가?

### 승격 후보
- Revit 성능 진단 체크리스트
- 모델 경량화 교육자료
