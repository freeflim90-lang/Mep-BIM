# LUA BIM LABS 추론 훈련 루프 세트 02

- 생성일: 2026-06-04
- 목적: 1차 추론을 곧바로 결론으로 확정하지 않고, 첨언/예외/증거 부족 상태를 반영해 수정하는 훈련
- 핵심 원칙: `신호 → 1차 추론 → 반례/첨언 → 수정 결론 → 승격 상태`

## 판정 상태 정의

- `가설`: 흥미로운 신호이나 근거가 부족하다.
- `교육자료 후보`: 사람의 이해와 판단이 먼저 필요하다.
- `체크리스트 후보`: 반복 확인이 필요하지만 자동화 전 단계다.
- `자동화 후보`: 규칙이 명확하고 예외가 적으며 반복 빈도가 높다.
- `보류`: 지금 실행하면 오판 또는 자원 낭비 가능성이 높다.

## 1. 첨언 부재를 승인으로 착각하는 문제

### 신호
다이제스트에서 "간섭 증가율을 자동 감지하는 룰을 만들자"는 제안이 나왔다. 대표는 "첨언 사항 없음"이라고 답했다.

### 1차 추론
대표가 반대하지 않았으므로 자동화 백로그에 등록한다.

### 반례/첨언
"첨언 사항 없음"은 승인일 수도 있지만, 검토 시간이 없거나 우선순위가 낮다는 뜻일 수도 있다. 특히 자동화 룰은 현장 예외가 많으면 잘못된 알람만 늘릴 수 있다.

### 수정 결론
자동화 백로그가 아니라 `체크리스트 후보`로 둔다. 다음 루프에서 최근 3개 프로젝트의 간섭 증가 패턴과 예외를 확인한 뒤 자동화 여부를 판단한다.

### 승격 상태
체크리스트 후보

### 추가 질문
- 간섭 증가율이 몇 퍼센트를 넘으면 실제 조치가 필요했는가?
- 단순 물량 증가와 품질 저하를 구분할 기준 데이터가 있는가?

## 2. 자동화하고 싶은 마음이 문제 정의를 앞서는 경우

### 신호
작업자가 매일 패밀리 파라미터를 수정한다. 팀은 "파라미터 자동 입력 애드인을 만들자"고 제안했다.

### 1차 추론
반복 업무이므로 자동화하면 된다.

### 반례/첨언
반복된다고 모두 자동화 대상은 아니다. 파라미터 수정이 설계 변경 반영인지, 템플릿 오류 보정인지, 작업자 교육 부족인지부터 구분해야 한다.

### 수정 결론
먼저 오류 유형을 2주간 기록한다. 값 규칙이 명확하고 예외가 적으면 자동화 후보, 규칙 이해 부족이면 교육자료 후보, 템플릿 문제이면 템플릿 개선 후보로 분리한다.

### 승격 상태
가설

### 추가 질문
- 수정되는 파라미터는 매번 같은가?
- 값이 도면/시방/장비표 중 어디에서 결정되는가?
- 자동 입력 실패 시 사람이 쉽게 확인할 수 있는가?

## 3. 고객 문의를 시장 검증으로 과대 해석하는 경우

### 신호
두 고객이 "납품 검수 자동화가 있으면 좋겠다"고 말했다.

### 1차 추론
시장 수요가 확인됐으니 납품 검수 SaaS를 개발한다.

### 반례/첨언
"있으면 좋겠다"는 구매 의사와 다르다. 또한 고객이 원하는 것은 SaaS가 아니라 납품 전 컨설팅, 체크리스트, 발주처 기준 해석일 수 있다.

### 수정 결론
제품 개발 전에 유료 진단 서비스나 수동 검수 패키지로 수요를 검증한다. 반복되는 검수 항목을 수집한 뒤 정량 규칙만 자동화 범위로 분리한다.

### 승격 상태
교육자료 후보

### 추가 질문
- 고객은 검수 실패로 어떤 비용을 실제로 겪었는가?
- 월 구독형 도구를 원하는가, 납품 전 1회 검수 서비스를 원하는가?

## 4. 평균 점수로 핵심 리스크를 숨기는 경우

### 신호
프로젝트 모델 품질 평균 점수는 91점이다. 하지만 지하 2층 기계실은 57점이고, 해당 구역은 시공 조율 핵심 구역이다.

### 1차 추론
평균이 90점을 넘었으므로 전체 품질은 양호하다.

### 반례/첨언
핵심 구역의 낮은 점수는 평균보다 중요할 수 있다. 특히 기계실, 전기실, 샤프트, 천장 내부처럼 간섭 비용이 큰 구역은 별도 기준이 필요하다.

### 수정 결론
전체 평균과 별개로 핵심 구역 최소 점수 기준을 둔다. 핵심 구역이 기준 미달이면 전체 통과를 보류한다.

### 승격 상태
체크리스트 후보

### 추가 질문
- 핵심 구역은 누가, 어떤 기준으로 지정하는가?
- 핵심 구역 점수 미달 시 전체 납품을 막을 것인가?

## 5. 빠른 납품을 고객 만족으로 착각하는 경우

### 신호
고객이 "오늘 우선 제출하고 검수는 다음 주에 하자"고 요청했다.

### 1차 추론
고객 요청이므로 오늘 바로 납품한다.

### 반례/첨언
고객 요청이라도 미검수 납품은 책임 소재를 흐린다. 이후 오류가 발견되면 "왜 확인하지 않고 제출했느냐"는 문제가 생길 수 있다.

### 수정 결론
정식 납품이 아니라 `임시 제출본`으로 명시하고, 미검수 항목과 제한사항을 문서화한다. 최소 필수 검수만 수행한 뒤 제출한다.

### 승격 상태
체크리스트 후보

### 추가 질문
- 임시 제출과 정식 납품을 구분하는 내부 문구가 있는가?
- 미검수 상태에서 절대 제출하면 안 되는 항목은 무엇인가?

## 6. 높은 매출 제안을 좋은 프로젝트로 오판하는 경우

### 신호
고객 A가 큰 예산으로 Revit 자동화 개발을 의뢰했다. 요구사항은 "우리 업무 전체를 편하게 해달라" 수준으로 넓다.

### 1차 추론
큰 매출 기회이므로 바로 개발에 착수한다.

### 반례/첨언
요구사항이 넓은 커스텀 개발은 범위 폭증과 유지보수 리스크가 크다. 큰 금액이더라도 정의되지 않은 문제는 팀 리소스를 잠식할 수 있다.

### 수정 결론
본 개발 전에 유료 요구사항 정의 단계를 분리한다. 업무흐름 분석, 자동화 후보 우선순위, 제외 범위, 유지보수 조건을 문서화한 뒤 개발 계약으로 넘어간다.

### 승격 상태
교육자료 후보

### 추가 질문
- 고객이 원하는 것은 개발 결과물인가, 업무 개선 컨설팅인가?
- 유지보수 비용과 책임 범위를 계약에 넣을 수 있는가?

## 7. 좋은 기술을 좋은 상품으로 착각하는 경우

### 신호
모델 오류를 AI로 분류하는 데모가 잘 작동했다. 내부에서는 "이제 품질검수 AI 상품을 만들 수 있다"고 본다.

### 1차 추론
기술 검증이 됐으므로 상품화한다.

### 반례/첨언
데모 성공은 상품 성공과 다르다. 실제 상품에는 데이터 입력 방식, 예외 처리, 결과 설명, 책임 범위, 고객 워크플로우 편입이 필요하다.

### 수정 결론
기술을 상품화하기 전에 검수 결과를 사람이 이해하고 승인할 수 있는 리포트 형태로 만든다. AI 판단은 권고로 표시하고, 최종 책임은 사용자 승인 흐름에 둔다.

### 승격 상태
가설

### 추가 질문
- AI 분류 오류가 생기면 누가 수정하고 기록하는가?
- 고객은 검수 결과를 어떤 형식으로 발주처에 제출하는가?

## 8. 문제를 사람 탓으로 단순화하는 경우

### 신호
같은 작업자가 계속 시트 번호를 잘못 입력한다.

### 1차 추론
작업자 주의 부족이므로 교육을 다시 한다.

### 반례/첨언
반복 실수는 개인 부주의가 아니라 UI, 명명 규칙, 검수 흐름, 템플릿 문제일 수 있다.

### 수정 결론
교육 전에 오류 발생 위치를 확인한다. 시트 번호 체계가 복잡하거나 입력 화면에서 검증이 없다면 템플릿/자동검사 개선이 더 효과적이다.

### 승격 상태
체크리스트 후보

### 추가 질문
- 같은 실수가 다른 작업자에게도 발생하는가?
- 시트 번호 규칙을 자동으로 검증할 수 있는가?

## 9. 데이터 부족을 빠른 결론으로 메우는 경우

### 신호
어느 프로젝트에서 Navisworks 간섭 검토 시간이 30% 줄었다. 팀은 새 조율 프로세스가 효과적이라고 판단했다.

### 1차 추론
새 프로세스가 효과가 있으므로 전사 표준으로 적용한다.

### 반례/첨언
검토 시간이 줄어든 이유가 모델 규모 축소, 검토 범위 축소, 숙련자 투입, 간섭 기준 완화일 수 있다.

### 수정 결론
전사 표준화 전에 최소 3개 프로젝트에서 동일 기준으로 비교한다. 시간 감소뿐 아니라 재작업률, 미검출 간섭, 현장 이슈도 함께 본다.

### 승격 상태
가설

### 추가 질문
- 검토 범위와 간섭 기준이 이전과 동일했는가?
- 시간은 줄었지만 품질 리스크가 늘지는 않았는가?

## 10. 자동화 실패를 도구 문제로만 보는 경우

### 신호
새 자동 태깅 도구를 배포했지만 사용률이 낮다.

### 1차 추론
도구 성능이 부족하므로 기능을 더 개발한다.

### 반례/첨언
사용률 저하는 성능 문제뿐 아니라 배포 방식, 교육 부족, 기존 업무흐름과의 충돌, 실패 시 복구 불안 때문일 수 있다.

### 수정 결론
기능 추가 전에 사용하지 않는 이유를 기록한다. 실패 케이스, 실행 시간, 되돌리기, 태그 품질, 사용자 신뢰를 분리해서 개선한다.

### 승격 상태
교육자료 후보

### 추가 질문
- 사용자가 도구를 실행하지 않는 순간은 언제인가?
- 결과를 믿지 못하는가, 실행 절차가 불편한가?

## 11. 낮은 오류 빈도를 낮은 리스크로 착각하는 경우

### 신호
IFC 내보내기 오류는 한 달에 한 번 정도만 발생한다.

### 1차 추론
빈도가 낮으니 우선순위를 낮춘다.

### 반례/첨언
IFC 납품 오류는 빈도가 낮아도 발주처 반려, 재납품, 일정 지연으로 이어질 수 있다.

### 수정 결론
빈도보다 치명도를 기준으로 최소 방어장치를 만든다. IFC 내보내기 전 필수 설정, 좌표, 속성, 분류체계를 체크하는 납품 전 루틴을 둔다.

### 승격 상태
체크리스트 후보

### 추가 질문
- IFC 오류가 실제 반려로 이어진 사례가 있는가?
- 내보내기 전 자동으로 검증 가능한 항목은 무엇인가?

## 12. 고객의 말과 실제 병목이 다른 경우

### 신호
고객은 "도면 자동화가 필요하다"고 말한다. 인터뷰를 해보니 실제로는 도면 생성보다 발주처별 표기 기준을 매번 해석하는 데 시간이 많이 든다.

### 1차 추론
고객이 도면 자동화를 말했으니 도면 생성 기능을 만든다.

### 반례/첨언
고객의 표현은 해결책 언어일 수 있고, 실제 병목은 기준 해석일 수 있다. 잘못 만들면 화려하지만 덜 필요한 기능이 된다.

### 수정 결론
도면 생성 자동화보다 발주처별 표기 기준 라이브러리, 체크리스트, 템플릿 추천 기능을 먼저 검토한다.

### 승격 상태
교육자료 후보

### 추가 질문
- 고객이 가장 오래 걸린다고 느끼는 단계와 실제 시간이 많이 쓰이는 단계가 같은가?
- 표기 기준을 라이브러리화할 수 있는가?

## 오늘의 사고 원칙

1. 첨언 부재는 승인으로 보지 않는다.
2. 반복 업무는 먼저 원인을 분류한 뒤 자동화한다.
3. 시장 신호는 구매 의사와 구분한다.

6. 빈도가 낮아도 치명도가 높으면 방어장치를 만든다.
