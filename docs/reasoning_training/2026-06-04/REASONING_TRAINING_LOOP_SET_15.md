# LUA BIM LABS 추론 훈련 루프 세트 15

- 생성일: 2026-06-04
- 목적: 의사결정 후 결과를 추적하고, 수정·확대·중단 조건을 명확히 한다.
- 훈련 초점: `결정 → 관찰 지표 → 수정 트리거 → 확대 조건 → 중단 조건`

## 모니터링 코드

- `M01 선행지표`: 결과가 나오기 전에 위험을 알려주는 지표다.
- `M02 결과지표`: 결정이 실제 효과를 냈는지 보여주는 지표다.
- `M03 부작용지표`: 의도하지 않은 비용, 저항, 품질 저하를 보여주는 지표다.
- `M04 수정 트리거`: 기준을 넘으면 운영 방식을 바꿔야 하는 신호다.
- `M05 확대 조건`: 더 넓게 적용해도 되는 조건이다.
- `M06 중단 조건`: 계속하면 손해가 커져 멈춰야 하는 조건이다.

## 1. IFC 사전 체크리스트 도입 후 모니터링

### 결정
IFC 납품 전 발주처별 사전 체크리스트를 적용한다.

### 관찰 지표
- 반려 건수
- 재납품 소요 시간
- 체크리스트 작성 시간
- 누락 항목 발견 수
- 발주처별 반복 오류

### 수정 트리거
체크리스트 작성 시간이 과도하게 늘거나, 발견 항목 대부분이 납품에 영향이 없으면 항목을 줄인다.

### 확대 조건
2건 이상 납품에서 반려 또는 재작업을 줄이면 공통 SOP로 확장한다.

### 중단 조건
발주처별 요구사항이 계속 바뀌어 체크리스트 유지 비용이 효과보다 크면 발주처별 수동 검토로 축소한다.

### 지식화 위치
IFC_OpenBIM, BIM_납품검수, EIRBEP_심사원

## 2. 자동 태깅 로그 개선 후 모니터링

### 결정
자동 태깅 도구에 실패 객체 로그, 변경 수량 리포트, 되돌리기 안내를 추가한다.

### 관찰 지표
- 도구 사용률
- 실패 객체 수
- 수동 수정 시간
- 되돌리기 발생 횟수
- 사용자 신뢰도 메모

### 수정 트리거
사용률은 늘었지만 수동 수정 시간이 줄지 않으면 태그 품질 규칙을 다시 본다.

### 확대 조건
2주 동안 사용률이 증가하고 수동 수정 시간이 30% 이상 줄면 다음 팀에 배포한다.

### 중단 조건
로그 개선 후에도 실패 원인이 모델/패밀리 예외로 대부분 발생하면 전면 자동화보다 반자동 승인 흐름으로 전환한다.

### 지식화 위치
Revit_Addin, QA_테스터, 빌드검증

## 3. 납품 전 검수 서비스 파일럿 후 모니터링

### 결정
SaaS 개발 전 납품 전 1회성 검수 서비스를 유료 파일럿으로 제공한다.

### 관찰 지표
- 유료 파일럿 수락률
- 반복 구매 의사
- 리포트 만족도
- 수동 검수 소요 시간
- 자동화 가능한 반복 항목 수

### 수정 트리거
리포트 만족도는 높지만 반복 구매 의사가 낮으면 가격, 범위, 납품 시점을 재검토한다.

### 확대 조건
3건 이상에서 반복 구매 의사와 자동화 가능한 항목이 확인되면 제품 후보를 분리한다.

### 중단 조건
고객마다 검수 문제가 완전히 달라 표준 리포트가 불가능하면 SaaS 후보에서 제외하고 컨설팅 서비스로 유지한다.

### 지식화 위치
BIM_납품검수, 제품패키징, 고객지원운영

## 4. 외주 접수 검수표 도입 후 모니터링

### 결정
외주 제출 직후 30분 빠른 접수 검수를 수행한다.

### 관찰 지표
- 접수 검수 소요 시간
- 치명 오류 조기 발견 수
- 반복 경미 오류 수
- 재제출 요청 횟수
- 외주사별 개선 추세

### 수정 트리거
치명 오류는 없고 경미 오류만 반복되면 검수표를 외주사 교육/착수 기준 자료로 전환한다.

### 확대 조건
재제출 요청 횟수 또는 내부 수정 시간이 줄면 모든 외주 제출에 적용한다.

### 중단 조건
검수표 적용 시간이 내부 수정 시간보다 크고 오류 감소가 없으면 항목을 축소한다.

### 지식화 위치
외주관리, BIM_납품검수

## 5. CS 표준 답변 보강 후 모니터링

### 결정
표준 답변에 고객 상황 확인 질문과 다음 액션 선택지를 추가한다.

### 관찰 지표
- 재문의율
- 최종 해결 시간
- 고객 불만 표현 수
- 표준 답변 사용률
- 상담자 수정 메모

### 수정 트리거
재문의율은 줄었지만 최종 해결 시간이 늘면 답변이 과하게 복잡한지 확인한다.

### 확대 조건
재문의율과 불만 표현이 함께 줄면 문의 유형별 템플릿으로 확장한다.

### 중단 조건
상담자가 표준 답변을 계속 수정해야 한다면 템플릿이 현장 언어와 맞지 않는 것이므로 재작성한다.

### 지식화 위치
CS_기술지원관, 고객지원CS, 고객지원운영

## 6. 요구사항 정의 단계 규모별 분리 후 모니터링

### 결정
대형 개발은 유료 요구사항 정의, 소형 개발은 간이 범위 가정서로 운영한다.

### 관찰 지표
- 견적 대비 실제 투입 시간
- 변경 요청 횟수
- 고객 계약 전 이탈률
- 정의 단계 소요 시간
- 범위 분쟁 건수

### 수정 트리거
소형 개발에서도 범위 분쟁이 반복되면 간이 범위 가정서를 보강한다.

### 확대 조건
대형 개발에서 재작업이 30% 이상 줄면 유료 정의 단계를 표준 절차로 둔다.

### 중단 조건
고객 이탈률이 급증하고 재작업 감소 효과가 약하면 정의 단계 산출물과 가격 구조를 다시 설계한다.

### 지식화 위치
요구사항분석, 견적심사원, BIM_프로젝트_견적산정

## 7. 교육자료를 온보딩 체크리스트로 전환한 후 모니터링

### 결정
파라미터 교육자료를 신규 작업자 온보딩 체크리스트와 작업 중 확인표로 전환한다.

### 관찰 지표
- 신규 작업자 오류 수
- 반복 오류 유형
- 체크리스트 사용률
- 교육 소요 시간
- 질문/재문의 항목

### 수정 트리거
체크리스트 사용률은 높지만 오류가 줄지 않으면 항목이 실제 오류 순간과 맞지 않는지 확인한다.

### 확대 조건
신규 작업자 3명 이상에서 관련 오류가 줄면 정식 온보딩 자료로 승격한다.

### 중단 조건
체크리스트가 너무 길어 사용률이 낮으면 필수 항목과 참고 항목으로 분리한다.

### 지식화 위치
교육컨설팅, Revit_Family제작, BIM_템플릿기획관

## 8. 기능 요청 triage 점수표 개선 후 모니터링

### 결정
기능 요청 점수표에 구조 변경 위험, 유지보수 부담, 반자동 대안 가능성을 추가한다.

### 관찰 지표
- 개발 착수 전 보류된 기능 수
- 기술 스파이크 결과
- 출시 후 유지보수 이슈 수
- 고객 요청 재발률
- 반자동 MVP 전환 비율

### 수정 트리거
고객 가치가 높은 요청이 계속 보류되면 기술 리스크를 줄이는 공통 구조 개선을 검토한다.

### 확대 조건
유지보수 이슈가 줄고 고객 가치 높은 기능이 안정적으로 출시되면 제품 의사결정 SOP로 확장한다.

### 중단 조건
점수표가 너무 복잡해 의사결정 속도를 떨어뜨리면 핵심 항목 5개로 축소한다.

### 지식화 위치
Revit_Addin, 프로그램개발, 제품패키징

## 오늘의 사고 원칙

1. 결정은 실행 순간이 아니라 모니터링 설계까지 포함해야 완성된다.
2. 좋은 결정도 부작용 지표를 정하지 않으면 과신으로 흐른다.
3. 확대 조건과 중단 조건은 실행 전에 정해야 해석이 흔들리지 않는다.
4. 사용률, 만족도, 클레임 감소는 결과지표이면서 착시 지표가 될 수 있다.
5. 모니터링 지표가 너무 많으면 현업이 보지 않으므로 선행·결과·부작용 지표를 각각 최소 1개씩 고른다.
