# LUA BIM LABS 추론 훈련 루프 세트 12

- 생성일: 2026-06-04
- 목적: 그럴듯한 결론이 깨지는 반례를 먼저 찾아 판단을 강화한다.
- 훈련 초점: `그럴듯한 결론 → 반례 조건 → 보강 판단 → 안전한 실행`

## 반례 코드

- `X01 단계 반례`: 설계/시공/납품 단계가 달라지면 결론이 깨진다.
- `X02 고객 반례`: 고객 유형, 발주처, 계약 방식이 달라지면 결론이 깨진다.
- `X03 데이터 반례`: 측정 기준, 표본, 로그가 달라지면 결론이 깨진다.
- `X04 운영 반례`: 담당자, 시간, 승인 흐름이 달라지면 결론이 깨진다.
- `X05 기술 반례`: 모델 구조, 패밀리, 버전, 애드인이 달라지면 결론이 깨진다.
- `X06 경제성 반례`: 개발/운영/유지보수 비용이 달라지면 결론이 깨진다.

## 1. 간섭 증가율 알림은 유용하다

### 그럴듯한 결론
간섭 증가율이 급등하면 조율 리스크가 커지므로 자동 알림이 유용하다.

### 반례 조건
- 초기 모델링 단계라 물량 증가가 자연스럽다.
- 중복 간섭이 제거되지 않았다.
- 핵심 구역이 아닌 곳에서만 증가했다.
- 설계 의도상 임시 배치가 많다.

### 반례 코드
X01 단계 반례, X03 데이터 반례

### 보강 판단
간섭 증가율은 단독 지표로 쓰지 않는다. 단계, 신규 객체 수, 중복 간섭, 핵심 구역 여부를 함께 본다.

### 안전한 실행
알림을 `관찰`, `주의`, `긴급`으로 나누고 초기 모델링 단계는 별도 기준을 적용한다.

## 2. 뷰 템플릿 diff는 납품 오류를 줄인다

### 그럴듯한 결론
템플릿 변경 이력을 기록하면 납품 전 표현 오류를 줄일 수 있다.

### 반례 조건
- 템플릿 변경은 많지만 실제 납품 시트와 무관하다.
- 관리 대상 템플릿이 너무 많아 확인 비용이 커진다.
- 링크 모델 표시 변경이 원인이지 템플릿 문제가 아니다.

### 반례 코드
X03 데이터 반례, X06 경제성 반례, X05 기술 반례

### 보강 판단
뷰 템플릿 diff는 핵심 납품 시트와 연결된 템플릿에만 적용한다.

### 안전한 실행
납품 3일 전 핵심 템플릿 스냅샷과 링크 표시 방식 확인을 함께 수행한다.

## 3. 유료 파일럿 수락은 제품 수요다

### 그럴듯한 결론
고객이 유료 파일럿을 수락했으므로 제품 수요가 검증됐다.

### 반례 조건
- 고객은 도구가 아니라 책임 있는 검수 리포트를 원했다.
- 1회성 납품 압박 때문에 구매했다.
- 반복 구매 의사가 없다.
- 고객 내부 담당자가 SaaS를 운영할 시간이 없다.

### 반례 코드
X02 고객 반례, X06 경제성 반례

### 보강 판단
유료 파일럿은 수요 신호지만, 제품 수요와 서비스 수요를 분리해야 한다.

### 안전한 실행
서비스로 먼저 수행하고, 반복 항목과 고객 셀프 사용 가능 항목만 제품 후보로 분리한다.

## 4. 교육 효과가 있으면 오류가 줄어든다

### 그럴듯한 결론
교육 만족도가 높고 참석자 이해도가 좋으면 실무 오류가 줄어든다.

### 반례 조건
- 교육 참석자가 실제 작업자가 아니다.
- 작업 중 기준을 볼 수 없다.
- 템플릿이나 UI가 오류를 유발한다.
- 교육 내용과 오류 유형이 다르다.

### 반례 코드
X04 운영 반례, X03 데이터 반례

### 보강 판단
교육 효과는 만족도가 아니라 관련 오류 감소로 판단한다.

### 안전한 실행
교육자료를 작업 체크리스트, 템플릿 설명, 온보딩 퀴즈로 전환한다.

## 5. 기능 요청이 많으면 개발 우선순위가 높다

### 그럴듯한 결론
여러 고객이 요청한 기능은 제품 우선순위가 높다.

### 반례 조건
- 고객마다 말은 같지만 실제 문제는 다르다.
- 기존 기능 교육으로 해결 가능하다.
- 구조 변경 비용이 매우 크다.
- 특정 고객의 업무 방식에만 맞는다.

### 반례 코드
X02 고객 반례, X05 기술 반례, X06 경제성 반례

### 보강 판단
기능 요청은 기능명이 아니라 문제 유형으로 묶고, 기술 리스크와 유지보수 비용을 함께 평가한다.

### 안전한 실행
전면 개발 전 반자동 MVP 또는 기술 스파이크를 수행한다.

## 6. 클레임 감소는 품질 개선이다

### 그럴듯한 결론
클레임이 줄었으니 서비스 품질이 좋아졌다.

### 반례 조건
- 납품 건수가 줄었다.
- 기록 채널이 바뀌었다.
- 고객이 공식 클레임을 포기했다.
- 내부 재작업은 오히려 늘었다.

### 반례 코드
X03 데이터 반례, X02 고객 반례

### 보강 판단
클레임 수가 아니라 납품 건수 대비 클레임율, 재작업률, 심각도, 비공식 불만을 함께 본다.

### 안전한 실행
클레임 감소를 성과로 보고하기 전 기록 채널과 재작업률을 확인한다.

## 7. 외주 오류는 외주사 교육으로 해결된다

### 그럴듯한 결론
외주사 오류가 반복되면 외주사 교육을 강화하면 된다.

### 반례 조건
- 착수 기준이 불명확했다.
- 내부 검수 기준이 바뀌었다.
- 납기가 과하게 짧았다.
- 좌표/파라미터 기준이 발주서에 없었다.

### 반례 코드
X04 운영 반례, X06 경제성 반례

### 보강 판단
외주 오류는 외주사 역량, 착수자료, 검수 기준, 납기 조건을 분리해서 본다.

### 안전한 실행
교육 전에 착수자료와 접수 검수표를 보강한다.

## 8. 빠른 응답은 좋은 CS다

### 그럴듯한 결론
1차 응답 시간이 짧으면 고객지원 품질이 좋다.

### 반례 조건
- 답변이 빠르지만 해결이 안 됐다.
- 고객 상황 확인 없이 표준 답변만 보냈다.
- 재문의율이 늘었다.
- 책임자와 다음 액션이 없다.

### 반례 코드
X04 운영 반례, X02 고객 반례

### 보강 판단
CS 품질은 응답 속도, 최종 해결 시간, 재문의율, 고객 상황 이해를 함께 본다.

### 안전한 실행
표준 답변에 상황 확인 질문과 다음 액션을 포함한다.

## 종합 반례 생성 양식

```text
그럴듯한 결론:
이 결론이 깨지는 조건:
반례 코드:
보강 판단:
안전한 실행:
추가 검증 데이터:
```

## 오늘의 사고 원칙

1. 좋은 결론일수록 먼저 깨지는 조건을 찾아야 한다.
2. 반례는 결론을 폐기하기 위한 것이 아니라 적용 조건을 선명하게 하기 위한 것이다.
3. 전면 규칙보다 조건부 규칙이 현장에 더 오래 살아남는다.
4. 고객 수요, 품질 지표, 교육 효과는 모두 반례를 품고 있다.
5. 반례가 많으면 자동화보다 체크리스트나 교육이 먼저일 수 있다.
