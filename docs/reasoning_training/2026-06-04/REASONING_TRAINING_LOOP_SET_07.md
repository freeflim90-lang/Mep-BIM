# LUA BIM LABS 추론 훈련 루프 세트 07

- 생성일: 2026-06-04
- 목적: 실패, 반려, 재작업, 고객 불만을 비난이 아니라 재발 방지 지식으로 전환한다.
- 훈련 초점: `사건 → 표면 원인 → 구조 원인 → 재발 방지 → 지식화`

## 사후 분석 코드

- `P01 입력 문제`: 자료, 기준, 승인, 요구사항이 늦거나 불명확했다.
- `P02 프로세스 문제`: 순서, 담당, 승인, 검수 단계가 불명확했다.
- `P03 도구 문제`: 자동화, 템플릿, 스크립트, 로그가 부족했다.
- `P04 지식 문제`: 작업자가 기준, 예외, 판단 근거를 몰랐다.
- `P05 커뮤니케이션 문제`: 고객/내부/외주 간 기대와 책임이 맞지 않았다.
- `P06 계약/범위 문제`: 포함/제외, 변경, 유지보수, 납품 기준이 불명확했다.

## 1. 납품 반려: IFC 속성 누락

### 사건
IFC 납품 후 발주처가 필수 속성 누락을 이유로 반려했다.

### 표면 원인
작업자가 속성 입력을 빠뜨렸다.

### 구조 원인
발주처별 필수 속성 체크리스트가 없었고, IFC 내보내기 전 자동/수동 검수 단계가 분리되어 있지 않았다.

### 사후 분석 코드
P02 프로세스 문제, P04 지식 문제

### 재발 방지
1. 발주처별 필수 속성표를 만든다.
2. IFC 내보내기 전 최소 검수 항목을 SOP에 넣는다.
3. 누락 속성을 자동 점검할 수 있는 항목과 수동 확인 항목을 분리한다.
4. 반려 이력을 발주처별 지식에 남긴다.

### 지식화 위치
IFC_OpenBIM, EIRBEP_심사원, BIM_납품검수

## 2. 재작업 폭증: 고객 변경 요청 누락

### 사건
고객이 회의에서 말한 변경 요청이 모델에 반영되지 않아 재작업이 발생했다.

### 표면 원인
회의 내용을 담당자가 놓쳤다.

### 구조 원인
회의록에 변경 항목, 담당자, 기한, 승인자가 분리되어 있지 않았다. 방향 합의와 실행 합의가 섞였다.

### 사후 분석 코드
P02 프로세스 문제, P05 커뮤니케이션 문제

### 재발 방지
1. 회의록에 `결정`, `미결정`, `변경 요청`, `담당자`, `기한`, `승인자`를 분리한다.
2. 변경 요청은 모델 작업 티켓으로 전환한다.
3. 고객 확인이 필요한 항목은 작업 전 승인 상태로 둔다.

### 지식화 위치
조율차장, 고객지원운영, 요구사항분석

## 3. 자동화 사고: 태그가 잘못 배치됨

### 사건
자동 태깅 도구 실행 후 일부 뷰에서 태그가 겹치고 잘못된 객체를 가리켰다.

### 표면 원인
도구 정확도가 낮았다.

### 구조 원인
실패 객체 로그, 실행 전 대상 범위 확인, 실행 후 검토 절차, 되돌리기 안내가 부족했다.

### 사후 분석 코드
P03 도구 문제, P02 프로세스 문제

### 재발 방지
1. 실행 전 대상 뷰와 객체 수를 표시한다.
2. 실행 후 변경 수량과 실패 객체를 리포트한다.
3. 태그 겹침 가능성을 경고한다.
4. 배포 전 샘플 모델 3개 이상에서 검증한다.

### 지식화 위치
Revit_Addin, QA_테스터, 빌드검증

## 4. 외주 품질 문제: 좌표 기준 불일치

### 사건
외주 모델을 링크했더니 좌표가 맞지 않아 내부 모델과 어긋났다.

### 표면 원인
외주사가 좌표 기준을 잘못 설정했다.

### 구조 원인
외주 착수 시 좌표 기준, 기준점, 공유 좌표 확인 절차가 명확하지 않았다. 접수 검수에서도 좌표를 먼저 확인하지 않았다.

### 사후 분석 코드
P01 입력 문제, P02 프로세스 문제

### 재발 방지
1. 외주 착수 자료에 좌표 기준 설명을 포함한다.
2. 외주 제출 접수 검수의 첫 항목을 좌표 확인으로 둔다.
3. 좌표 불일치 시 재제출 기준과 책임을 계약/발주서에 명시한다.

### 지식화 위치
외주관리, BIM_납품검수, BIM_지침서

## 5. 고객 불만: 빠른 답변이지만 해결 안 됨

### 사건
CS가 빠르게 답변했지만 고객은 "상황을 이해하지 못한 답변"이라고 불만을 표시했다.

### 표면 원인
답변 내용이 부족했다.

### 구조 원인
표준 답변이 문의 유형별 상황 확인 질문 없이 바로 해결책을 제시했다. 고객의 제약 조건과 긴급도를 확인하지 않았다.

### 사후 분석 코드
P05 커뮤니케이션 문제, P04 지식 문제

### 재발 방지
1. 표준 답변 앞에 상황 확인 질문을 넣는다.
2. 기술 해결, 일정 영향, 책임 범위 답변을 분리한다.
3. 고객 긴급도와 현재 막힌 지점을 먼저 확인한다.

### 지식화 위치
CS_기술지원관, 고객지원CS, 고객지원운영

## 6. 계약 분쟁: 추가 개발 범위

### 사건
고객은 추가 기능이 계약 범위라고 주장했고, 내부 팀은 변경 요청이라고 봤다.

### 표면 원인
계약 해석이 달랐다.

### 구조 원인
제외 범위, 변경 요청 기준, 유지보수 범위, 추가 비용 기준이 견적서에 충분히 명시되지 않았다.

### 사후 분석 코드
P06 계약/범위 문제

### 재발 방지
1. 견적서에 포함/제외 범위를 분리한다.
2. 변경 요청 판정 기준을 명시한다.
3. 유지보수와 추가 개발을 구분한다.
4. 기능별 승인 기준을 둔다.

### 지식화 위치
견적심사원, 요구사항분석, BIM_프로젝트_견적산정

## 7. 교육 실패: 교육 후에도 같은 오류 반복

### 사건
파라미터 교육을 했지만 같은 입력 오류가 계속 발생했다.

### 표면 원인
작업자가 교육 내용을 기억하지 못했다.

### 구조 원인
교육 내용이 작업 중 확인표와 템플릿 안내로 연결되지 않았다. 오류가 발생하는 순간에 기준을 볼 수 없었다.

### 사후 분석 코드
P04 지식 문제, P02 프로세스 문제

### 재발 방지
1. 교육자료를 작업 체크리스트로 변환한다.
2. 템플릿 또는 패밀리 설명에 입력 기준을 넣는다.
3. 신규 작업자 온보딩 퀴즈를 만든다.
4. 오류 유형별 짧은 예시를 추가한다.

### 지식화 위치
교육컨설팅, Revit_Family제작, BIM_템플릿기획관

## 8. 일정 지연: 승인 대기 누락

### 사건
프로젝트가 4일 지연됐다. 실제 작업 시간은 충분했지만 고객 승인 대기 시간이 길었다.

### 표면 원인
작업 일정 관리가 부족했다.

### 구조 원인
일정표에 작업 시간만 있고 승인 대기, 피드백 반영, 재검수 시간이 별도 항목으로 들어가지 않았다.

### 사후 분석 코드
P02 프로세스 문제, P05 커뮤니케이션 문제

### 재발 방지
1. 일정표에 고객 승인 대기 시간을 별도 항목으로 둔다.
2. 승인 지연 시 납품 일정 영향 문구를 표준화한다.
3. 승인 요청에는 마감일과 지연 영향을 함께 쓴다.

### 지식화 위치
프로젝트분석, 고객지원운영, BIM_프로젝트_견적산정

## 9. 대시보드 오판: 평균 품질 점수만 보고 통과

### 사건
품질 평균 점수는 높았지만 핵심 구역 오류로 납품 전 긴급 수정이 발생했다.

### 표면 원인
검수자가 핵심 구역을 놓쳤다.

### 구조 원인
대시보드가 평균 점수를 강조하고 핵심 구역 최저 점수, 치명 오류 수, 구역 가중치를 표시하지 않았다.

### 사후 분석 코드
P03 도구 문제, P02 프로세스 문제

### 재발 방지
1. 핵심 구역 최저 점수를 표시한다.
2. 치명 오류 수를 평균 점수보다 우선 표시한다.
3. 핵심 구역 기준 미달 시 전체 통과를 막는다.

### 지식화 위치
BIM_납품검수, QA_테스터, 프로젝트분석

## 10. MVP 과속: 내부 도구를 고객에게 너무 빨리 배포

### 사건
내부에서 잘 쓰던 스크립트를 고객에게 배포했지만 고객 모델에서 오류가 발생했다.

### 표면 원인
고객 모델이 달랐다.

### 구조 원인
내부 도구와 고객 배포판의 품질 기준을 구분하지 않았다. 고객 모델 다양성, 설치 환경, 실패 로그, 지원 절차를 검증하지 않았다.

### 사후 분석 코드
P03 도구 문제, P06 계약/범위 문제

### 재발 방지
1. MVP, 내부 도구, 고객 배포판 기준을 분리한다.
2. 배포 전 모델 다양성 테스트를 수행한다.
3. 고객 지원 범위와 알려진 제한사항을 문서화한다.
4. 실패 로그와 복구 절차를 넣는다.

### 지식화 위치
Revit_Addin, 빌드검증, 제품패키징
