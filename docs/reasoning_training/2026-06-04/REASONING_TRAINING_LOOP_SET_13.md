- 훈련 초점: `충돌 상황 → 각 선택지의 손익 → 우선순위 기준 → 실행 순서`

## 우선순위 판단 코드

- `A01 치명도 우선`: 실패 시 납품 반려, 신뢰 하락, 계약 리스크가 큰 일을 먼저 한다.
- `A02 가역성 우선`: 되돌리기 어려운 결정을 먼저 검토하고, 되돌릴 수 있는 일은 뒤로 둔다.
- `A03 병목 제거`: 여러 일을 막고 있는 선행 조건을 먼저 해결한다.
- `A04 학습가치 우선`: 이후 반복 판단을 개선할 데이터를 주는 일을 먼저 한다.
- `A05 재사용성 우선`: 여러 프로젝트/고객에 다시 쓰일 수 있는 일을 먼저 한다.
- `A06 신뢰 회복 우선`: 고객 또는 내부 신뢰가 손상된 경우 회복 액션을 먼저 한다.

## 1. 납품 검수 vs 신규 기능 개발

### 충돌 상황
오늘 납품 전 IFC 검수가 필요하다. 동시에 고객이 요청한 애드인 신규 기능을 이번 주 안에 보여달라고 한다.

### 선택지
- A: IFC 검수를 먼저 한다.
- B: 신규 기능 프로토타입을 먼저 만든다.
- C: 둘 다 조금씩 진행한다.

### 손익 분석
- IFC 검수 지연은 납품 반려와 신뢰 하락으로 이어질 수 있다.
- 신규 기능 프로토타입 지연은 기대 관리로 조정 가능하다.
- 둘 다 조금씩 하면 납품 리스크도 줄지 않고 기능도 충분히 검증되지 않을 수 있다.

### 우선순위 코드
A01 치명도 우선, A02 가역성 우선

### 최종 판단
IFC 검수를 먼저 한다. 신규 기능은 고객에게 시연 범위와 일정을 재조정해 알린다.

### 실행 순서
1. IFC 필수 속성, 좌표, 분류체계 최소 검수
2. 납품 상태와 제한사항 문서화
3. 신규 기능 프로토타입 일정 재공지
4. 다음 검증 범위 확정

## 2. 외주사 피드백 vs 내부 교육자료 작성

### 충돌 상황
외주사 A의 반복 오류 피드백을 보내야 한다. 내부 작업자용 파라미터 교육자료도 업데이트해야 한다.

### 선택지
- A: 외주사 피드백을 먼저 보낸다.
- B: 내부 교육자료를 먼저 작성한다.
- C: 공통 오류 기준표를 먼저 만든다.

### 손익 분석
- 외주사 피드백만 보내면 내부에도 같은 오류가 반복될 수 있다.
- 내부 교육자료만 만들면 외주 재제출 비용이 계속 발생한다.
- 공통 오류 기준표는 외주 피드백과 내부 교육자료의 기반이 된다.

### 우선순위 코드
A03 병목 제거, A05 재사용성 우선

### 최종 판단
공통 오류 기준표를 먼저 만든 뒤, 외주 피드백과 내부 교육자료로 나눠 배포한다.

### 실행 순서
1. 반복 오류 5개 분류
2. 올바른 예시/잘못된 예시 작성
3. 외주사 피드백 문구 작성
4. 내부 교육자료에 같은 기준 연결

## 3. 고객 불만 응대 vs 표준 답변 개선

### 충돌 상황
한 고객이 표준 답변이 상황을 이해하지 못한다고 불만을 표시했다. 동시에 CS 표준 답변 전체를 개선해야 한다.

### 선택지
- A: 해당 고객에게 먼저 개별 응대한다.
- B: 표준 답변 전체를 먼저 개선한다.
- C: 불만 유형을 정리한 뒤 고객 응대와 표준 답변 개선을 같이 한다.

### 손익 분석
- 개별 응대는 신뢰 회복에 필요하다.
- 전체 개선을 먼저 하면 당장 불만 고객은 더 기다린다.
- 불만 유형 정리는 다음 고객에게 재사용되지만, 초기 응대가 늦어질 수 있다.

### 우선순위 코드
A06 신뢰 회복 우선, A04 학습가치 우선

### 최종 판단
먼저 고객에게 상황 재확인과 후속 조치 일정을 보낸다. 이후 해당 사례를 표준 답변 개선의 첫 사례로 사용한다.

### 실행 순서
1. 고객에게 상황 이해 부족을 인정하고 확인 질문 발송
2. 해결 담당자와 예상 시점 안내
3. 불만 원인 분류
4. 표준 답변의 첫 문장과 후속 질문 개선

## 4. 요구사항 정의 미팅 vs 빠른 견적 발송

### 충돌 상황
고객은 오늘 견적을 원한다. 내부적으로는 요구사항이 불명확해 정의 미팅이 필요하다고 본다.

### 선택지
- A: 빠르게 견적을 보낸다.
- B: 견적을 보류하고 정의 미팅만 요청한다.
- C: 가정 기반 예산 범위와 정의 미팅을 함께 제안한다.

### 손익 분석
- 빠른 견적은 수주 속도는 높일 수 있지만 범위 분쟁 위험이 크다.
- 정의 미팅만 요청하면 고객이 답답하게 느낄 수 있다.
- 예산 범위와 가정 조건을 함께 제시하면 속도와 리스크를 균형 있게 관리할 수 있다.

### 우선순위 코드
A02 가역성 우선, A06 신뢰 회복 우선

### 최종 판단
가정 기반 예산 범위를 보내고, 상세 견적은 요구사항 정의 후 확정한다고 안내한다.

### 실행 순서
1. 현재 가정 조건 명시
2. 포함/제외 범위 초안 작성
3. 예산 범위 제시
4. 요구사항 정의 미팅 일정 제안

## 5. 자동화 개발 vs 수동 체크리스트

### 충돌 상황
IFC 검수 자동화 아이디어가 있다. 하지만 바로 다음 납품에는 시간이 부족하다.

### 선택지
- A: 자동화 개발을 시작한다.
- B: 수동 체크리스트를 만든다.
- C: 납품 후까지 아무것도 하지 않는다.

### 손익 분석
- 자동화 개발은 장기 가치가 있지만 지금 납품을 보호하지 못할 수 있다.
- 수동 체크리스트는 빠르게 적용 가능하고, 이후 자동화 요구사항의 근거가 된다.
- 아무것도 하지 않으면 같은 리스크가 반복된다.

### 우선순위 코드
A01 치명도 우선, A04 학습가치 우선

### 최종 판단
수동 체크리스트를 먼저 만들고, 적용 결과를 자동화 요구사항으로 전환한다.

### 실행 순서
1. 납품 전 최소 확인 항목 7개 작성
2. 이번 납품에 적용
3. 누락/오탐/수정 시간 기록
4. 자동화 가능한 항목 분리

## 6. 기능 요청 다수 vs 기술 부채 상환

### 충돌 상황
고객들이 새 기능을 많이 요청한다. 개발자는 기존 구조를 정리하지 않으면 유지보수가 위험하다고 말한다.

### 선택지
- A: 요청 기능을 계속 추가한다.
- B: 기능 개발을 멈추고 구조 개선만 한다.
- C: 신규 기능을 반자동/작은 범위로 제한하고 핵심 구조를 정리한다.

### 손익 분석
- 기능 추가만 하면 단기 고객 반응은 좋지만 장기 유지보수가 악화된다.
- 구조 개선만 하면 고객 기대 관리가 필요하다.
- 제한 기능과 구조 개선을 병행하면 속도는 느리지만 위험을 줄인다.

### 우선순위 코드
A02 가역성 우선, A05 재사용성 우선

### 최종 판단
핵심 구조 개선을 먼저 잡고, 고객 요청은 작은 반자동 기능 또는 설정 옵션으로 제한한다.

### 실행 순서
1. 구조 변경 위험이 큰 요청 표시
2. 반자동 대안 설계
3. 핵심 구조 개선 범위 확정
4. 고객에게 단계별 제공 계획 안내

## 7. 프로젝트 일정 회복 vs 사후 분석

### 충돌 상황
프로젝트가 지연되어 일정 회복이 급하다. 동시에 왜 지연됐는지 사후 분석이 필요하다.

### 선택지
- A: 일정 회복만 한다.
- B: 사후 분석만 한다.
- C: 오늘 복구 액션과 30분 미니 사후 분석을 병행한다.

### 손익 분석
- 일정 회복만 하면 같은 지연이 반복될 수 있다.
- 사후 분석만 하면 현재 납품을 놓칠 수 있다.
- 짧은 사후 분석은 완벽하지 않아도 반복 병목을 잡는 데 충분할 수 있다.

### 우선순위 코드
A01 치명도 우선, A04 학습가치 우선

### 최종 판단
오늘 복구 액션을 먼저 정하고, 같은 날 30분 미니 사후 분석으로 대기/승인/재작업 시간을 분리한다.

### 실행 순서
1. 납품에 필요한 최소 작업 식별
2. 고객 승인 대기 항목 분리
3. 30분 사후 분석 기록
4. 다음 프로젝트 견적 기준에 반영

## 8. 대표 첨언 요청 vs 실무자 확인

### 충돌 상황
추론 다이제스트에 자동화 후보가 나왔다. 대표에게 바로 첨언을 요청할지, 실무자에게 먼저 조건을 확인할지 고민된다.

### 선택지
- A: 대표에게 바로 묻는다.
- B: 실무자에게 먼저 조건과 예외를 확인한다.
- C: 둘 다 동시에 묻는다.

### 손익 분석
- 대표에게 바로 물으면 전략 방향은 빠르게 얻지만 실무 예외가 부족할 수 있다.
- 실무자에게 먼저 물으면 질문 품질이 좋아진다.
- 둘 다 동시에 물으면 답변이 서로 다른 기준으로 섞일 수 있다.

### 우선순위 코드
A03 병목 제거, A04 학습가치 우선

### 최종 판단
실무자에게 먼저 조건/예외/데이터를 확인한 뒤, 대표에게 승격 방향을 묻는다.

### 실행 순서
1. 실무자에게 예외 조건 3개 질문
2. 자동화/교육/체크리스트 후보로 1차 분류
3. 대표에게 승격 방향 질문
4. 피드백 상태값으로 기록

## 오늘의 사고 원칙

1. 둘 다 중요한 일은 먼저 선행 병목과 치명도를 분리한다.
2. 되돌리기 어려운 결정은 빠른 실행보다 범위와 책임 확인이 먼저다.
3. 수동 체크리스트는 자동화 실패가 아니라 자동화 요구사항을 만드는 전 단계다.
4. 고객 신뢰가 손상된 경우에는 완벽한 시스템 개선보다 빠른 상황 확인이 먼저다.
5. 대표 첨언은 실무 예외를 정리한 뒤 요청할수록 실행 가능한 판단이 된다.
