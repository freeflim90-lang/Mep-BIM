# LUA BIM LABS 추론 훈련 루프 세트 06

- 생성일: 2026-06-04
- 목적: 검증된 추론 결과를 어떤 지식 베이스, SOP, 체크리스트, 자동화 요구사항으로 전환할지 판단한다.
- 훈련 초점: `검증된 판단 → 지식화 위치 → 문서 형태 → 운영 책임 → 재검토 주기`

## 지식화 형태 코드

- `K01 원칙`: 반복 적용할 사고 기준으로 기록한다.
- `K02 체크리스트`: 납품/검수/운영 전 확인 항목으로 기록한다.
- `K03 SOP`: 담당자, 순서, 산출물, 승인 조건이 있는 절차로 기록한다.
- `K04 교육자료`: 사람이 먼저 이해해야 하는 개념과 사례로 기록한다.
- `K05 자동화 요구사항`: 로그, 입력, 출력, 예외, 실패 조건이 있는 개발 요구사항으로 기록한다.
- `K06 상품/서비스 정의`: 유료 패키지, 범위, 제외 항목, 결과물로 기록한다.

## 1. 간섭 증가 알림의 조건부 지식화

### 검증된 판단
간섭 증가 알림은 모든 주차에 적용하면 알림이 많지만, 핵심 구역과 반복 간섭 비율을 함께 보면 조율 지연 방지에 도움이 된다.

### 지식화 판단
전면 자동화 요구사항보다 제한 조건이 붙은 체크리스트와 대시보드 규칙으로 기록한다.

### 권장 형태
K02 체크리스트, K05 자동화 요구사항

### 기록 위치
- `BIM_납품검수`
- `조율차장`
- `Navisworks_Addin`

### 문서화 항목
1. 핵심 구역 정의
2. 신규 객체 증가율 보정
3. 반복 간섭 제외 기준
4. 긴급/주의/관찰 알림 단계
5. 알림 후 조율 회의 개최 조건

### 재검토 주기
프로젝트 3건 적용 후 오탐률과 실제 조치율을 비교한다.

## 2. 뷰 템플릿 diff의 운영 절차화

### 검증된 판단
뷰 템플릿 diff는 납품 직전 핵심 템플릿에만 적용할 때 비용 대비 효과가 좋다.

### 지식화 판단
템플릿 관리 원칙이 아니라 납품 전 SOP로 전환한다.

### 권장 형태
K03 SOP, K02 체크리스트

### 기록 위치
- `BIM_템플릿기획관`
- `BIM_납품검수`
- `QA_테스터`

### 문서화 항목
1. 핵심 템플릿 선정 기준
2. 납품 3일 전 스냅샷 생성
3. 변경 승인자
4. 변경 diff 확인 방법
5. 오류 발생 시 복구 순서

### 재검토 주기
납품 2건마다 스냅샷 관리 시간이 과도한지 점검한다.

## 3. 자동 태깅 도구 개선 요구사항화

### 검증된 판단
자동 태깅 사용률 저하는 결과 신뢰, 실행 절차, 사용법 문제가 함께 작용했다.

### 지식화 판단
교육자료만 만들면 부족하다. 도구 개선 요구사항과 짧은 사용법 자료를 함께 만든다.

### 권장 형태
K05 자동화 요구사항, K04 교육자료

### 기록 위치
- `Revit_Addin`
- `QA_테스터`
- `교육컨설팅`

### 문서화 항목
1. 실행 전 대상 객체 수 표시
2. 실행 후 변경 수량 표시
3. 실패 객체 목록과 실패 사유
4. 되돌리기 안내
5. 1페이지 사용법 카드

### 재검토 주기
개선 배포 후 2주간 사용률, 실패율, 수동 수정 시간을 비교한다.

## 4. 납품 전 검수 서비스 상품화

### 검증된 판단
고객 일부는 SaaS보다 납품 전 1회성 검수 서비스에 먼저 비용을 지불한다.

### 지식화 판단
자동화 제품 후보로 바로 넘기지 않고 서비스 패키지로 정의한다.

### 권장 형태
K06 상품/서비스 정의

### 기록 위치
- `BIM_납품검수`
- `고객지원운영`
- `제품패키징`

### 문서화 항목
1. 서비스 대상
2. 포함 검수 항목
3. 제외 항목
4. 제출물 형식
5. 리포트 샘플
6. 소요 시간과 가격 범위
7. 고객 책임 입력자료

### 재검토 주기
유료 파일럿 3건 후 반복 항목을 자동화 후보로 분리한다.

## 5. IFC 체크리스트의 발주처별 분기

### 검증된 판단
IFC 체크리스트는 특정 발주처에서 강하게 효과가 있고, 다른 발주처에서는 효과가 약하다.

### 지식화 판단
공통 체크리스트와 발주처별 확장 체크리스트로 나눠야 한다.

### 권장 형태
K02 체크리스트, K03 SOP

### 기록 위치
- `IFC_OpenBIM`
- `EIRBEP_심사원`
- `BIM_납품검수`

### 문서화 항목
1. 공통 IFC 내보내기 설정
2. 좌표/단위/분류체계 확인
3. 발주처별 필수 속성
4. 반려 이력
5. 예외 승인 기록

### 재검토 주기
발주처별 납품 1건마다 반려/재작업 항목을 업데이트한다.

## 6. 외주 빠른 검수의 접수 기준화

### 검증된 판단
외주 빠른 검수는 치명 오류 검출 증거는 부족하지만, 반복 경미 오류를 조기에 드러낸다.

### 지식화 판단
폐기하지 않고 외주 접수 기준과 외주사 피드백 자료로 전환한다.

### 권장 형태
K02 체크리스트, K04 교육자료

### 기록 위치
- `외주관리`
- `BIM_납품검수`

### 문서화 항목
1. 접수 시 30분 빠른 검수 항목
2. 경미 오류 반복 패턴
3. 외주사별 오류 유형
4. 재제출 요청 문구
5. 정밀 검수 전환 조건

### 재검토 주기
외주사별 3회 제출 후 평가표에 반영한다.

## 7. CS 표준 답변의 관계 품질 보강

### 검증된 판단
표준 답변은 재문의를 줄였지만, 일부 고객은 상황 이해가 부족하다고 느꼈다.

### 지식화 판단
표준 답변을 폐기하지 않고 상황 확인 문장과 다음 액션 선택지를 넣어 보강한다.

### 권장 형태
K04 교육자료, K03 SOP

### 기록 위치
- `CS_기술지원관`
- `고객지원운영`
- `고객지원CS`

### 문서화 항목
1. 문의 유형별 첫 문장
2. 고객 상황 확인 질문
3. 기술 답변과 일정 답변의 문체 차이
4. 다음 액션 선택지
5. 후속 확인 기준

### 재검토 주기
월 1회 재문의율과 불만 표현을 함께 검토한다.

## 8. 요구사항 정의 단계의 규모별 분리

### 검증된 판단
유료 요구사항 정의 단계는 복잡한 개발에서 재작업을 줄이지만, 작은 요청에는 과할 수 있다.

### 지식화 판단
전 고객 공통 절차가 아니라 프로젝트 규모별 SOP로 전환한다.

### 권장 형태
K03 SOP, K06 상품/서비스 정의

### 기록 위치
- `요구사항분석`
- `견적심사원`
- `BIM_프로젝트_견적산정`

### 문서화 항목
1. 소형/중형/대형 개발 구분 기준
2. 간이 범위 가정서
3. 유료 요구사항 정의 산출물
4. 고객 승인 조건
5. 견적 변경 기준

### 재검토 주기
커스텀 개발 5건마다 견적 대비 실제 투입 시간을 비교한다.

## 9. 교육 효과의 온보딩 전환

### 검증된 판단
파라미터 교육은 참석자에게 효과가 있었지만 미참석자에게는 확산되지 않았다.

### 지식화 판단
일회성 교육자료를 온보딩 자료와 작업 중 체크리스트로 전환한다.

### 권장 형태
K04 교육자료, K02 체크리스트

### 기록 위치
- `교육컨설팅`
- `Revit_Family제작`
- `BIM_템플릿기획관`

### 문서화 항목
1. 필수 파라미터 정의
2. 흔한 입력 오류
3. 프로젝트 단계별 예외
4. 작업 중 확인표
5. 신규 작업자 온보딩 퀴즈

### 재검토 주기
신규 작업자 3명 적용 후 오류 유형 변화를 확인한다.

## 10. 기능 요청 triage의 기술 리스크 반영

### 검증된 판단
고객 가치가 높은 기능도 구조 변경과 유지보수 리스크가 크면 바로 개발하면 안 된다.

### 지식화 판단
기능 요청 점수표에 기술 리스크를 공식 항목으로 추가한다.

### 권장 형태
K01 원칙, K03 SOP

### 기록 위치
- `Revit_Addin`
- `프로그램개발`
- `제품패키징`

### 문서화 항목
1. 고객 수
2. 문제 치명도
3. 재사용성
4. 개발 시간
5. 구조 변경 위험
6. 유지보수 부담
7. 반자동 대안 가능성

### 재검토 주기
분기별로 기능 요청 백로그와 실제 개발 결과를 비교한다.

## 오늘의 사고 원칙

1. 지식화는 단순 기록이 아니라 다음 사람이 같은 판단을 반복할 수 있게 만드는 일이다.
2. SOP에는 담당자, 순서, 산출물, 승인 조건이 있어야 한다.
3. 자동화 요구사항에는 성공 조건보다 실패 조건과 로그가 더 중요할 때가 많다.
4. 교육자료는 작업 중 확인표와 온보딩으로 이어져야 효과가 확산된다.
5. 상품화는 기능이 아니라 범위, 제외 항목, 책임, 결과물 정의에서 시작한다.
