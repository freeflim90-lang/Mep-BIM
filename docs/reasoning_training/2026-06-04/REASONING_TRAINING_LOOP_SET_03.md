# LUA BIM LABS 추론 훈련 루프 세트 03

- 생성일: 2026-06-04
- 목적: 프로젝트 운영, 납품, 계약, 개발, 고객 대응에서 그럴듯하지만 위험한 1차 결론을 교정한다.
- 훈련 초점: `오판 유형 식별 → 근거 재검토 → 실행 전 방어장치 → 지식화`

## 오판 유형 코드

- `C01 성급한 일반화`: 적은 사례를 전체 규칙으로 확대한다.
- `C02 원인 단순화`: 복합 원인을 하나로 단정한다.
- `C03 해결책 선점`: 문제 정의 전에 해결책부터 고른다.
- `C04 지표 착시`: 평균, 총량, 성공률 같은 지표가 위험을 숨긴다.
- `C05 책임 경계 누락`: 납품, 검수, 승인, 유지보수 책임이 불명확하다.
- `C06 상품화 과속`: 내부 기술 또는 고객 문의를 바로 제품화한다.

## 1. 설계 변경 요청이 작아 보이는 경우

### 신호
고객이 "덕트 일부만 살짝 옮기면 된다"고 말했다. 모델러는 2시간이면 끝날 것 같다고 판단했다.

### 1차 결론
작은 수정이므로 바로 반영한다.

### 오판 유형
C02 원인 단순화, C05 책임 경계 누락

### 교정 추론
덕트 위치 변경은 단순 형상 이동이 아니라 간섭, 행거, 천장고, 장비 접속, 도면 태그, 수량표까지 영향을 줄 수 있다. 요청이 작게 들려도 영향 범위가 작다는 뜻은 아니다.

### 실행 전 방어장치
1. 변경 대상 구간과 연결 객체를 확인한다.
2. 관련 뷰, 시트, 태그, 수량표 변경 여부를 확인한다.
3. 변경 사유가 설계 변경인지 시공 조율인지 기록한다.
4. 추가 검수 시간이 필요한지 고객에게 알린다.

### 수정 결론
바로 반영하지 않고 영향 범위 확인 후 작업한다. 단순 수정으로 보이는 요청도 `변경 영향 검토` 체크리스트를 통과해야 한다.

### 승격 상태
체크리스트 후보

## 2. 테스트 통과를 배포 가능으로 착각하는 경우

### 신호
Revit 애드인 자동 태깅 기능이 샘플 모델 1개에서 정상 작동했다.

### 1차 결론
테스트가 통과했으므로 고객사에 배포한다.

### 오판 유형
C01 성급한 일반화, C06 상품화 과속

### 교정 추론
샘플 모델 1개는 프로젝트별 템플릿, 언어, 패밀리 이름, 카테고리 예외, 뷰 스케일 조건을 대표하지 못한다. 정상 작동보다 실패 시 복구와 로그가 더 중요할 수 있다.

### 실행 전 방어장치
1. 최소 3개 모델에서 테스트한다.
2. 태그 실패 객체와 실패 사유를 로그로 남긴다.
3. 실행 전후 변경 수량을 리포트한다.
4. 되돌리기 또는 백업 안내를 제공한다.

### 수정 결론
고객 배포 전 베타 검증으로 격상한다. 배포 상태는 `자동화 후보`가 아니라 `검증 중`으로 기록한다.

### 승격 상태
가설

## 3. 견적 요청을 수주 가능성으로 오해하는 경우

### 신호
잠재 고객이 "BIM 자동화 개발 견적서를 보내달라"고 요청했다.

### 1차 결론
고객이 구매 의사가 있으니 상세 견적을 바로 보낸다.

### 오판 유형
C03 해결책 선점, C06 상품화 과속

### 교정 추론
견적 요청은 가격 탐색, 내부 예산 확보, 비교 견적, 실제 구매 의사 중 하나일 수 있다. 요구사항이 불명확한 상태에서 상세 견적을 주면 가격만 기준이 되고 범위 리스크가 커진다.

### 실행 전 방어장치
1. 견적 목적을 확인한다.
2. 필수 요구사항, 제외 범위, 납기, 유지보수 여부를 묻는다.
3. 상세 견적 전 유료 진단 또는 범위 정의 단계를 제안한다.
4. 범위가 불명확하면 금액 대신 단계별 예산 범위를 제시한다.

### 수정 결론
바로 상세 견적을 보내지 않는다. `요구사항 정의 미팅 → 범위 가정서 → 견적` 순서로 진행한다.

### 승격 상태
교육자료 후보

## 4. 클레임 감소를 품질 향상으로 단정하는 경우

### 신호
최근 한 달간 고객 클레임이 40% 감소했다.

### 1차 결론
서비스 품질이 좋아졌다.

### 오판 유형
C04 지표 착시

### 교정 추론
클레임 감소는 품질 향상일 수도 있지만, 고객 사용량 감소, 문의 채널 변경, 고객 피로, 내부 기록 누락 때문일 수도 있다. 클레임 수만으로 품질을 단정하면 위험하다.

### 실행 전 방어장치
1. 프로젝트 수와 납품 건수 대비 클레임 비율을 본다.
2. 문의 채널별 기록 누락을 확인한다.
3. 재작업률, 승인 지연, 내부 수정 건수도 함께 본다.
4. 클레임 심각도별 변화를 분리한다.

### 수정 결론
클레임 감소는 긍정 신호이나 품질 향상으로 확정하지 않는다. 보조 지표와 함께 검증한다.

### 승격 상태
가설

## 5. 발주처 기준을 현장 관행으로 덮는 경우

### 신호
발주처 EIR에는 특정 파라미터 제출이 필수라고 되어 있다. 현장 담당자는 "그거 보통 안 봐요"라고 말했다.

### 1차 결론
현장에서 안 본다고 했으니 생략한다.

### 오판 유형
C05 책임 경계 누락

### 교정 추론
현장 관행과 문서 기준이 충돌할 때는 문서 기준이 계약 리스크를 만든다. 담당자가 안 본다고 해도 납품 검수 또는 감사 단계에서 문제가 될 수 있다.

### 실행 전 방어장치
1. EIR/BEP의 필수 조건을 확인한다.
2. 생략하려면 발주처 승인 또는 회의록 근거를 남긴다.
3. 생략이 모델 품질, IFC, 수량 산출에 미치는 영향을 확인한다.
4. 내부 납품 검수표에 예외로 기록한다.

### 수정 결론
문서 기준을 우선한다. 생략은 구두 관행이 아니라 승인된 예외로만 처리한다.

### 승격 상태
체크리스트 후보

## 6. 개발 속도를 팀 역량으로 착각하는 경우

### 신호
개발자가 3일 만에 Dynamo 스크립트 MVP를 만들었다.

### 1차 결론
팀이 빠르게 개발할 수 있으니 기능을 더 많이 약속한다.

### 오판 유형
C01 성급한 일반화, C06 상품화 과속

### 교정 추론
MVP 제작 속도와 운영 가능한 도구 제작 속도는 다르다. 예외 처리, UI, 로그, 권한, 문서화, 유지보수까지 포함하면 시간이 크게 늘어난다.

### 실행 전 방어장치
1. MVP, 내부 도구, 고객 배포판을 구분한다.
2. 예외 처리와 유지보수 시간을 별도로 산정한다.
3. 고객 약속 전 검증 범위를 문서화한다.
4. 기능 추가보다 실패 조건 정의를 먼저 한다.

### 수정 결론
빠른 MVP는 가능성 신호로만 본다. 고객 약속은 운영 버전 산정 후 한다.

### 승격 상태
교육자료 후보

## 7. 내부 만족도를 고객 가치로 착각하는 경우

### 신호
내부 팀이 새 대시보드가 보기 좋고 편하다고 평가했다.

### 1차 결론
고객에게도 가치가 있을 것이다.

### 오판 유형
C01 성급한 일반화, C06 상품화 과속

### 교정 추론
내부 팀과 고객은 목적, 데이터 접근권한, 용어, 의사결정 구조가 다르다. 내부 만족도는 사용성 신호이지 고객 가치 검증은 아니다.

### 실행 전 방어장치
1. 고객이 대시보드로 어떤 결정을 내리는지 확인한다.
2. 고객에게 필요한 지표와 내부 지표를 분리한다.
3. 고객 용어로 라벨과 리포트를 바꾼다.
4. 고객 1곳에서 실제 회의 자료로 사용해본다.

### 수정 결론
내부 대시보드는 고객용 상품이 아니라 고객 검증 전 프로토타입이다.

### 승격 상태
가설

## 8. 일정 지연을 작업량 문제로만 보는 경우

### 신호
이번 프로젝트가 예정보다 5일 늦어졌다. 팀은 작업량이 많아서라고 보고했다.

### 1차 결론
다음 프로젝트에는 인력을 더 투입한다.

### 오판 유형
C02 원인 단순화

### 교정 추론
일정 지연은 작업량, 입력자료 지연, 승인 대기, 재작업, 의사결정 병목, 외주 품질 문제 등이 섞인 결과다. 인력 추가가 병목을 해결하지 못할 수 있다.

### 실행 전 방어장치
1. 지연 시간을 원인별로 분류한다.
2. 대기 시간과 실제 작업 시간을 분리한다.

4. 승인권자와 입력자료 도착 시점을 확인한다.

### 수정 결론
인력 추가 전에 병목 분석을 한다. 대기/승인 지연이 크면 운영 규칙 개선이 우선이다.

### 승격 상태
체크리스트 후보

## 9. 문의 응답 속도를 고객지원 품질로 착각하는 경우

### 신호
CS 평균 1차 응답 시간이 10분으로 줄었다.

### 1차 결론
고객지원 품질이 좋아졌다.

### 오판 유형
C04 지표 착시

### 교정 추론
빠른 응답은 좋지만 해결 품질과 다르다. 1차 응답이 빨라도 재문의가 늘거나 해결 시간이 길어지면 고객 경험은 나빠질 수 있다.

### 실행 전 방어장치
1. 1차 응답 시간과 최종 해결 시간을 함께 본다.
2. 재문의율과 동일 이슈 반복률을 확인한다.
3. 해결 문서화 여부를 확인한다.
4. 고객 만족 또는 불만 표현을 분류한다.

### 수정 결론
응답 속도는 보조 지표다. CS 품질은 해결률, 재문의율, 지식화율과 함께 본다.

### 승격 상태
체크리스트 후보

## 10. 외주 결과물을 내부 품질로 착각하는 경우

### 신호
외주사가 모델을 기한 내 제출했다. 파일을 열어보니 겉보기에는 문제가 없어 보인다.

### 1차 결론
납품 가능 상태로 본다.

### 오판 유형
C04 지표 착시, C05 책임 경계 누락

### 교정 추론
겉보기와 납품 품질은 다르다. 파라미터, 좌표, 링크, 명명 규칙, 뷰/시트, IFC 내보내기, 발주처 기준 충족 여부는 별도로 검수해야 한다.

### 실행 전 방어장치
1. 외주 납품 검수표를 적용한다.
2. 치명 오류와 경미 오류를 분리한다.
3. 수정 책임과 재제출 기한을 명확히 한다.
4. 내부 납품 전 최종 검수자를 지정한다.

### 수정 결론
기한 내 제출은 접수 완료일 뿐이다. 납품 가능 판정은 내부 검수 후에만 한다.

### 승격 상태
체크리스트 후보

## 12. 기능 요청 증가를 제품 성장으로 착각하는 경우

### 신호
사용자들이 애드인에 기능 요청을 많이 보내고 있다.

### 1차 결론
기능을 많이 추가하면 제품이 성장한다.

### 오판 유형
C03 해결책 선점, C06 상품화 과속

### 교정 추론
기능 요청 증가는 관심 신호이지만, 무분별한 추가는 복잡도와 유지보수 비용을 키운다. 요청 빈도, 사용자 유형, 문제의 공통성, 기존 기능으로 해결 가능 여부를 봐야 한다.

### 실행 전 방어장치
1. 요청을 문제 유형으로 묶는다.
2. 매출/재사용/유지보수 영향으로 점수화한다.
3. 기존 기능 개선으로 해결 가능한지 확인한다.
4. 기능 추가보다 워크플로우 단순화가 나은지 검토한다.

### 수정 결론
기능 요청은 바로 개발하지 않고 제품 의사결정 백로그로 분류한다.

### 승격 상태
체크리스트 후보

## 오늘의 사고 원칙

1. 작은 변경 요청도 영향 범위가 크면 작은 일이 아니다.
2. 샘플 성공은 배포 가능성과 다르다.
3. 견적 요청은 구매 의사가 아니라 탐색 신호일 수 있다.
4. 문서 기준과 현장 관행이 충돌하면 승인된 예외만 믿는다.
5. 빠른 MVP는 고객 약속의 근거가 아니라 검증 시작점이다.
6. 회의 방향 합의는 담당자, 기한, 승인자가 있어야 실행 합의가 된다.
