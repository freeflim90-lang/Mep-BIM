# LUA BIM LABS 추론 훈련 루프 세트 14

- 생성일: 2026-06-04
- 목적: 리스크와 가치를 점수화하되, 숫자가 숨기는 책임·치명도·맥락을 보정한다.
- 훈련 초점: `기본 점수 → 점수 착시 → 보정 질문 → 보정 판정 → 실행 기준`

## 점수 보정 코드

- `S01 치명도 보정`: 발생 빈도는 낮아도 납품 반려, 안전, 계약 리스크가 크면 점수를 올린다.
- `S02 책임 보정`: 고객/발주처/내부 승인 책임이 불명확하면 점수를 올린다.
- `S03 재사용성 보정`: 여러 프로젝트에 반복 적용되면 점수를 올린다.
- `S04 운영비 보정`: 유지보수, 교육, 검수 비용이 크면 점수를 낮춘다.
- `S05 증거수준 보정`: 근거가 약하면 점수를 낮추거나 검증으로 돌린다.
- `S06 신뢰 보정`: 고객 또는 내부 신뢰에 직접 영향을 주면 점수를 올린다.

## 기본 점수식

```text
기본 우선순위 = 빈도 x 치명도 x 재사용성 - 운영비
보정 우선순위 = 기본 우선순위 + 책임/신뢰/증거수준 보정
```

점수는 결론이 아니라 대화 시작점이다. 점수가 높아도 책임 경계가 불명확하면 바로 실행하지 않는다.

## 1. IFC 오류는 드물지만 반려 가능성이 큼

### 상황
IFC 속성 누락 오류는 최근 2개월간 1회만 발생했다. 그러나 발생 시 발주처 반려와 재납품이 필요하다.

### 기본 점수
- 빈도: 1
- 치명도: 5
- 재사용성: 4
- 운영비: 2
- 기본 우선순위: 18

### 점수 착시
빈도가 낮아 보류하기 쉽다. 하지만 반려 리스크와 책임 비용이 점수에 충분히 반영되지 않았다.

### 보정 질문
- 반려 시 재납품 일정과 고객 신뢰 손상은 어느 정도인가?
- 발주처별 필수 속성표가 있는가?
- 자동 확인 가능한 항목은 몇 개인가?

### 보정 코드
S01 치명도 보정, S02 책임 보정

### 보정 판정
우선순위를 올린다. 전면 자동화보다 발주처별 IFC 사전 체크리스트를 먼저 만든다.

### 실행 기준
다음 IFC 납품 전 공통 필수 속성, 좌표, 분류체계만이라도 최소 검수한다.

## 2. 자동 태깅 기능 요청이 많지만 유지보수 비용도 큼

### 상황
고객 4곳이 자동 태깅 고도화를 요청했다. 개발자는 기존 구조를 바꾸면 유지보수 비용이 커진다고 본다.

### 기본 점수
- 빈도: 4
- 치명도: 3
- 재사용성: 5
- 운영비: 5
- 기본 우선순위: 55

### 점수 착시
고객 수와 재사용성 때문에 점수가 높다. 그러나 구조 변경 위험과 유지보수 비용이 장기 리스크를 키운다.

### 보정 질문
- 반자동 승인 흐름으로 고객 문제를 줄일 수 있는가?
- 구조 변경 없이 해결 가능한 1차 범위는 무엇인가?
- 실패 로그와 되돌리기 없이 배포해도 되는가?

### 보정 코드
S04 운영비 보정, S05 증거수준 보정

### 보정 판정
전면 개발은 보류하고 기술 스파이크와 반자동 MVP로 돌린다.

### 실행 기준
구조 변경 위험 평가, 실패 로그 설계, 반자동 MVP 범위를 먼저 문서화한다.

## 3. 고객 불만 1건이지만 핵심 고객임

### 상황
CS 표준 답변에 대한 불만은 1건뿐이다. 그러나 해당 고객은 반복 프로젝트를 주는 핵심 고객이다.

### 기본 점수
- 빈도: 1
- 치명도: 3
- 재사용성: 3
- 운영비: 1
- 기본 우선순위: 8

### 점수 착시
빈도만 보면 낮은 우선순위다. 하지만 핵심 고객 신뢰와 반복 매출 영향이 반영되지 않았다.

### 보정 질문
- 이 고객의 불만이 다른 고객에게도 잠재적으로 발생할 수 있는가?
- 불만 원인은 문체인가, 상황 이해 부족인가, 책임 경계인가?

### 보정 코드

### 보정 판정
우선순위를 올린다. 전체 CS 개편보다 해당 고객 신뢰 회복과 표준 답변 첫 문장 보강을 먼저 한다.

### 실행 기준
오늘 중 상황 확인 질문과 후속 일정 안내를 보낸다.

## 4. 외주 경미 오류가 자주 발생함

### 상황
외주 모델에서 경미 오류가 매번 발생한다. 치명 오류는 없지만 내부 수정 시간이 누적된다.

### 기본 점수
- 빈도: 5
- 치명도: 2
- 재사용성: 4
- 운영비: 2
- 기본 우선순위: 38

### 점수 착시
치명도가 낮아 보이지만 반복 수정 시간이 누적된다. 외주사별 편차도 점수에 보이지 않는다.

### 보정 질문
- 경미 오류 수정 시간이 월 몇 시간인가?
- 외주사별로 오류 유형이 반복되는가?
- 착수자료 개선으로 줄일 수 있는가?

### 실행 기준
오류 유형 5개를 기준표로 만들고 외주사별 제출 3회 후 변화량을 본다.

## 5. 교육 만족도는 높지만 오류 개선 근거가 약함

### 상황
파라미터 교육 만족도는 높다. 그러나 오류 감소 데이터는 아직 없다.

### 기본 점수
- 빈도: 3
- 치명도: 3
- 재사용성: 4
- 운영비: 2
- 기본 우선순위: 34

### 점수 착시
만족도가 높으면 효과도 높다고 느끼기 쉽다. 그러나 만족도는 실무 오류 감소의 직접 증거가 아니다.

### 보정 질문
- 교육 내용과 연결된 오류 유형이 줄었는가?
- 참석자와 실제 작업자가 같은가?
- 작업 중 체크리스트가 제공됐는가?

### 보정 코드
S05 증거수준 보정

### 보정 판정
교육 반복이 아니라 효과 측정 설계를 먼저 한다. 관련 오류만 분리해 전후 비교한다.

### 실행 기준
교육 전후 오류 유형 매핑표와 참석자/미참석자 비교 기준을 만든다.

## 6. 요구사항 정의 단계가 비용처럼 보임

### 상황
고객은 요구사항 정의 비용을 부담스러워한다. 내부적으로는 정의 단계가 없으면 재작업이 크다.

### 기본 점수
- 빈도: 4
- 치명도: 4
- 재사용성: 5
- 운영비: 3
- 기본 우선순위: 77

### 점수 착시
운영비가 보이기 때문에 고객에게 부담처럼 보인다. 하지만 재작업, 범위 분쟁, 일정 지연을 줄이는 방어장치다.

### 보정 질문
- 요구사항 정의가 없었던 프로젝트의 재작업 비용은 얼마인가?
- 소형 요청에도 같은 절차가 필요한가?
- 산출물을 고객이 가치로 느끼게 보여줄 수 있는가?

### 보정 코드
S02 책임 보정, S03 재사용성 보정

### 보정 판정
대형/복합 개발에는 우선순위를 높인다. 소형 개발은 간이 범위 가정서로 대체한다.

### 실행 기준
소형/중형/대형 구분 기준과 각 단계별 산출물을 정의한다.

## 7. 뷰 템플릿 diff는 효과가 있지만 운영비가 큼

뷰 템플릿 diff가 표현 오류를 잡았지만 매일 관리하기에는 부담이 크다.

- 운영비: 4
- 기본 우선순위: 32

### 보정 질문

S04 운영비 보정, S01 치명도 보정

전면 운영은 보류하고 납품 직전 핵심 템플릿 SOP로 제한 승격한다.

## 8. 클레임 감소가 성과처럼 보임

### 상황
클레임이 40% 줄었다. 팀은 서비스 품질 개선 성과로 보고하려 한다.

### 기본 점수
- 빈도: 4
- 치명도: 3
- 재사용성: 3
- 운영비: 1
- 기본 우선순위: 35

### 점수 착시
좋은 지표가 곧 좋은 성과처럼 보인다. 하지만 납품 건수, 기록 채널, 내부 재작업률이 빠져 있다.

### 보정 질문
- 납품 건수 대비 클레임율도 줄었는가?
- 비공식 불만과 재문의는 줄었는가?
- 내부 재작업률도 줄었는가?

### 보정 코드
S05 증거수준 보정

### 보정 판정
성과 보고 전 추가 검증으로 돌린다. 클레임 수만으로 품질 개선을 확정하지 않는다.

### 실행 기준
납품 건수 대비 클레임율, 재작업률, 채널별 불만 기록을 함께 보고한다.

## 오늘의 사고 원칙

1. 점수는 판단을 돕는 도구이지 결론이 아니다.
2. 빈도가 낮아도 치명도와 책임이 크면 우선순위를 올린다.
3. 점수가 높아도 증거가 약하면 검증으로 돌린다.
4. 운영비가 큰 항목은 폐기보다 제한 승격을 먼저 검토한다.
5. 고객 신뢰와 계약 책임은 단순 빈도 점수로 과소평가되기 쉽다.
