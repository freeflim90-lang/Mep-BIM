# LUA BIM LABS 추론 훈련 루프 세트 20

- 생성일: 2026-06-04
- 목적: 커뮤니케이션 후속 응답을 실제 실행 티켓, 완료 기준, 검증 항목으로 전환한다.
- 훈련 초점: `후속 응답 → 실행 티켓 → 담당자 → 완료 기준 → 검증 데이터`

## 실행 전환 코드

- `ACT01 고객 확인`: 고객이 선택하거나 승인해야 하는 항목이다.
- `ACT02 내부 작업`: 내부 팀이 산출물을 만들어야 하는 항목이다.
- `ACT03 검수/QA`: 품질 기준 충족 여부를 확인해야 하는 항목이다.
- `ACT04 문서/SOP`: 반복될 절차나 기준을 문서화해야 하는 항목이다.
- `ACT05 개발/자동화`: 도구, 로그, 리포트, 자동화 기능으로 구현해야 하는 항목이다.
- `ACT06 기록/증빙`: 책임 경계, 승인, 변경 이력, 근거를 남겨야 하는 항목이다.

## 1. 임시 제출을 최소 검수 제출본으로 기록

### 후속 응답
오늘 제출 처리는 가능하되, 최소 검수 제출본으로 기록하고 미검수 항목을 명시한다.

### 실행 티켓
최소 검수 제출본 메일과 미검수 항목 첨부 작성

### 담당자
PM 또는 납품 담당자

### 완료 기준
- 메일 본문에 `최소 검수 제출본` 상태가 명시됨
- 확인된 항목과 미검수 항목이 분리됨
- 정식 검수 완료본 제출 예정일이 명시됨

### 검증 데이터
메일 사본, 미검수 항목 목록, 정식 제출 일정

### 실패 조건
고객이 정식 납품으로 오해할 수 있는 표현이 남아 있으면 실패다.

### 실행 전환 코드
ACT01 고객 확인, ACT06 기록/증빙

## 2. 일정 단축 요청을 범위 선택지로 전환

### 후속 응답
핵심 변경 우선 반영과 전체 변경 반영 중 고객이 선택할 수 있게 한다.

### 실행 티켓
일정 단축 선택지 표 작성

### 담당자
PM, 실무 리드

### 완료 기준
- 핵심 반영 범위와 전체 반영 범위가 구분됨
- 각 선택지별 제출 가능일이 명시됨
- 고객 선택이 기록됨

### 검증 데이터
선택지 표, 고객 회신, 일정표 변경 이력

### 실패 조건
범위 선택 없이 단순히 더 빠른 일정만 약속하면 실패다.

### 실행 전환 코드
ACT01 고객 확인, ACT06 기록/증빙

## 3. 자동 태깅 사용 가능 범위 안내를 검증 티켓으로 전환

### 후속 응답
지정 뷰와 지정 카테고리에서 사용하고, 수정 사유를 기록한다.

### 실행 티켓
자동 태깅 제한 적용 파일럿 운영

### 담당자
개발자, QA, 실무 사용자

### 완료 기준
- 적용 대상 뷰와 카테고리가 명시됨
- 수동 수정 사유 기록 양식이 준비됨
- 파일럿 기간과 결과 리뷰일이 정해짐

### 검증 데이터
사용률, 수동 수정 시간, 수정 사유, 실패 객체 로그

### 실패 조건
사용 가능 범위가 불명확하거나 수정 사유가 기록되지 않으면 실패다.

### 실행 전환 코드
ACT03 검수/QA, ACT05 개발/자동화

## 4. 외주 기준 미수령 반응을 착수 패키지 개선으로 전환

### 후속 응답
착수 시 전달 기준을 내부 확인하고, 다음 제출부터 기준표를 공통 기준으로 적용한다.

### 실행 티켓
외주 착수 패키지 기준표 포함 여부 점검

### 담당자
외주관리 담당자, PM

### 완료 기준
- 기존 착수자료 전달 이력이 확인됨
- 기준표가 외주 착수 패키지에 포함됨
- 다음 제출부터 적용 기준이 외주사에 재공지됨

### 검증 데이터
착수자료 이력, 기준표 파일, 외주사 회신

### 실패 조건
외주사 책임만 기록하고 내부 전달 이력을 확인하지 않으면 실패다.

### 실행 전환 코드
ACT04 문서/SOP, ACT06 기록/증빙

## 5. 내부 보고용 견적 요청을 범위 견적으로 전환

### 후속 응답
가정 조건이 붙은 예산 범위를 제공하고, 상세 견적은 요구사항 정의 후 확정한다.

### 실행 티켓
가정 기반 범위 견적서 작성

### 담당자
견적 담당자, 요구사항 분석 담당자

### 완료 기준
- 가정 조건이 명시됨
- 포함/제외 범위가 분리됨
- 변동 가능성이 표시됨
- 상세 견적 확정 조건이 명시됨

### 검증 데이터
범위 견적서, 고객 전달 메일, 요구사항 정의 미팅 일정

### 실패 조건
고정 견적으로 오해될 수 있는 금액 표현이 있으면 실패다.

### 실행 전환 코드
ACT01 고객 확인, ACT06 기록/증빙

## 6. 제품화 시점 질문을 파일럿 판단 기준으로 전환

### 후속 응답
유료 파일럿 3건에서 반복 항목과 셀프 사용 가능 항목이 확인되면 제품화 후보로 분리한다.

### 실행 티켓
납품 검수 서비스 파일럿 제품화 판단표 작성

### 담당자
제품기획, 검수 서비스 담당자, 대표 보고 담당자

### 완료 기준
- 파일럿 수 기준이 정해짐
- 반복 검수 항목 기록표가 있음
- 셀프 사용 가능 항목 분류 기준이 있음
- 제품화 판단일이 정해짐

### 검증 데이터
파일럿 리포트, 반복 항목 목록, 고객 재구매 의사, 셀프 사용 가능성 평가

### 실패 조건
파일럿 결과 없이 제품 개발 일정만 먼저 잡으면 실패다.

### 실행 전환 코드
ACT04 문서/SOP, ACT06 기록/증빙

## 7. 기술 리스크 우려를 상세 검토 전환 조건으로 전환

### 후속 응답
1차 필터에도 구조 변경 위험과 유지보수 부담을 필수 항목으로 남긴다.

### 실행 티켓
기능 요청 1차 필터 개정

### 담당자
제품기획, 개발 리드

### 완료 기준
- 1차 필터에 구조 변경 위험 항목이 포함됨
- 유지보수 부담 항목이 포함됨
- 두 항목 중 하나라도 높으면 상세 검토로 전환하는 기준이 명시됨

### 검증 데이터
개정된 triage 표, 샘플 기능 요청 3건 적용 결과

### 실패 조건
1차 필터가 단순 고객 요청 수만 보고 통과시키면 실패다.

### 실행 전환 코드
ACT04 문서/SOP, ACT05 개발/자동화

## 8. 상담자 문의 분류 어려움을 예시 표로 전환

### 후속 응답
일정 영향, 비용/계약 책임, 고객 업무 중단이 포함되면 고위험 문의로 분류한다.

### 실행 티켓
고위험/일반 문의 예시 분류표 작성

### 담당자
CS 운영 담당자, 기술지원 리드

### 완료 기준
- 문의 예시 10개 이상이 분류됨
- 고위험 문의 기준 3개가 명시됨
- 상세 템플릿 적용 조건이 정해짐
- 상담자가 애매할 때 문의할 담당자가 명시됨

### 검증 데이터
문의 분류표, 상담자 피드백, 재문의율 변화

### 실패 조건
분류 기준이 추상적이라 상담자가 다시 임의 판단해야 하면 실패다.

### 실행 전환 코드
ACT04 문서/SOP, ACT03 검수/QA

## 오늘의 사고 원칙

1. 좋은 후속 응답은 반드시 실행 티켓으로 바뀌어야 한다.
2. 실행 티켓에는 담당자보다 완료 기준이 더 중요하다.
3. 완료 기준이 없으면 커뮤니케이션은 친절하지만 실행은 흐려진다.
4. 고객 선택, 내부 작업, QA, 문서화, 개발, 증빙을 분리해야 일이 공중에 뜨지 않는다.
5. 실패 조건을 먼저 적으면 애매한 완료 보고를 줄일 수 있다.
