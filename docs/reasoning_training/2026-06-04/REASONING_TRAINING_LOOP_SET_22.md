# LUA BIM LABS 추론 훈련 루프 세트 22

- 생성일: 2026-06-04
- 목적: 완료 검증에서 발견된 부분 완료, 증빙 부족, 기준 불일치를 재작업 계획으로 전환한다.
- 훈련 초점: `검증 판정 → 부족한 항목 → 재작업 범위 → 우선순위 → 재완료 기준`

## 재작업 전환 코드

- `RW01 증빙 보강`: 실제 작업은 됐지만 증거가 부족해 기록을 보강한다.
- `RW02 범위 보정`: 작업 범위가 원래 기준과 달라 포함/제외 범위를 다시 맞춘다.
- `RW03 산출물 수정`: 산출물 자체가 기준을 만족하지 못해 수정한다.
- `RW04 승인 재요청`: 고객, 외주, 내부 승인 또는 선택이 필요하다.
- `RW05 샘플 검증`: 표나 기준을 실제 사례에 적용해 검증한다.
- `RW06 SOP 반영`: 반복 방지를 위해 절차나 템플릿을 수정한다.

## 1. 최소 검수 제출본 메일 상태 문구 누락

### 검증 판정
DONE04 기준 불일치. 메일은 발송됐지만 `최소 검수 제출본` 상태가 명확하지 않다.

### 부족한 항목
- 제출본 상태 문구
- 미검수 항목 목록
- 정식 제출 예정일

### 재작업 범위
정정 메일을 보내고, 기존 제출본의 상태를 명확히 재기록한다.

### 우선순위
높음. 고객이 정식 납품으로 오해할 수 있어 책임 리스크가 크다.

### 재완료 기준
- 정정 메일 발송
- 미검수 항목 첨부
- 정식 제출 예정일 명시
- 내부 납품 기록에 제출본 상태 업데이트

### 재작업 전환 코드
RW01 증빙 보강, RW02 범위 보정

## 2. 일정 선택지 표는 보냈지만 고객 선택이 없음

### 검증 판정
DONE06 보류 유지. 선택지 표는 작성됐지만 고객 승인 전이다.

### 부족한 항목
- 고객 선택 회신
- 회신 기한
- 승인 전 작업 착수 금지 기준

### 재작업 범위
고객에게 선택지 회신 기한을 요청하고, 승인 전 진행 가능한 준비 작업과 승인 후 작업을 분리한다.

### 우선순위
높음. 고객 선택 없이 작업하면 범위 분쟁이 생길 수 있다.

### 재완료 기준
- 고객 선택 회신 확보
- 회신 기한 기록
- 승인 전/후 작업 구분
- 일정표 업데이트

### 재작업 전환 코드
RW04 승인 재요청, RW02 범위 보정

## 3. 자동 태깅 파일럿 실행 데이터 부족

### 검증 판정
DONE03 증빙 부족. 파일럿은 실행됐지만 수정 사유와 실패 로그가 없다.

### 부족한 항목
- 수동 수정 사유
- 실패 객체 로그
- 적용 범위와 실제 실행 범위 비교

### 재작업 범위
파일럿 완료를 취소하지는 않되, `데이터 부족 파일럿`으로 상태를 낮추고 재검증을 설계한다.

### 우선순위
중간. 기능 자체보다 판단 근거가 부족한 문제다.

### 재완료 기준
- 수정 사유 기록 양식 배포
- 실패 객체 로그 수집
- 동일 조건으로 1주 재검증
- 결과 리뷰 기록

### 재작업 전환 코드
RW01 증빙 보강, RW05 샘플 검증

## 4. 외주 기준표만 만들고 공지가 안 됨

### 검증 판정
DONE02 부분 완료. 기준표는 있으나 외주사 적용 공지가 없다.

### 부족한 항목
- 외주사 공지 메일
- 적용 시점
- 외주사 확인 회신
- 착수 패키지 포함 여부

### 재작업 범위
기준표를 착수 패키지에 넣고, 다음 제출부터 적용한다고 외주사에 공지한다.

### 우선순위
중간. 다음 외주 제출 전까지는 반드시 완료해야 한다.

### 재완료 기준
- 착수 패키지 업데이트
- 외주사 공지 발송
- 적용 시점 명시
- 회신 또는 확인 기록 확보

### 재작업 전환 코드
RW04 승인 재요청, RW06 SOP 반영

## 5. 범위 견적서가 고정 견적으로 오해될 수 있음

### 검증 판정
DONE04 기준 불일치. 견적서가 범위 견적인지 명확하지 않다.

### 부족한 항목
- 가정 조건
- 변동 가능성 문구
- 제외 범위
- 상세 견적 확정 조건

### 재작업 범위
견적서를 재발행하거나 보완 메일을 보내 `가정 기반 범위 견적`임을 명확히 한다.

### 우선순위
높음. 계약/범위 분쟁으로 이어질 수 있다.

### 재완료 기준
- 가정 조건 추가
- 포함/제외 범위 분리
- 상세 견적 확정 조건 명시
- 고객에게 보완본 전달

### 재작업 전환 코드
RW02 범위 보정, RW01 증빙 보강

## 6. 제품화 판단표에 파일럿 데이터 책임자가 없음

### 검증 판정
DONE02 부분 완료. 판단표는 있으나 데이터 수집 책임이 없다.

### 부족한 항목
- 파일럿별 데이터 담당자
- 판단 회의일
- 반복 항목 기록 기준
- 셀프 사용 가능성 평가 기준

### 재작업 범위
제품화 판단표를 책임자와 일정이 있는 운영표로 바꾼다.

### 우선순위
중간. 당장 고객 리스크는 낮지만 제품화 판단이 흐려진다.

### 재완료 기준
- 파일럿 담당자 지정
- 판단 회의일 기록
- 반복 항목 기록 기준 추가
- 셀프 사용 가능성 평가 기준 추가

### 재작업 전환 코드
RW01 증빙 보강, RW06 SOP 반영

## 7. 기능 요청 필터는 개정됐지만 샘플 적용이 없음

### 검증 판정
DONE02 부분 완료. 표는 개정됐지만 실제 요청에 적용되지 않았다.

### 부족한 항목
- 샘플 기능 요청 3건 적용 결과
- 상세 검토 전환 사례
- 보류/반자동 대안 판정 근거

### 재작업 범위
최근 기능 요청 3건에 개정 필터를 적용하고, 기존 판단과 차이를 비교한다.

### 우선순위
중간. 표가 실제 의사결정에 작동하는지 확인해야 한다.

### 재완료 기준
- 샘플 3건 적용
- 각 요청의 판정 코드 기록
- 상세 검토 전환 기준 검증
- 개발 리드 피드백 확보

### 재작업 전환 코드
RW05 샘플 검증, RW01 증빙 보강

## 8. 문의 분류표가 상담자에게 어렵다는 피드백

### 검증 판정
DONE05 재작업 필요. 문서는 있으나 현장 사용성이 낮다.

### 부족한 항목
- 상담자 기준 이해도
- 애매한 문의 예시
- 에스컬레이션 경로
- 고위험/일반 판단 기준의 단순화

### 재작업 범위
분류표를 현장 언어로 줄이고, 애매한 문의 예시와 에스컬레이션 경로를 추가한다.

### 우선순위
중간. 잘못 분류하면 고객 리스크 대응이 늦어질 수 있다.

### 재완료 기준
- 고위험 기준 3개 이하로 단순화
- 애매한 문의 예시 5개 추가
- 상담자 2명 샘플 분류 테스트
- 에스컬레이션 담당자 명시

### 재작업 전환 코드
RW03 산출물 수정, RW05 샘플 검증

## 오늘의 사고 원칙

1. 미완료를 발견했을 때 바로 비난하지 말고 부족한 항목을 분리한다.
2. 재작업은 "다시 하기"가 아니라 기준에 맞지 않은 부분만 좁혀 고치는 일이다.
3. 외부 승인이나 고객 선택이 없으면 작업을 더 해도 완료가 될 수 없다.
4. 샘플 적용이 빠진 표준표는 실제 운영 지식이 아니라 초안이다.
5. 재완료 기준이 없으면 같은 미완료가 반복된다.
