# LUA BIM LABS 추론 훈련 루프 세트 21

- 생성일: 2026-06-04
- 목적: 실행 티켓의 완료 보고를 검증하고, 실제 완료/부분 완료/재작업/보류를 판정한다.
- 훈련 초점: `완료 보고 → 필요한 증거 → 검증 질문 → 판정 → 후속 조치`

## 완료 검증 코드

- `DONE01 실제 완료`: 완료 기준과 증빙이 모두 충족됐다.
- `DONE02 부분 완료`: 일부 기준은 충족됐지만 잔여 항목이 있다.
- `DONE03 증빙 부족`: 완료됐다고 말하지만 확인 자료가 없다.
- `DONE04 기준 불일치`: 작업은 했지만 원래 완료 기준과 다르다.
- `DONE05 재작업 필요`: 산출물이 기준을 만족하지 못해 수정이 필요하다.
- `DONE06 보류 유지`: 외부 승인, 고객 선택, 데이터 부족으로 완료 판정할 수 없다.

## 1. 최소 검수 제출본 메일 발송 완료 보고

### 완료 보고
고객에게 오늘 제출본 메일을 보냈습니다.

### 필요한 증거
- 메일 본문
- `최소 검수 제출본` 상태 문구
- 확인 항목과 미검수 항목 목록
- 정식 검수 완료본 제출 예정일

### 검증 질문
- 고객이 정식 납품으로 오해하지 않도록 상태가 명시됐는가?
- 미검수 항목이 본문 또는 첨부에 남아 있는가?
- 정식 제출 예정일이 기록됐는가?

### 판정
상태 문구와 미검수 항목이 있으면 DONE01. 파일만 보냈고 상태가 불명확하면 DONE04.

### 후속 조치
상태가 불명확하면 정정 메일을 보내고, 메일 제목 또는 본문에 제출본 상태를 보강한다.

## 2. 일정 단축 선택지 표 작성 완료 보고

### 완료 보고
고객에게 일정 선택지를 보냈습니다.

### 필요한 증거
- 선택지 표
- 핵심 반영/전체 반영 범위 구분
- 선택지별 제출 가능일
- 고객 선택 회신

### 검증 질문
- 일정만 다른가, 실제 반영 범위도 다르게 설명됐는가?
- 고객이 어떤 선택지를 승인했는가?
- 승인 전 작업이 진행되고 있지는 않은가?

### 판정
고객 선택 회신까지 있으면 DONE01. 선택지 표만 보냈으면 DONE06.

### 후속 조치
고객 선택이 없으면 작업 착수 기준을 보류로 기록하고 회신 기한을 요청한다.

## 3. 자동 태깅 제한 파일럿 완료 보고

### 완료 보고
자동 태깅 파일럿을 진행했습니다.

### 필요한 증거
- 적용 대상 뷰와 카테고리 목록
- 실행 횟수와 사용률
- 수동 수정 시간
- 수정 사유 기록
- 실패 객체 로그

### 검증 질문
- 제한 범위가 사전에 정의된 범위와 일치하는가?
- 사용률만 기록하고 수정 사유는 빠지지 않았는가?
- 실패 객체뿐 아니라 수동 수정된 성공 객체도 기록됐는가?

### 판정
수정 사유와 실패 로그가 있으면 DONE01. 실행만 했고 데이터가 없으면 DONE03.

### 후속 조치
데이터가 부족하면 파일럿 완료가 아니라 재검증으로 돌리고 기록 양식을 먼저 보강한다.

## 4. 외주 착수 패키지 개선 완료 보고

### 완료 보고
외주 착수 패키지에 기준표를 넣었습니다.

### 필요한 증거
- 개정된 착수 패키지 파일
- 기준표 포함 위치
- 외주사 공지 메일
- 다음 제출부터 적용한다는 회신 또는 확인 기록

### 검증 질문
- 기준표가 실제 착수 패키지 안에 포함됐는가, 별도 파일로만 존재하는가?
- 외주사에 적용 시점이 전달됐는가?
- 기존 미전달 이력은 내부 기록으로 남겼는가?

### 판정
패키지 개정과 외주사 공지가 모두 있으면 DONE01. 기준표만 만들었으면 DONE02.

### 후속 조치
외주사 공지와 적용 시점 확인을 추가한다.

## 5. 범위 견적서 작성 완료 보고

### 완료 보고
내부 보고용 견적서를 보냈습니다.

### 필요한 증거
- 견적서 본문
- 가정 조건
- 포함/제외 범위
- 변동 가능성 문구
- 상세 견적 확정 조건

### 검증 질문
- 고정 견적으로 오해될 표현이 있는가?
- 제외 범위가 충분히 구체적인가?
- 요구사항 정의 후 견적 확정이라는 조건이 명시됐는가?

### 판정
가정 조건과 변동 가능성이 명확하면 DONE01. 금액만 전달됐으면 DONE04.

### 후속 조치
금액만 전달된 경우 `범위 견적`임을 정정하고 제외 범위를 보강한다.

## 6. 제품화 판단표 작성 완료 보고

### 완료 보고
납품 검수 서비스 제품화 판단표를 만들었습니다.

### 필요한 증거
- 파일럿 수 기준
- 반복 검수 항목 기록표
- 셀프 사용 가능 항목 분류 기준
- 판단일
- 필요한 데이터 담당자

### 검증 질문
- 파일럿 결과 없이 제품화 일정만 적혀 있지 않은가?
- 반복 항목과 사람 판단 항목이 분리됐는가?
- 판단일과 데이터 책임자가 있는가?

### 판정
판단 기준과 데이터 책임자가 있으면 DONE01. 제품화 희망 일정만 있으면 DONE04.

### 후속 조치
파일럿 3건 기준, 반복 항목 수집자, 판단 회의일을 추가한다.

## 7. 기능 요청 1차 필터 개정 완료 보고

### 완료 보고
기능 요청 triage 표를 개정했습니다.

### 필요한 증거
- 개정된 triage 표
- 구조 변경 위험 항목
- 유지보수 부담 항목
- 상세 검토 전환 조건
- 샘플 요청 3건 적용 결과

### 검증 질문
- 고객 요청 수만으로 통과하지 않도록 기술 리스크 항목이 있는가?
- 상세 검토 전환 기준이 숫자 또는 상태값으로 명확한가?
- 실제 샘플에 적용해 봤는가?

### 판정
샘플 적용 결과까지 있으면 DONE01. 표만 개정했으면 DONE02.

### 후속 조치
최근 요청 3건에 적용해 의사결정 결과를 비교한다.

## 8. 고위험/일반 문의 분류표 완료 보고

### 완료 보고
문의 분류표를 만들었습니다.

### 필요한 증거
- 문의 예시 10개 이상
- 고위험 기준 3개
- 상세 템플릿 적용 조건
- 애매할 때 문의할 담당자
- 상담자 피드백

### 검증 질문
- 실제 상담자가 구분할 수 있을 만큼 예시가 구체적인가?
- 일정 영향, 비용/계약 책임, 고객 업무 중단 기준이 포함됐는가?
- 애매한 문의의 에스컬레이션 경로가 있는가?

### 판정
상담자 피드백까지 있으면 DONE01. 문서만 만들고 현장 확인이 없으면 DONE02.

### 후속 조치
상담자 1~2명에게 샘플 문의를 분류하게 하고 기준을 보정한다.

## 종합 완료 검증 양식

```text
완료 보고:
필요한 증거:
검증 질문:
판정 코드:
부족한 증거:
후속 조치:
재검토일:
```

## 오늘의 사고 원칙

1. 완료 보고는 증거가 아니라 검증 대상이다.
2. 파일을 만들었다는 말은 배포, 승인, 적용, 검증을 의미하지 않는다.
3. 고객 선택이나 외부 승인이 남아 있으면 완료가 아니라 보류다.
4. 샘플 적용 없이 표만 만든 것은 부분 완료다.
5. 완료 기준과 다른 산출물이 나오면 열심히 했어도 기준 불일치다.
