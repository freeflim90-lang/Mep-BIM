# LUA BIM LABS 추론 훈련 루프 세트 04

- 생성일: 2026-06-04
- 목적: 애매한 신호를 바로 결론으로 만들지 않고, 작은 검증 실험으로 바꾸는 훈련
- 훈련 초점: `가설 설정 → 필요한 데이터 → 검증 방법 → 판정 기준 → 다음 액션`

## 검증 판정 상태

- `증거 부족`: 지금 결론을 내리면 위험하다.
- `소규모 검증`: 1~2주 또는 1~3개 프로젝트에서 작게 확인한다.
- `반복 검증`: 여러 프로젝트/고객에서 같은 패턴이 나오는지 본다.
- `운영 반영`: 체크리스트, 교육자료, 템플릿, SOP에 반영한다.
- `제품 후보`: 유료화 또는 애드인/SaaS 기능으로 검토한다.

## 1. 간섭 증가 알림 룰 검증

### 신호
간섭 건수가 주간 단위로 크게 증가할 때마다 조율 회의가 늦어진다.

### 가설
간섭 증가율을 자동 알림으로 감지하면 조율 회의 지연을 줄일 수 있다.

### 필요한 데이터
- 주차별 신규 객체 수
- 주차별 간섭 건수
- 중복 간섭 제거 후 건수
- 조율 회의 개최일
- 실제 조치가 필요했던 간섭 수

### 검증 방법
최근 3개 프로젝트에서 `간섭 증가율 / 신규 객체 증가율`을 계산하고, 실제 조율 지연이 발생한 주차와 비교한다.

### 판정 기준
- 증가율 알림이 실제 조치 필요 주차의 70% 이상을 잡으면 체크리스트 후보
- 알림은 많지만 실제 조치가 적으면 보류
- 특정 구역/공종에서만 유효하면 제한 룰로 운영 반영

### 다음 액션
Navisworks 또는 품질 대시보드에서 주간 변화량 리포트 초안을 만든다.

## 2. 뷰 템플릿 변경 감지 검증

### 신호
납품 직전에 뷰 템플릿 변경으로 시트 표현 오류가 발생한다.

### 가설
주요 뷰 템플릿 설정 diff를 기록하면 납품 전 오류를 줄일 수 있다.

### 필요한 데이터
- 납품 전후 뷰 템플릿 설정
- 변경 사용자와 동기화 시점
- 영향을 받은 시트 수
- 복구 소요 시간
- 재발 여부

### 검증 방법
주요 템플릿 5개를 선정해 매일 설정 스냅샷을 저장하고, 변경 항목과 실제 오류 발생을 비교한다.

### 판정 기준
- 변경 감지가 복구 시간을 30% 이상 줄이면 운영 반영
- 변경은 많지만 오류 영향이 적으면 교육자료 후보
- 스냅샷 관리 비용이 크면 수동 체크리스트로 축소

### 다음 액션
템플릿 중요도 등급을 정하고 납품 전 잠금/승인 절차를 설계한다.

## 3. 자동 태깅 도구 사용률 검증

### 신호
자동 태깅 도구를 배포했지만 사용률이 낮다.

### 가설
사용률 저하는 기능 부족보다 실패 후 복구 불안과 결과 신뢰 부족 때문이다.

### 필요한 데이터
- 사용자별 실행 횟수
- 실행 성공률
- 실패 객체 수
- 수동 수정 시간
- 사용자가 도구를 쓰지 않은 이유

### 검증 방법
2주 동안 실행 로그와 짧은 사용자 메모를 함께 수집한다. `안 씀`, `썼지만 수정함`, `성공적으로 사용함`을 구분한다.

### 판정 기준
- 실패 후 복구 불안이 가장 크면 되돌리기/로그 개선
- 태그 품질 문제가 크면 룰 개선
- 사용법 미숙이면 교육자료 후보

### 다음 액션
도구 실행 후 변경 수량, 실패 객체, 되돌리기 안내를 한 화면에 표시하는 개선안을 만든다.

## 4. 납품 검수 서비스 수요 검증

### 신호
여러 고객이 BIM 납품 검수가 어렵다고 말한다.

### 가설
고객은 SaaS보다 납품 전 1회성 검수 패키지를 먼저 구매할 가능성이 높다.

### 필요한 데이터
- 고객이 겪은 검수 실패 사례
- 납품 전 검수에 지불 가능한 금액
- 반복되는 검수 항목
- 긴급도
- 내부 담당자의 검수 역량

### 검증 방법
고객 5곳에 30분 인터뷰를 진행하고, 1회성 수동 검수 패키지 제안을 테스트한다.

### 판정 기준
- 2곳 이상이 유료 파일럿을 수락하면 제품 후보
- 관심은 있으나 지불 의사가 없으면 교육자료/체크리스트 후보
- 고객마다 문제가 전혀 다르면 컨설팅 상품으로 유지

### 다음 액션
납품 전 1회 검수 패키지 범위, 제외 항목, 결과 리포트 양식을 만든다.

## 5. IFC 오류 방어장치 검증

### 신호
IFC 내보내기 오류는 드물지만 발생하면 납품 반려로 이어진다.

### 가설
IFC 사전 검수 체크리스트가 납품 반려 리스크를 줄인다.

### 필요한 데이터
- 과거 IFC 반려 원인
- 내보내기 설정값
- 좌표/분류/속성 누락 여부
- 재납품 소요 시간
- 발주처별 요구사항

### 검증 방법
최근 IFC 납품 5건을 역검토해 반려 가능 항목을 체크리스트로 만들고, 다음 2건에 적용한다.

### 판정 기준
- 체크리스트 적용 후 재작업이 줄면 운영 반영
- 항목이 발주처마다 크게 다르면 발주처별 버전으로 분리
- 자동 검출 가능한 항목이 50% 이상이면 자동화 후보

### 다음 액션
IFC_OpenBIM 지식 베이스에 발주처별 내보내기 사전 점검 항목을 추가한다.

## 6. 외주 모델 품질 검증

### 신호
외주사가 제출한 모델은 겉보기엔 괜찮지만 내부 검수에서 뒤늦게 오류가 발견된다.

### 가설
외주 제출 직후 30분짜리 빠른 검수로 치명 오류를 조기 발견할 수 있다.

### 필요한 데이터
- 외주 제출 시점
- 치명 오류 유형
- 발견 시점
- 수정 요청 횟수
- 재제출 소요 시간

### 검증 방법
다음 외주 제출 3건에 대해 `좌표, 링크, 필수 파라미터, 뷰/시트, 파일 용량, IFC 가능성`만 빠르게 확인한다.

### 판정 기준
- 치명 오류를 조기에 70% 이상 잡으면 운영 반영
- 경미 오류만 잡히면 정밀 검수 단계로 이동
- 외주사별 편차가 크면 외주사 평가표에 반영

### 다음 액션
외주관리 지식 베이스에 빠른 접수 검수표와 정밀 검수표를 분리한다.

## 7. 고객지원 응답 품질 검증

### 신호
CS 1차 응답 시간은 빨라졌지만 고객 재문의가 줄지 않는다.

### 가설
빠른 답변보다 정확한 해결 문서와 후속 확인이 재문의를 줄인다.

### 필요한 데이터
- 1차 응답 시간
- 최종 해결 시간
- 재문의율
- 답변 유형
- 지식 베이스 등록 여부

### 검증 방법
최근 문의 30건을 `단순 안내`, `기술 해결`, `일정/계약`, `불만 처리`로 나누고 재문의율을 비교한다.

### 판정 기준
- 지식 문서가 없는 유형에서 재문의가 높으면 QA 문서화
- 기술 해결 문의에서 재문의가 높으면 답변 품질 개선
- 일정/계약 문의가 반복되면 안내 문구 템플릿 작성

### 다음 액션
CS_기술지원관과 고객지원운영 지식 베이스에 반복 문의별 표준 답변을 연결한다.

## 8. 요구사항 정의 단계 검증

### 신호
커스텀 개발 프로젝트에서 범위 변경이 반복된다.

### 가설
본 개발 전 유료 요구사항 정의 단계를 두면 범위 변경과 재작업이 줄어든다.

### 필요한 데이터
- 초기 요구사항 문서 수준
- 변경 요청 횟수
- 재작업 시간
- 고객 승인 지연
- 견적 대비 실제 투입 시간

### 검증 방법
다음 커스텀 개발 2건에 요구사항 정의 단계를 별도 산정하고, 기존 프로젝트와 변경 요청 횟수를 비교한다.

### 판정 기준
- 변경 요청 또는 재작업 시간이 30% 이상 줄면 운영 반영
- 고객이 정의 비용을 거부하면 무료 범위와 유료 범위를 분리
- 정의 단계가 길어져도 개발 리스크가 줄면 표준 계약 절차로 검토

### 다음 액션
요구사항분석과 BIM_프로젝트_견적산정 지식 베이스에 사전 정의 산출물 목록을 추가한다.

## 9. 교육 효과 검증

### 신호
파라미터 교육 후 오류가 줄었다고 보고됐다.

### 가설
교육이 오류 감소에 기여했지만, 템플릿 개선이나 검수 강화도 함께 영향을 줬을 수 있다.

### 필요한 데이터
- 교육 전후 오류 유형별 건수
- 교육 참석자와 미참석자 비교
- 같은 기간 템플릿 변경 여부
- 검수 기준 변경 여부
- 작업 범위 변화

### 검증 방법
교육 내용과 직접 관련된 오류만 따로 분리하고, 참석자/미참석자 또는 팀별 변화를 비교한다.

### 판정 기준
- 교육 관련 오류만 줄면 교육자료 운영 반영
- 전체 오류가 줄었지만 관련 오류 변화가 작으면 다른 요인 탐색
- 미참석자도 같이 줄면 템플릿/검수 영향 가능성 검토

### 다음 액션
교육컨설팅 지식 베이스에 교육 전후 측정표를 추가한다.

## 10. 기능 요청 우선순위 검증

### 신호
애드인 사용자들이 다양한 기능을 요청한다.

### 가설
기능 요청의 양보다 문제 공통성과 재사용성이 우선순위 판단에 더 중요하다.

### 필요한 데이터
- 요청 고객 수
- 요청 빈도
- 해결하려는 실제 문제
- 예상 개발 시간
- 유지보수 부담
- 유료 전환 가능성

### 검증 방법
요청을 기능명이 아니라 문제 유형으로 묶고, `고객 수 x 치명도 x 재사용성 / 개발·유지보수 비용`으로 점수화한다.

### 판정 기준
- 3곳 이상 같은 문제를 겪고 비용 대비 효과가 높으면 제품 후보
- 특정 고객만 필요한 기능이면 커스텀 개발 후보
- 기존 기능/교육으로 해결 가능하면 보류

### 다음 액션
Revit_Addin과 제품패키징 지식 베이스에 기능 요청 triage 표를 연결한다.

## 오늘의 사고 원칙

1. 좋은 추론은 결론보다 검증 방법을 남긴다.
2. 자동화 후보는 로그, 실패 조건, 책임 범위가 함께 정의되어야 한다.
3. 고객 수요는 인터뷰가 아니라 유료 파일럿으로 강하게 검증된다.
4. 낮은 빈도·높은 치명도 문제는 작은 체크리스트라도 먼저 만든다.
5. 교육 효과는 관련 오류만 분리해서 본다.
6. 기능 요청은 기능명이 아니라 문제 유형으로 묶는다.
