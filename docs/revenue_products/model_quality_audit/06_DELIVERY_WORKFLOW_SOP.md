## 1. 수행 흐름

| 단계 | 활동 | 산출물 | 담당 |
|---:|---|---|---|
| 1 | 문의 접수 | 고객 접수 정보서 | 영업/PM |
| 2 | 자료 및 보안 조건 확인 | 자료 요청 목록, NDA 여부 | PM/보안 |
| 3 | 견적 및 범위 확정 | 견적서, SOW | PM/견적 |
| 4 | 모델 수령 | 수령 로그, 파일 목록 | PM |
| 5 | 1차 파일 점검 | 파일 열람/버전/링크 확인 | 품질검증 |
| 6 | 상세 품질진단 | 체크리스트 | 품질검증/MEP 리드 |
| 7 | 보고서 작성 | 품질진단 보고서 | 테크니컬 라이터/PM |
| 8 | 내부 리뷰 | 리뷰 코멘트 | PM/리드 |
| 9 | 고객 제출 | PDF, XLSX, 회의자료 | PM |
| 10 | 후속 제안 | 재검토/교육/컨설팅 제안 | 영업/PM |

## 2. 역할

| 역할 | 책임 |
|---|---|
| PM | 범위, 일정, 고객 커뮤니케이션, 최종 승인 |
| MEP 리드 | 공종별 기술 판단, 간섭 우선순위 판단 |
| 품질검증 담당 | 체크리스트 수행, 증빙 정리 |
| 테크니컬 라이터 | 보고서 구조화, 문장 정리, 외부 제출 형식 정비 |
| 보안 담당 | 자료 수령/보관/삭제 기준 확인 |
| 영업 담당 | 견적, 후속 상품 제안, 고객 관계 관리 |

## 3. 내부 품질 게이트

| 게이트 | 통과 기준 |
|---|---|
| G1 자료 수령 | 모델명, 버전, 수령일, 보안 조건 기록 |
| G2 검토 착수 | 범위와 제외 항목이 SOW와 일치 |
| G3 보고서 초안 | 주요 이슈에 등급, 영향, 조치 의견 포함 |
| G4 내부 리뷰 | 과도한 단정, 범위 외 판단, 근거 없는 표현 제거 |
| G5 고객 제출 | 문서번호, 버전, 배포등급, 제한사항 확인 |

---

## 4. 보고서 작성 원칙

1. 고객을 비난하는 표현을 쓰지 않는다.
2. 문제는 현상, 영향, 조치 순서로 작성한다.
3. Critical/Major 이슈는 스크린샷 또는 모델 위치를 함께 제시한다.
4. “설계 오류”라는 표현은 근거가 명확할 때만 사용한다.
5. 불확실한 사항은 “추가 확인 필요”로 구분한다.
